SECTION 1 - FLOOR 1 IDENTITY
================================================================================

One-line identity:
FLOOR 1 = Connection Layer

Meaning:
Floor 1 exists to:
- connect
- fetch
- subscribe
- receive
- route
- monitor
- reconnect
- maintain source status

That is all.

Why this floor exists:
If the system cannot reliably receive market reality,
all upper floors become blind.
So Floor 1 is not the floor where intelligence starts.
It is the floor where visibility starts.

Floor 1 should be judged by:
- stability
- recoverability
- source observability
- handoff clarity
- operational trust

Not by:
- intelligence
- setup accuracy
- signal quality
- trade quality

================================================================================

RULE 1
Floor 1 = Connection Layer only.

RULE 2
Floor 1 must not become smart.

RULE 3
Floor 1 will NOT do:
- validation
- cleaning
- intelligence
- scoring
- setup detection
- interpretation

RULE 4
Floor 1 may produce operational facts,
but not market judgments.

Allowed examples:
- source state = DEGRADED
- latency = 420ms
- reconnect_count = 3
- heartbeat_age = 2.1s

Not allowed examples:
- source is reliable enough for trading
- market is bullish
- this setup is strong
- feed quality supports long bias

Floor 1 must hand off packets to Floor 2 cleanly,
not process them into higher-layer meaning.

RULE 6
Floor 1 must remain simple enough that failure diagnosis is easy.

RULE 7
If Floor 1 fails, the whole system becomes blind.
Therefore simplicity and reliability matter more than cleverness.

================================================================================

Current live primary = Angel One

Macro/support feeds = mixed by data type

Therefore Floor 1 source architecture should be designed as:

A. Primary Live Market Source
- Angel One

B. Future Backup Source Slots
- architecture-ready but not active now

C. Macro / Support Feed Sources
- mixed by data type
- each feed may come from the most practical source

D. Manual / Semi-Manual Source Lane

================================================================================
SECTION 5 - ADAPTER ARCHITECTURE
================================================================================

Locked direction:
Hybrid Adapter Architecture

This means two distinct layers:
1. Source Adapter Layer
2. Feed Adapter Layer

This separation should be strong,
but shared utilities are allowed.

-------------------------------------------------------------------------------
5.1 SOURCE ADAPTER LAYER
-------------------------------------------------------------------------------
Purpose:
Handle source-specific realities.

Responsibilities:
- source protocol handling
- source authentication integration
- source session usage
- source connection lifecycle
- source reconnect attempts
- source-specific packet receiving

Examples:
- Angel One Adapter
- Future Backup Adapter
- Macro Source Adapter
- Manual Source Adapter

Why this layer exists:
Different sources behave differently.
Source-specific concerns should not leak directly into feed routing logic.

-------------------------------------------------------------------------------
5.2 FEED ADAPTER LAYER
-------------------------------------------------------------------------------
Purpose:
Handle feed identity and packet routing after source acquisition.

Responsibilities:
- feed identity tagging
- feed routing classification
- packet forwarding by feed type
- envelope association
- handoff to Floor 2 path

Examples:
- Spot Feed Adapter
- Options Feed Adapter
- VIX Feed Adapter
- Calendar Feed Adapter
- Macro Feed Adapter

Why this layer exists:
The same source may carry multiple feed types.
And future source switching should not break feed contracts.

-------------------------------------------------------------------------------
5.3 SHARED UTILITIES (ALLOWED)
-------------------------------------------------------------------------------
Shared utilities may exist for:
- logging
- retry helpers
- connection utilities
- common packet wrappers

Why shared utilities are allowed:
They reduce duplication without collapsing the source/feed separation.

SECTION 6 - AUTH LIFECYCLE (DETAILED)
================================================================================

Locked direction:
Auth module separate, but connection lifecycle managed by Floor 1.

Meaning:
Authentication should not be mixed blindly into all connection code.
But Floor 1 must remain responsible for operational handling of auth-related
connection realities.

Auth split model:

A. Auth Module responsibilities
- login
- token generation
- token refresh
- credential lifecycle handling
- auth-specific failures

B. Floor 1 responsibilities around auth
- session monitoring
- auth failure detection
- reconnect attempts after auth problems
- source recovery attempts
- surfacing AUTH_FAILED lifecycle state

Why this split is strongest:
- auth logic stays modular
- connection lifecycle stays operationally coherent
- responsibilities remain clean

Important explicit state:
AUTH_FAILED

Why AUTH_FAILED must exist:
Because:
- AUTH_FAILED != DEGRADED
- AUTH_FAILED != STALE
- AUTH_FAILED != DISCONNECTED

It is a distinct operational problem and must be visible as such.

SECTION 7 - CONNECTION LIFECYCLE (DETAILED)
================================================================================

Locked direction:
Graded lifecycle with retry loop.

Lifecycle states:
- HEALTHY
- DEGRADED
- STALE
- AUTH_FAILED
- DISCONNECTED

Connection lifecycle stages:

1. CONNECT
-------------------------------------------------------------------------------
- source adapter initiates connection
- auth/session checked as needed
- feed subscriptions started
- initial source state becomes HEALTHY if normal

2. MONITOR
-------------------------------------------------------------------------------
- heartbeat observed
- packet arrival continuity observed
- latency tracked
- reconnect count tracked
- lifecycle state tracked

3. DEGRADED
-------------------------------------------------------------------------------
Entered when:
- latency rises materially
- heartbeat quality weakens
- packet flow becomes unstable
- source health weakens but not fully stale/disconnected

Behavior:
- keep receiving if possible
- keep retry/recovery observation active
- report degraded fact upward

4. STALE
-------------------------------------------------------------------------------
Entered when:
- feed has stopped updating meaningfully
- packet age/heartbeat age crosses threshold
- source appears alive but truth freshness is weak

Behavior:
- continue lifecycle handling
- continue retries/recovery as appropriate
- report stale fact upward

5. AUTH_FAILED
-------------------------------------------------------------------------------
Entered when:
- auth/token/session problem breaks source usability

Behavior:
- auth recovery path triggered
- reconnect may depend on auth refresh
- report explicit AUTH_FAILED state upward

6. DISCONNECTED
-------------------------------------------------------------------------------
Entered when:
- source is no longer connected or reachable

Behavior:
- retry loop remains active
- operator/system visibility remains available
- no fake healthy signal emitted

7. RECONNECT / RECOVERY
-------------------------------------------------------------------------------
- retry loop continues
- recovery attempts logged
- lifecycle state updates continue
- if source recovers, state can move back upward through recovery path

Why graded lifecycle is strongest:
A binary connected/disconnected model is too weak.
Upper floors and debugging need richer operational truth.

SECTION 8 - MONITORING MODEL
================================================================================

Locked answer:
Hybrid monitoring
- deep for core feeds
- lighter for support feeds

Core feeds requiring deeper monitoring:
- Spot Feed
- Options Feed
- VIX Feed
- Primary Market Feed

Support feeds with lighter monitoring:
- Economic Calendar
- FII/DII feeds
- Global Reference feeds
- secondary support feeds

Mandatory monitoring facts:
- latency
- heartbeat
- reconnect_count
- lifecycle_state

Expanded meaning:
- latency = packet delay / responsiveness fact
- heartbeat = source liveness rhythm fact
- reconnect_count = operational instability fact
- lifecycle_state = current connection health state

Why monitoring must remain factual:
Floor 1 reports:
- latency = 420ms
- state = DEGRADED

Floor 1 must not say:
- source is trading-safe

That judgment belongs higher.

SECTION 9 - PACKET ENVELOPE CONTRACT (INDEPENDENT)
================================================================================

Locked answer:
Lean operational envelope

Mandatory packet envelope fields:
- source
- feed_type
- connection_id
- packet_id
- routing_id
- received_at
- source_timestamp (if available)

Why this envelope is correct:
- enough for governance and routing
- not bloated with intelligence meaning
- not overloaded with auth internals

Important note:
Auth/session internals should not be stuffed into packet envelopes.
Those remain operational internals.

Envelope purpose:
- packet governance
- source identity
- feed identity
- traceability
- routing clarity

Not envelope purpose:
- business intelligence
- trade context
- interpreted market meaning

Mandatory source health facts:

SECTION 12 - FLOOR 1 -> FLOOR 2 HANDOFF CONTRACTS (DETAILED)
================================================================================

This is one of the most important sections in Floor 1.

Floor 1 must hand off to Floor 2:

1. Original Raw Packet
-------------------------------------------------------------------------------
Purpose:
- preserve exact source truth

2. Minimal Source Envelope
-------------------------------------------------------------------------------
Purpose:
- provide governance fields around raw truth

3. Feed Routing Identity
-------------------------------------------------------------------------------
Examples:
- SPOT_FEED
- OPTIONS_FEED
- VIX_FEED
- CALENDAR_FEED
- MACRO_FEED

Purpose:
- let Floor 2 classify ingress cleanly

4. Source Health Facts
-------------------------------------------------------------------------------
- lifecycle_state
- latency
- heartbeat_age
- reconnect_count

Purpose:
- let Floor 2 preserve operational truth

5. Manual Source Tags where relevant
-------------------------------------------------------------------------------
Purpose:
- preserve governance parity across manual and automated feeds

Why this handoff matters:
If Floor 1 hands off too little,
Floor 2 loses truth and operational context.
If Floor 1 hands off too much processed meaning,
Floor 1 starts drifting upward.

This handoff therefore stays operational and factual.

Major feed trace chain:

Floor 1 traceability role:

- not full lifecycle interpretation

SECTION 14 - FAILOVER PHILOSOPHY
================================================================================

Locked answer:
Graded failover / graceful degradation

If fallback source is unavailable,
Floor 1 behavior should be:
HEALTHY
 ↓
DEGRADED
 ↓
STALE
 ↓
DISCONNECTED

with retry loop continuing.

Why this is correct:
- immediate hard stop is too brittle
- graceful degradation gives upper floors richer operational truth
- retry keeps recovery alive

Important architecture note:
Floor 1 reports state.
Floor 1 does not decide whether trade should stop.

That remains a higher-floor concern.

SECTION 15 - FUTURE BACKUP HOOKS (INDEPENDENT)
================================================================================

Locked direction:
Current primary = Angel One
Future backup hooks = architecture-ready, not implemented now

Why future backup hooks matter now:
Even if only one source exists today,
architecture should not hard-code the impossible assumption that this will never
change.

Future backup hooks can include:
- future backup source adapter slots
- common feed contracts that survive source replacement
- source policy registry entries
- fallback-ready routing interfaces

This does NOT mean backup is implemented now.
It means future scale is not blocked by today’s structure.

SECTION 16 - WHY FLOOR 1 MUST STAY SIMPLE
================================================================================

This is one of the strongest architecture truths of the whole system.

Floor 1 failure blinds the entire system.
Therefore Floor 1 should be:
- simple
- stable
- observable
- recoverable

If Floor 1 becomes too smart:
- bugs become harder to isolate
- failures become harder to classify
- responsibilities blur
- trust in the full stack drops

Therefore the strongest version of Floor 1 is NOT the smartest one.
It is the most reliable one.

Locked separation:
- Floor 1 = Connection Layer
- Floor 2 = Truth Layer
- Floor 3 = Intelligence Layer
- Floor 4 = Interpretation Layer
- Floor 5 = Decision Layer

This must remain unbroken.

SECTION 20 - SOURCE AUTHORITY AND CONNECTIVITY OWNERSHIP CLARIFICATION
================================================================================

Floor 1 already defines Source Adapter Layer and Feed Adapter Layer.
V1.1 clarifies source authority boundaries.

Source truth ownership remains at the source/connection level until handed to
Floor 2 for truth preservation.

Clarified ownership split:
- Source Adapter Layer owns source-specific connectivity behavior
- Feed Adapter Layer owns feed identity and routing behavior
- Floor 1 owns connection-lifecycle truth and source-health truth
- Floor 1 does NOT own validated truth, cleaned truth, or structured truth

Why this tightening is needed:
It prevents future implementation from allowing adapters to silently become truth
or interpretation owners.

This also means:
- broker connectivity ownership stays in Floor 1
- auth/session operational handling stays in Floor 1 lifecycle governance
- Floor 1 remains the external source gateway, not the truth layer

SECTION 23 - DUPLICATE SOURCE HANDLING BOUNDARY
================================================================================

Duplicate source handling must remain boundary-safe.

Clarification:
- Floor 1 may preserve identifiers useful for duplicate detection
  (connection_id, packet_id, routing_id, timestamps, source tags)

It keeps Floor 1 lean and keeps validation ownership in Floor 2 where it belongs.

Floor 1 preserves enough ingress identity so downstream duplicate validation has
strong evidence to work with.

Replay readiness in later layers is much stronger when ingress lineage is stable.
