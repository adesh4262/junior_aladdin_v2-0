================================================================================
SECTION 1 - FLOOR 2 IDENTITY
================================================================================

One-line identity:
FLOOR 2 = Data Truth Layer

Meaning:
- Floor 1 brings market data into the system
- Floor 2 preserves, validates, cleans, structures, traces, and exposes it
- Floor 3 then uses that truth to create intelligence

Floor 2 exists because raw market data is not enough.
But raw market data must also not be destroyed, hidden, or over-interpreted.
So Floor 2 must do two things at once:
1. preserve original truth
2. make usable clean structured truth available

It must also preserve system trust by making all important data:
- reviewable
- replayable
- source-traceable
- quality-measured
- contract-defined

SECTION 7 - DUAL RAW MODEL (LOCKED)
================================================================================

Locked model:
Dual Raw

Raw truth must exist in two forms:

1. Original Raw
- exactly as received from source
- source quirks preserved
- useful for source bug investigation and audit

2. Normalized Raw Envelope
- source truth wrapped in standard operational envelope
- same truth, but standardized for downstream system handling

Why both are needed:
Original Raw preserves source truth.
Normalized Raw Envelope preserves system usability.

If only original raw exists:
- downstream usage becomes messy
- system has to repeatedly normalize in many places

If only normalized raw exists:
- original source truth can be lost
- parser/source disputes become hard to debug

Therefore dual raw is the safest and strongest architecture.

Example normalized raw envelope concept:
- source
- packet_type
- raw_payload_ref
- fetched_at
- received_at
- source_sequence if available
- envelope_id

This is still raw layer.
It is not interpretation.

SECTION 8 - MAJOR / MINOR CLASSIFICATION MODEL
================================================================================

Locked answer:
Major / Minor classification is used for BOTH:
- storage classification
- processing priority classification

Why this matters:
If classification is only for storage,
it loses operational value.
If classification is only for processing,
it loses governance clarity.

Suggested interpretation:

Major Data
- tick data
- candle streams
- options chain / OI snapshots
- core market feeds

Minor Data
- support feeds
- auxiliary feeds
- secondary references
- slower contextual packets

Effects of Major / Minor classification:
1. Storage priority
2. Processing priority
3. Validation strictness
4. Replay importance
5. Retention behavior
6. Output routing priority

This is one of the strongest Data Center governance decisions.

Floor 2 creates facts.
Higher floors create interpretation.

This preserves the architecture boundary:
- Floor 2 = Truth Layer
- Floor 3 = Intelligence Layer

- verify architecture correctness

2. Cleaned Replay
3. Structured Replay

Raw Replay
- source truth inspection
- parser/source debugging

Cleaned Replay

- anomaly repair verification

Structured Replay
- Floor 3 and above testing support
- operational stream simulation

Replay responsibilities of Floor 2:
- maintain replay-ready storage organization
- expose replay contracts
- support stage-specific replay access
- preserve enough metadata so replay can be trusted

Replay must be treated as a core capability, not an afterthought.
