SECTION 1 - WHAT FLOOR 4 REALLY IS
================================================================================

FLOOR 4 is the expert interpretation floor.

FLOOR 3 produces calculations.
FLOOR 5 needs judgment.
FLOOR 4 exists to convert calculated intelligence into structured expert opinion.

Why this floor is necessary:
If Captain directly reads raw calculations, then Captain becomes a god-object.
That would create:
- too much logic in one place
- weaker explainability
- difficult maintenance
- poor scalability
- re-reading complexity on every decision cycle

Therefore the role of Floor 4 is:
1. read calculations from Floor 3
2. interpret them domain-wise
3. maintain domain memory
4. identify primary and backup setups
5. define invalidation
6. produce structured reports for Captain
7. reduce complexity before it reaches Captain

One-line definition:
Department Heads are domain experts who convert calculation outputs into
Captain-readable intelligence.

SECTION 2 - FINAL HEAD STRUCTURE
================================================================================

Locked core heads = 6

1. Technical Head
2. SMC Head
3. ICT Head
4. Options Head
5. Macro Head
6. Psychology Head

Important locked points:
- Price Action is NOT a separate head
- basic candle pattern logic belongs inside Technical Head
- contextual candle behavior (rejection, displacement, reclaim quality) belongs
  inside SMC / ICT interpretation
- Macro is a light-gate/context head, not a dominant trade engine
- Psychology is a balanced brake head, not a directional trade head

Why only 6 heads?
Because Floor 4 should remain Captain-compatible.
Adding too many heads would:
- increase conflict
- increase Captain parsing load
- increase overlap
- create false intelligence instead of strong intelligence

Therefore current strongest architecture is 6 clear heads.

SECTION 5 - COMMON OPERATING RHYTHM OF HEADS
================================================================================

Locked answer:
Heads use a Hybrid rhythm.

A. 1-minute base refresh
-------------------------------------------------------------------------------
Purpose:
- keep Captain updated at intraday speed
- refresh tactical interpretation
- refresh setup relevance
- refresh confidence and invalidation relevance

Why needed:
Captain’s heavy thinking happens at 1m candle close.
Heads must therefore provide updated interpretable intelligence on that base rhythm.

B. 5-minute / 15-minute deep refresh
-------------------------------------------------------------------------------
Purpose:
- deeper structural refresh
- stronger zone validity update
- bigger timeframe confirmation refresh

Why needed:
Many important structural truths do not change meaningfully every 1 minute.
If every head fully re-thinks structure on every 1m candle, noise and overwork rise.
5m/15m refresh gives stronger structural truth.

C. Tick-level light watch
-------------------------------------------------------------------------------
Purpose:
- trigger watch only
- invalidation touch awareness
- micro confirmation watch for already-known setups

Why needed:
If heads do full thinking on every tick, the system will become noisy and slow.
Therefore tick use is limited to lightweight observation only.

Strong design reason:
This rhythm aligns exactly with Captain’s two-speed architecture.
That is why Floor 4 remains Captain-compatible.

SECTION 6 - COMMON MEMORY PRINCIPLE
================================================================================

Locked answer:
Memory depth is Head-specific.

Why not fixed memory for all heads?
Because all domains think differently.
Examples:
- SMC needs longer structural memory
- Psychology mainly needs intraday behavior memory
- Options may need session + recent shift memory
- Macro may need current day + prior carryover context

Therefore all heads should share a common memory design pattern,
but not a uniform memory depth.

Common memory design principles:
- memory must support Captain, not bloat the head
- memory should keep only domain-relevant state
- memory must make invalidation and setup selection easier
- memory should help avoid repeating known mistakes
- stale memory must be removable

Every head must answer internally:
- what do I need to remember?
- for how long?
- why does remembering this improve Captain’s decision quality?

- hybrid preserves both utility and clarity

Meaning:
Floor 3 provides quality material.
Floor 4 Head interprets that material and creates confidence.

Captain-aligned interpretation rules:

- final conviction always belongs to Captain
- head report may suggest opportunity, but never approve trade
- invalidation is mandatory in all heads

Quality -> Confidence clarification:
- Floor 3 may provide quality-side support inputs such as:
  - smc_quality_score
  - ict_delivery_score

- candidate_grade is NOT trade approval

Meaning:

Why invalidation must be mandatory:
Because Captain’s thesis integrity tracking and active trade supervision depend on it.
Without invalidation, a head only says “I like this,”
but never says “I am wrong if this breaks.”
That is not professional intelligence.

SECTION 8 - FRESHNESS + STATE SYSTEM
================================================================================

Locked:
- freshness_score + freshness_tag + last_deep_update
- head state = READY / UNCERTAIN / STALE

Why freshness matters:
Captain must not make strong decisions on stale expert views.
But stale meaning differs by head type.

Examples:
- Macro slightly stale may still be tolerable
- SMC stale may be much more dangerous
- Options stale near expiry may be very dangerous

Why state matters:
Captain should not treat all reports equally.
If a head is:
- READY -> normal trust
- UNCERTAIN -> reduced interpretive weight
- STALE -> caution / reduced trust / possible block depending on head type

Confidence interaction clarification:
- READY should usually support normal confidence formation when other evidence aligns
- UNCERTAIN should push the Head toward more cautious confidence
- STALE should reduce trust materially and therefore should not support strong confidence casually
- this is a qualitative contract, not a hard mathematical formula
- freshness and state together influence how confidently a Head can speak
- state/freshness conditions may support or weaken confidence expression,
  but confidence ownership still remains with Floor 4

This becomes especially powerful when combined with Captain’s weighted confluence.

SECTION 9 - PRIMARY + BACKUP SETUP MODEL
================================================================================

Locked:
Each head can carry max 2 active setups.
- primary_setup
- backup_setup

Why 2 is the strongest balance:
- 1 setup = too rigid, not enough flexibility
- 3+ setups = noise and decision confusion
- 2 setups = focused yet flexible

Primary setup means:
- strongest currently preferred opportunity idea in this domain

Backup setup means:
- next best meaningful alternative
- may support same or alternate direction depending on regime

Locked answer:
Primary/backup direction can be dynamic by regime.
Meaning:
- trend day -> both may be same direction
- range day -> backup may be opposite-side alternate

Why explicit primary + backup is important:
Because generic setup lists increase Captain load.
Captain needs hierarchy, not raw option overload.


================================================================================
SECTION 10 - TRIGGER MODEL
================================================================================

Locked:
Hybrid trigger style.

Normal setup:
- simple zone trigger
- zone hit + one confirmation

Premium setup:
- multi-condition trigger
- zone + reclaim + volume + context + structural support

Why hybrid is strongest:
- not all setups deserve complex trigger chains
- but best setups should not be reduced to shallow triggers

Captain benefit:
- normal setups stay lightweight
- top-grade setups stay high quality

Trigger output must be standardized.
That is why trigger_formatter.py is important.

SECTION 11 - CONFLICT HANDLING INSIDE A HEAD
================================================================================

Locked:
Hybrid conflict reporting.

If internal conflict exists, the head should not pretend certainty.
It should report:
- neutral or weak bias
- reduced confidence
- both bull_case and bear_case

Why this matters:
Captain should receive truth, not forced clarity.
Professional systems must preserve uncertainty honestly.

This is also why Floor 4 exists:
not to fake neatness,
but to convert complexity into honest structured interpretation.

SECTION 12 - FLOOR SUMMARY DESIGN
================================================================================

Locked answer:
Captain receives both:
- Floor Summary
- Individual Head Reports

Why this is strongest:
- summary reduces routine parsing load
- full reports preserve drill-down depth
- Captain can stay efficient without becoming blind

Important rule:
Floor Summary does NOT make the final decision.
Floor Summary compresses the state.
Captain still judges.

Floor Summary purpose:
- overall bias snapshot
- overall confidence snapshot
- active setup count
- head health snapshot
- conflict presence
- strongest opportunities snapshot
- stale warnings snapshot

Suggested Floor Summary fields:
- summary_timestamp
- floor_bias_snapshot
- floor_confidence_snapshot
- active_setup_count
- primary_setups_by_head
- backup_setups_by_head
- ready_heads_count
- uncertain_heads_count
- stale_heads_count
- conflict_present
- stale_warning_present
- strongest_domain_signal
- strongest_context_signal
- strongest_risk_warning
- invalidation_summary
- freshness_summary
- stale_head_overview
- summary_witness_lines

How Floor Summary is created:
1. collect all head states
2. collect all head primary/backup setup summaries
3. collect bias alignment and confidence shape
4. collect stale/conflict warnings
5. collect compact invalidation view across relevant heads
6. collect freshness summary and stale-head overview
7. compress into Captain-readable snapshot

When Captain should use Floor Summary first:
- routine heavy cycle start
- quick first-pass understanding
- fast check whether deep drill-down is even necessary

When Captain should drill down into individual reports:
- conflict present
- strong conviction candidate exists
- stale core head exists
- context quality issue present
- intervention review required
- opposite case appears significant

Why this design is important:
If Captain always reads all detailed reports first, Floor 4 failed to reduce complexity.
If Captain only reads summary, Floor 4 failed to preserve depth.
Therefore Summary + Full Reports is the correct middle path.
