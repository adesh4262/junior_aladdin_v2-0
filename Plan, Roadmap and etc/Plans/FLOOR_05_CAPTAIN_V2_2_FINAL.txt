- a raw data collector
- a data cleaner
- an indicator calculator
- an SMC detector
- an ICT detector

- final judge
- final trade permission authority
- final confluence evaluator
- final trade constructor
- final battlefield supervisor
- final capital-discipline enforcer
- final intervention authority in exceptional situations

Shortest correct definition:
Captain = the top-floor decision brain that listens to expert heads,
understands the day’s market story, and decides whether to trade, wait, or block.

SECTION 3 - CAPTAIN PHILOSOPHY
================================================================================

The Captain must behave like a disciplined institutional decision brain.
Its strength is not in looking at everything.
Its strength is in judging the right things in the right order.

Core philosophy:
1. Context before action
2. Confluence before conviction
3. Failure before greed
4. Discipline before aggression
5. One battle at a time
6. Silence is a valid output
7. Intervention is rare, not emotional

Strong Captain does NOT mean:
- over-intelligent chaos
- too many conditions
- decision paralysis
- reacting to every tick
- overtrading because data is available

Strong Captain means:
- few things deeply
- not everything lightly

This is a design law.
Captain must avoid the Over-Intelligence Trap.

opposite_case_engine.py

personality_engine.py

- adaptive by session, regime, quality, and battlefield conditions

setup_memory_store.py
- stores active/rejected/failed setup memory

- stores important zones remembered by Captain

- builds, arms, updates, expires, and selects prepared plans
- provides lightweight tick-watchable plans after heavy cycle
- central prepared-action system of Captain

setup_expiry_manager.py
- manages setup validity windows by setup class
- removes stale or invalid plans

confidence_decay_engine.py
- gradually lowers conviction when setup freshness weakens over time
- different from full expiry

trade_class_engine.py
- assigns trade class to approved trade idea

trade_constructor.py

active_trade_supervisor.py

intervention_engine.py
- handles rare Captain overrides during active trade

silence_reason_logger.py

loss_lock_manager.py

override_guard.py
- manages popup / confirmation / override permission for real-mode rule breach

session_policy.py
- session start behavior, session aggression windows, session behavior modifiers

SECTION 7 - CAPTAIN'S TWO-SPEED OPERATING MODEL
================================================================================

Captain uses a two-speed brain model.
This is a locked architectural principle.

A. HEAVY THINKING CYCLE
-------------------------------------------------------------------------------
Method: update_plan_on_candle_close()
Trigger: 1-minute candle close

Why 1-minute candle close?
- fast enough for intraday reaction
- disciplined enough to avoid tick-chaos analysis
- still able to use 3m / 5m / 15m context for decision quality

Heavy cycle responsibilities:
1. run permission gate
2. refresh market story
3. update narrative timeline
4. read latest head reports
5. compute weighted confluence
6. check opposite case
7. run pre-mortem logic
8. compute conviction state
9. determine personality / mood
10. assign trade class if needed
11. update setup memory
12. build prepared armed plans
13. apply setup expiry logic
14. apply confidence decay logic
15. if active trade exists, review thesis integrity
16. write decision snapshot
17. output state: TRADE / WAIT / BLOCKED / PREPARED

This is the main thinking cycle of Captain.

B. LIGHT OBSERVATION CYCLE
-------------------------------------------------------------------------------
Method: observe_active_trigger_on_tick()
Trigger: every tick

Light cycle responsibilities:
1. do NOT recompute heavy analysis
2. do NOT recalculate full structure
3. watch currently armed plans only
4. watch active trade thesis integrity only
5. detect dangerous battlefield changes
6. check intervention threshold if active trade exists
7. trigger execution quickly if valid prepared plan condition is hit

Heavy cycle = thinking.
Light cycle = watching and reacting.

SECTION 14 - CAPITAL, PAPER, REAL, AND REAL-MODE LOSS LOCK
================================================================================

Locked answer:
Capital is manually set by operator on dashboard.

Modes:
- ALERT -> always conceptually active
- PAPER -> uses operator-defined paper capital
- REAL -> uses operator-defined usable real capital

Captain does not invent capital.
Captain must respect the provided limit.

REAL mode special lock logic:
- no fixed hard total trade count
- paper mode may continue freely
- real mode discipline is based on losing trades

Locked real-mode discipline rule:
- 3 losing trades -> REAL mode becomes locked/off
- ALERT + PAPER remain active
- if REAL is attempted again after lock, popup appears every time
- operator must explicitly confirm override for each next REAL trade
- losing trade counter resets next trading day

Locked interpretation:
- any losing trade counts
- not only strict stop-loss hit

Responsible modules:
- loss_lock_manager.py
- override_guard.py

SECTION 22 - ARMED PLANS SYSTEM (APPROVED + FULLY WIRED)
================================================================================

This is one of the two newly approved strong ideas.
It is now a core part of Captain architecture.

Captain should not spend the whole day desperately hunting for trades.
Captain should become prepared.

Meaning:
- after heavy analysis, Captain may create one or more structured prepared plans
- these are not active trades
- these are conditional action blueprints
- they describe what must happen for action to become valid

Armed plan should contain:
- plan_id
- setup class
- direction
- originating heads/support summary
- trigger condition

- invalidation reference
- readiness / urgency note
- timestamp

Where it lives:
- built after heavy cycle
- stored in setup_memory_store / armed_plan_engine
- watched by light tick cycle

Why this makes Captain stronger:
- Captain becomes prepared, not reactive
- Captain does not need heavy thinking at trigger moment
- entry timing improves without tick-chaos thinking
- plan validity and expiry can be controlled cleanly

Important relation to WAIT rule:
- WAIT still means no generic random active trigger
- but if heavy cycle explicitly creates an approved armed plan,
  then the light cycle is allowed to watch it
- therefore Captain remains disciplined, not blind

Responsible module:
- armed_plan_engine.py

SECTION 35 - REPORT TRUST WEIGHTING (NEW)
================================================================================

This section is the most important Floor 4 alignment upgrade.
It extends Sections 8, 24, and 29 without replacing them.

Captain must now build trust weighting for each head report.
Trust weighting is NOT the same as the head's own confidence.

Important distinction:
- head confidence = how strong that head believes its current opportunity/context is
- report trust weighting = how much Captain should trust that head right now

Captain report trust weighting should depend on 5 things:

1. HEAD TYPE PRIORITY
- core heads (SMC, ICT) inherently matter more for structural truth
- support heads (Technical, Options) matter next
- Macro is lighter permission context
- Psychology is brake authority, not directional trust input

2. HEAD STATE
- READY -> normal trust
- UNCERTAIN -> reduced trust
- STALE -> much lower trust or caution escalation

3. FRESHNESS
- freshness_score
- freshness_tag
- last_deep_update

Captain should reduce trust if a report is not fresh enough for current market speed.

4. CONTEXT QUALITY (SMC / ICT only)
- context_quality_score modifies Captain trust for those core heads

5. INVALIDATION QUALITY / INVALIDATION CLARITY
- Captain may also consider how clearly and usefully a head defines invalidation
- examples of qualitative invalidation conditions:
  - invalidation clear
  - invalidation well-defined
  - invalidation ambiguous
  - invalidation weakly specified
- invalidation quality is a trust-support factor
- invalidation quality is NOT confidence
- invalidation quality is NOT conviction
- invalidation quality is NOT trade approval
- this should remain a qualitative contract, not a hard formula

Why this is necessary:
An SMC setup may exist technically,
but if context quality is poor, Captain should not trust it fully.
A head may also appear directional, but if it cannot define a clear invalidation,
Captain should trust that view less aggressively.
Conceptual trust formula (not code, just logic):

report_trust =
  base_head_priority_weight
  x head_state_modifier
  x freshness_modifier
  x (context_quality_modifier, if head is SMC/ICT)

Examples:

Case 1:
- SMC Head
- state = READY
- freshness high
- context_quality_score high
Result: Captain should trust it strongly.

Case 2:
- Macro Head
- state = STALE
- freshness low
Result: Captain should reduce trust, but not necessarily block trade.

Case 3:
- ICT Head
- state = STALE
Result: Captain must become much stricter because a core structural head is stale.

This is exactly why Floor 4 required contextual stale handling.
