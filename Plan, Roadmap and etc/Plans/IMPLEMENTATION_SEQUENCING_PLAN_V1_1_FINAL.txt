SECTION 1 - BUILD PHILOSOPHY
================================================================================

LOCKED BUILD DIRECTION:
BOTTOM → UP

Meaning:
FLOOR 1 (Market Connection) built first.
FLOOR 5 (Captain) built last.
Sides built after Floors are operational.

Why bottom-up is locked:
- each floor depends on the floor below it for truth/data
- you cannot test a Head without calculation inputs
- you cannot test Captain without Head reports
- you cannot test Dashboard without backend data flowing
- you cannot test Execution without Captain decisions
- building top-down means building blind — no data to test against

OVERALL BUILD ORDER BLOCK:

PHASE 0  → Foundation / Project Skeleton / Shared Types / Config
PHASE 1  → FLOOR 1 — Market Connection (raw data ingress)
PHASE 2  → FLOOR 2 — Data Center (truth/validation/structure/replay)
PHASE 3  → SIDE C — Base Memory Layer (journal/event stores)
PHASE 4  → FLOOR 3 — Calculation Engines (intelligence factory)
PHASE 5  → FLOOR 4 — Department Heads (interpretation/confidence)
PHASE 6  → FLOOR 5 — Captain (decision/conviction/judge)
PHASE 7  → SIDE A — Execution (action layer)
PHASE 8  → SIDE B — Dashboard (operator terminal)
PHASE 9  → Full Integration (end-to-end wiring)
PHASE 10 → Testing Gates
PHASE 11 → Production Readiness (Paper → Real)

CRITICAL RULE:
No phase starts until ALL dependencies of that phase are verified complete.

CRITICAL RULE 2:
Every phase must define explicit READINESS CRITERIA (conditions to START)
and VALIDATION CRITERIA (how to VERIFY the phase output).
These are distinct from SUCCESS CRITERIA (conditions to EXIT the phase).

CRITICAL RULE 3:
Every phase must define WHY THIS PHASE FIRST — the dependency justification
that explains why this phase sits in this exact position in the build order.
Without this justification, the build order is ambiguous.

SECTION 2 - IMPLEMENTATION PRINCIPLES
================================================================================

PRINCIPLE 1: TRUTH BEFORE INTELLIGENCE
Without truthful data, all intelligence is garbage.
Floor 1 and Floor 2 must be rock-solid before any calculation logic is written.

PRINCIPLE 2: INTELLIGENCE BEFORE JUDGMENT
Floor 3 must produce correct states/maps/scores before Floor 4 can interpret.
Floor 4 must produce correct confidence/reports before Floor 5 can judge.

PRINCIPLE 3: INTERPRETATION BEFORE DECISION
Heads first, Captain later.
Captain without Heads is an empty brain.

PRINCIPLE 4: DECISION BEFORE EXECUTION
Side A has nothing to execute until Captain produces an approved intent.
But Side A's paper broker and state machine can be built in parallel.

PRINCIPLE 5: MEMORY IS CROSS-CUTTING, NOT ORDERED
Side C (Memory) receives events from all layers.
Build Side C base infrastructure early (Phase 3) so it can start ingesting.
But full Side C richness grows as each floor/side emits events.

PRINCIPLE 6: DASHBOARD IS LAST
Side B consumes from all floors and sides.
Building it earlier means building against unstable/nonexistent backends.
Build Side B API + Cockpit only after Floors 1-5 + Side A emit real data.

PRINCIPLE 7: REPLAY IS FIRST-CLASS FROM PHASE 2
Replay capability must be designed in Floor 2 from day one.
Retrofitting replay later is exponentially harder.

PRINCIPLE 8: NO LAYER ABSORBS THE LAYER ABOVE
If a layer starts doing the job of the layer above it, stop and refactor.

PRINCIPLE 9: TEST ISOLATED, INTEGRATE LATER
Each floor must be testable with mocked inputs from the floor below.
Integration testing is Phase 9, not Phase 1.

PRINCIPLE 10: CONTRACTS BEFORE CODE
Define the input/output contracts of each module before writing logic.
Contracts are the backbone of a 5-floor system.

PRINCIPLE 11: ARCHITECTURE LAWS ARE ENFORCED AUTOMATICALLY
Add these checks to your CI/linting pipeline. If any violation is found, it's a P0 bug:
- No `confidence` variables/modules/classes in floor_3_calculations/ (belongs to Floor 4)
- No `conviction` variables/modules/classes in floor_4_heads/ (belongs to Floor 5)
- No `quality` used as confidence label in floor_4_heads/
- No trade approval logic below floor_5_captain/
- No setup generation in macro_head.py or psychology_head.py
- Dashboard API routes prefix with `request_` not `execute_`
- Side C stores have no `score`/`rank`/`recommend` methods
- No Floor 3 direct packet consumption as Captain routine input path
- No execution logic in dashboard (Side B never places orders directly)
- No raw Side C store consumption from Side B (must go through query layer)

PRINCIPLE 12: LOCAL-FIRST, CLOUD-OPTIONAL
This system is designed for a single developer on a single laptop.
Zero mandatory paid infrastructure. Zero mandatory cloud dependency.
Zero mandatory enterprise stack.
Future cloud/scale upgrades are possible but never required for core operation.
SQLite for dev. PostgreSQL only if/when scale demands it.
No Docker/Kubernetes required. No cloud account required.
The system must work on a laptop with Python, a broker API key, and an internet connection.

DEPENDENCY VALIDATION SUMMARY:

PHASE 0 (Foundation) — First because: nothing depends on it, everything depends on it.
  Without project skeleton, types, config, logging — no module can be written.
  No phase can start until Phase 0 is complete.

PHASE 1 (Floor 1) — First floor because: Floor 1 is closest to the market.
  Without market data ingress, no data exists to process.
  No phase above Floor 1 can be tested with real data.
  Phase 1 is the system's window to the outside world.

PHASE 2 (Floor 2) — Second because: raw data is not usable.
  Floor 1 produces raw packets. Floor 2 validates, cleans, structures them.
  Floor 3 cannot compute intelligence on raw/unvalidated data.
  Phase 2 transforms noise into truth.

PHASE 3 (Side C Base) — Third because: memory should start from day one.
  Side C base infrastructure needs Floor 2 data contracts (event formats, schemas).
  Once data contracts are defined, Side C can begin building storage foundations.
  Side C does NOT need Floor 3+ to be complete — it stores events from all layers.
  Starting Side C early means every subsequent phase can emit events immediately.

PHASE 4 (Floor 3) — Fourth because: structured truth must exist first.
  Floor 3 requires Floor 2's validated candles, tick streams, options snapshots.
  Without clean data, all intelligence is garbage.
  Floor 3 is the first layer that creates meaning from data.

PHASE 5 (Floor 4) — Fifth because: interpretation requires intelligence.
  Heads cannot interpret what Floor 3 hasn't calculated yet.
  Floor 4 is entirely dependent on Floor 3's domain outputs.
  Without Floor 3 states/maps/scores, Heads have nothing to form opinions on.

PHASE 6 (Floor 5) — Sixth because: judgment requires interpreted opinions.
  Captain requires Head reports and Floor Summary from Floor 4.
  Without expert interpretation, Captain would need to redo all analysis directly.
  Captain is the culmination of everything below — it must be built last.

PHASE 7 (Side A) — Seventh because: execution requires decisions.
  Side A executes Captain's approved intents.
  Without Captain, there's nothing to execute.
  Paper broker and state machine can be built in parallel after contracts are defined.

PHASE 8 (Side B) — Eighth because: visibility requires something to see.
  Dashboard consumes data from ALL floors and sides.
  Building it earlier means building against nonexistent/stub backends.
  Dashboard is the last thing built because it shows everything that came before.

PHASE 9-11 — Final because: integration, testing, production come after build.
  Full integration requires all components to exist.
  Testing requires a complete integrated system.
  Production readiness requires tested, verified code.

SECTION 5 - BUILD READINESS RULES
================================================================================

RULE 1: CONTRACT FIRST, CODE SECOND
Before writing a single line of code in any module:
- define the input contract (what this module receives)
- define the output contract (what this module produces)
- define the error contract (what happens when input is invalid)

RULE 2: MOCK BEFORE INTEGRATE
Each floor must be testable with mock data from the floor below.
If Floor 3 cannot be tested with fake Floor 2 data, the architecture is wrong.

RULE 3: ONE MODULE AT A TIME
Do not build two modules in parallel within the same floor.
Build one, test it, verify its output, then move to the next.

RULE 4: NO MODULE LEFT UNTESTED
Every module must have:
- a unit test (isolated logic)
- an integration test (with its immediate dependencies)
Before it is considered "complete."

RULE 5: CONTRACT TEST AFTER EVERY INTEGRATION
Whenever two floors/sides are connected for the first time:
- verify every output field matches what the consumer expects
- verify error conditions are handled

RULE 6: STOP AT FAILURE
If a phase's exit criteria are not met, do NOT proceed to the next phase.
Fixing problems later costs 10x more.

RULE 7: REPLAY READINESS FROM PHASE 2
Floor 2 must be replay-capable from day one.
No retrofitting replay later.

RULE 8: ALERT MODE FROM PHASE 7
Side A's ALERT mode must be operational before PAPER mode testing begins.
ALERT is always active — it's the safety net.

RULE 9: PAPER BEFORE REAL
No REAL mode testing until PAPER mode has:
- 100+ simulated trades
- zero critical failures in state machine
- verified risk gate behavior

RULE 10: DASHBOARD LAST
Do not build dashboard panels for data that does not yet exist upstream.

RULE 11: LOCAL-FIRST BUILD
Everything must work on a single laptop with zero paid cloud services.
SQLite for storage, local filesystem for logs, localhost for dashboard.
Cloud/PostgreSQL/Docker are future scale options, not current requirements.

RULE 12: NEVER BUILD WHAT YOU CANNOT TEST
If you cannot test a module in isolation with mocked inputs, restructure it.
Untestable code is untrustworthy code.

SECTION 6 - MODULE READINESS RULES
================================================================================

Each module in the system must satisfy these criteria before it is considered
ready for integration:

MODULE READINESS CHECKLIST:
[ ] Input contract defined
[ ] Output contract defined
[ ] Error contract defined
[ ] Core logic implemented
[ ] Unit tests passing (>80% coverage)
[ ] Edge cases handled (empty input, corrupt input, timeout)
[ ] Logging implemented (input, output, errors at minimum)
[ ] No hardcoded values (all configurable via config)
[ ] No direct dependencies on modules above it
[ ] Does not duplicate logic that exists elsewhere
[ ] Type annotations complete (if typed language)
[ ] Can be tested in isolation with mocked inputs

MODULE INTEGRATION READINESS CHECKLIST:
[ ] All dependent modules satisfy Module Readiness Checklist
[ ] Contract alignment verified between producer and consumer
[ ] Integration test passes (real producer → real consumer)
[ ] Error propagation works (if producer fails, consumer handles gracefully)
[ ] Latency/performance within acceptable bounds

SECTION 7 - BOTTOM-UP BUILD STRATEGY
================================================================================

The build strategy follows a strict bottom-up dependency chain.
Each phase builds exactly one logical layer of the system.

LAYER BREAKDOWN:

PHASE 0: FOUNDATION
- Project skeleton
- Shared types, enums, base classes
- Configuration system
- Logging framework
- Testing infrastructure
- CI/CD pipeline

PHASE 1: DATA INGRESS
- Floor 1 — Market Connection
  Raw data enters the system.
  Purpose: prove that Angel One data can be received, enveloped, and routed.

PHASE 2: DATA TRUTH
- Floor 2 — Data Center
  Raw data becomes trusted, validated, structured truth.
  Purpose: prove that raw ticks become clean 1m candles + options snapshots.

PHASE 3: MEMORY BASE
- Side C — Base Infrastructure
  Event ingestion, routing, storage.
  Purpose: preserve system memory from day one.

PHASE 4: INTELLIGENCE
- Floor 3 — Calculation Engines
  Structured truth becomes structured intelligence.
  Purpose: prove that clean candles produce correct SMC/ICT/Technical states.

PHASE 5: INTERPRETATION
- Floor 4 — Department Heads
  Intelligence becomes expert confidence + invalidation.
  Purpose: prove that states/maps produce correct head reports.

PHASE 6: DECISION
- Floor 5 — Captain
  Confidence becomes conviction and final decision.
  Purpose: prove that head reports produce correct TRADE/WAIT/BLOCKED decisions.

PHASE 7: ACTION
- Side A — Execution
  Decision becomes action (ALERT/PAPER/REAL).
  Purpose: prove that Captain intent becomes a properly executed trade.

PHASE 8: VISIBILITY
- Side B — Dashboard
  All layers become visible to the operator.
  Purpose: prove that operator can monitor, analyze, control, and review.

PHASE 9: INTEGRATION
- End-to-end system wiring.
  All layers connected with real data flow.
  Purpose: prove the full chain works together.

PHASE 10: TESTING
- System-wide validation.
  Purpose: prove the system is safe, correct, and robust.

PHASE 11: PRODUCTION
- Paper mode → Real mode transition.
  Purpose: safe live deployment.

SECTION 9 - PHASE 1: FLOOR 1 BUILD
================================================================================

WHY THIS PHASE FIRST (among floors):
- Floor 1 is the system's connection to the outside world.
- Without market data ingress, no data exists for any other floor to process.
- Floor 2 cannot validate/clean data it never receives.
- Floor 1 has no dependencies on other floors — only on external APIs.
- This is the only floor that connects to the broker/exchange.
- If Floor 1 doesn't work, nothing above it can be tested with real data.

WHY THIS PHASE BEFORE PHASE 2:
- Floor 2 requires Floor 1 data to validate, clean, and structure.
- Floor 2 tests need real packet formats from Floor 1.
- Floor 2's handoff contract is defined by what Floor 1 produces.

WHY NOT LATER:
- Delaying Floor 1 means delaying ALL real-data testing.
- Without real data, every floor above is built against assumptions.

WHAT TO BUILD:

File: floor_1_connection/auth_manager.py

- Token generation, storage, refresh
- Session management
- Auth failure detection
- Config: client_id, api_key, pin from config

1.2 SOURCE ADAPTERS
File: floor_1_connection/source_adapters.py

- Source lifecycle: CONNECT → HEALTHY → DEGRADED → STALE → AUTH_FAILED → DISCONNECTED → RECONNECT

- Connection_id generation per session

1.3 FEED ADAPTERS

SECTION 21 - CROSS-PHASE RISKS
================================================================================

RISK 1: CONTRACT DRIFT
Risk: During Phase 4 (Floor 3), the output contracts change slightly,
  requiring Floor 4 (not yet built) adjustments.
Mitigation: Define ALL contracts before Phase 1 starts.
  Document them in shared/contracts.py.
  Any change must be reviewed against ALL consumers.
  Create contract tests that fail if contracts drift.

RISK 2: SCOPE CREEP IN FLOOR 3
Risk: Building too many indicators/domains in Floor 3 before Floor 4 is ready.
Mitigation: Build Minimum Viable Domain outputs first.
  Structure + SMC + ICT baseline → then Technical → then Options → then Macro.
  Get Floor 4 working with baseline Floor 3 before enhancing Floor 3.

RISK 3: LOCAL-ONLY BLINDNESS
Risk: Building only for local laptop leads to architecture that can't scale.
Mitigation: Design contracts for scale even if running locally.
  SQLite now, but schema designed for PostgreSQL migration.
  Local logging now, but structured for aggregation later.
  Principle: build local, design for scale.

RISK 4: TEST DATA QUALITY
Risk: Tests use clean, perfect data that never occurs in real markets.
Mitigation: Use recorded real market data for replay tests.
  Include: gap opens, data gaps, corrupt ticks, duplicate packets.

RISK 5: ARCHITECTURE VIOLATIONS DURING STRESS
Risk: During deadline pressure, developer puts confidence logic in Floor 3
  "just to make it work."
Mitigation: Automated linting in CI. Code review required for all cross-floor changes.
  If violated, P0 bug. Fix before next commit.

RISK 6: PAPER MODE COMPLACENCY
Risk: Paper mode works, so REAL mode is activated without thorough verification.
Mitigation: Paper mode must run minimum 1 week (5 trading days).
  Paper mode must execute minimum 10 trades.
  All paper trades must be reviewed before REAL activation.

RISK 7: CAPITAL LOSS PSYCHOLOGY
Risk: First REAL loss leads to panic → override system → increase risk.
Mitigation: Loss lock is non-overridable by Captain.
  Override requires explicit popup confirmation.
  Override is logged and audited.

RISK 8: SIDE C OVER-ENGINEERING
Risk: Building complex Side C features before floors emit real events.
Mitigation: Side C stays simple until floors produce real events.
  Richness grows WITH floors, not before them.

RISK 9: DASHBOARD REDESIGN FROM BACKEND CHANGES
Risk: Backend contracts change → Dashboard needs UI redesign.
Mitigation: Define ALL backend contracts before Phase 8 starts.
  Dashboard API layer abstracts backend changes from UI.
  Contract tests catch breaking changes early.

RISK 10: SINGLE DEVELOPER BURNOUT
Risk: Solo developer trying to build 12 phases sequentially burns out.
Mitigation: Celebrate each phase exit. Take breaks between major phases.
  Phase 0 is fast. Phase 3 is fast. Phases 4 and 6 are the hardest.
  Build in parallel where possible (Phase 3 with 4, Phase 5 with 7 prep).

SECTION 22 - CRITICAL FAILURE SCENARIOS
================================================================================

SCENARIO 1 — FLOOR 1 DISCONNECT MID-TRADE
What happens:
- Angel One WebSocket disconnects
- Real-time ticks stop
- Active trade management loses price updates
Handling:
- Floor 1 → DEGRADED → STALE → DISCONNECTED
- Floor 2 detects source health change → CRITICAL signal
- Captain data_health_signal → CRITICAL
- Side A Data Health Policy → safety escalation
- Existing trade continues under last known protection
- Kill switch available for emergency flatten

SCENARIO 2 — FLOOR 2 VALIDATION COLLAPSE
What happens:
- Corrupt tick data passes all validators
- Floor 3 computes intelligence on bad data
- Heads form opinions on false zones
What prevents this:
- Multi-layer validation (duplicate, timestamp, continuity, schema, corruption)
- Tier A validation for critical feeds
- Metadata side-channel marks data quality for higher floors
- Review engine detects anomaly bursts
Recovery: Replay correct data through Floor 2 → verify same/different output

SCENARIO 3 — FLOOR 5 MAKES WRONG DECISION
What happens:
- Captain produces TRADE with low conviction that loses money
- System has no "correctness oracle" for individual trades
What prevents this:
- Weighted confluence (SMC > ICT > Technical > Options > Macro)
- Opposite case engine reduces one-sided conviction
- Loss lock prevents cascade of bad decisions
- Psychology block is non-overridable
- Paper mode catches systematic errors before REAL
Recovery: Review decision snapshot, adjust thresholds in config, not code

SCENARIO 4 — SIDE A BROKER REJECTION
What happens:
- REAL order rejected by broker
- Reason: insufficient margin, invalid strike, market closed
Handling:
- Risk Gate captures rejection
- Blocked Action Journal records severity
- Captain notified of rejection reason
- No automatic retry without Captain re-approval
Recovery: Fix rejection cause, get new Captain approval, retry

SCENARIO 5 — SIDE A UNKNOWN STATE
What happens:
- Order placed → no acknowledgement received (timeout)
- System does not know if order filled, rejected, or still pending
Handling:
- State machine transitions to UNKNOWN/PENDING_RECONCILE
- Bounded reconciliation attempts
- If unrecoverable, escalate to CRITICAL
- Dashboard UNKNOWN state visibly surfaced
Recovery: Manual inspection of broker account, manual reconciliation action

SCENARIO 6 — MEMORY CORRUPTION (SIDE C)
What happens:
- Side C store gets corrupted or inconsistent data
- Query layer returns wrong history
What prevents this:
- Append-first design (no mutation, so corruption is detectable)
- Approved emitters only (no random writes)
- Event Router validates before writing
- Retention manager provides clean/archive boundaries
Recovery: Replay from Floor 2 raw data, rebuild affected store

SCENARIO 7 — LOCAL LAPTOP FAILURE
What happens:
- Laptop crashes, reboots, loses power
- All in-memory state lost
What prevents this:
- Side C stores events durably
- Floor 2 stores data durably
- Session can be reconstructed from replay
- In-memory tables can be rebuilt from stored events
Recovery: Restart system → Side C provides last known state → Floor 2 provides data → system resumes

SCENARIO 8 — CAPITAL EXCEEDED
What happens:
- Multiple partial fills exceed intended capital allocation
- Position size exceeds comfort level
What prevents this:
- Risk Gate checks capital before every order
- Position Manager tracks filled quantity vs intended
- One-trade rule prevents second entry
- Reduce-only safety adjustment available
Recovery: Immediate flatten if position exceeds safe limits. log and review.

    Don't try to load years of history before the pipeline works.

SECTION 26 - IMPLEMENTATION ANTI-PATTERNS
================================================================================

ANTI-PATTERN 1: "BIG BANG" INTEGRATION
Bad: Build everything, then try to connect it all at once.
Good: Build floor-by-floor, integrate each floor with its consumers immediately.
Contract tests between each floor/side pair.

ANTI-PATTERN 2: PERFECTION BEFORE FUNCTION
Bad: Spend weeks optimizing Floor 3 SMC detection before Floor 4 exists.
Good: Get baseline SMC working, move to Floor 4, enhance SMC later.
Perfection is the enemy of progress in a 5-floor system.

ANTI-PATTERN 3: GOD MODULE
Bad: One module in Floor 3 that does SMC + ICT + Technical + Options.
Good: Each domain is its own module. Orchestrator coordinates, does not compute.

ANTI-PATTERN 4: ARCHITECTURE DRIFT
Bad: During Phase 4, a developer puts confidence logic in Floor 3.
Good: Architecture laws are enforced in code review + automated linting.
"If it's not in the plan, it doesn't belong."

ANTI-PATTERN 5: HARDCODING CONFIGURATION
Bad: API keys, thresholds, capital limits hardcoded in Python files.
Good: Everything in config. Config loaded from files + env vars.
No secrets in code. No magic numbers in logic.

ANTI-PATTERN 6: WRITING WITHOUT READING
Bad: Writing Floor 3 SMC engine without reading the SMC Head's input contract.
Good: Before writing any module, read the contracts of ALL its consumers.

ANTI-PATTERN 7: HIDING FAILURE
Bad: Silently catching exceptions to keep the system running.
Good: Logging failures, propagating errors, alerting operator.
Silent failures create untrustworthy systems.

ANTI-PATTERN 8: COPY-PASTE ACROSS FLOORS
Bad: Copying validation logic from Floor 2 into Floor 3 "just in case."
Good: Each floor trusts its inputs from the floor below.
If Floor 2 validates, Floor 3 does not re-validate.

ANTI-PATTERN 9: DASHBOARD BACKEND LOGIC
Bad: Dashboard API recomputing values that belong in Floors 3-5.
Good: Dashboard reveals what floors produce. It does not reinterpret.

ANTI-PATTERN 10: IGNORING REPLAY
Bad: "We'll add replay later." (Said every failed trading system.)
Good: Replay is first-class from Phase 2.
If it can't be replayed, it can't be debugged.

ANTI-PATTERN 11: CLOUD-FIRST BUILD
Bad: Designing for cloud/Docker/PostgreSQL from day one.
Good: Build local-first. SQLite, local files, localhost.
Add cloud infrastructure only when scale demands it.

ANTI-PATTERN 12: BOTTOMLESS CONFIGURATION
Bad: Every behavior needs a config flag. System becomes untunable.
Good: Config for thresholds and operational settings.
Logic stays in code. Config does not replace architecture.

SECTION 31 - SIDE C TIMING VALIDATION
================================================================================

WHY SIDE C BASE COMES BEFORE FLOOR 3:

This is one of the most questioned build order decisions because it appears
to violate the strict bottom-up dependency chain. Here is the explicit
architectural justification.

THE ARGUMENT AGAINST (why it looks wrong):
- Side C is a "side" — intuitively, sides should come after floors.
- Floor 2 is the data layer. Floor 3 is the intelligence layer.
- Side C (Memory) doesn't fit neatly between data and intelligence.
- It feels like Side C should come after all floors are built.

WHY THIS ARGUMENT IS WRONG:

ARGUMENT 1: DEPENDENCY ANALYSIS
- Side C depends on: Floor 2 data contracts (event formats, schemas, record families).
- Floor 3 depends on: Floor 2 structured truth (candles, ticks, snapshots).
- Side C does NOT depend on Floor 3, 4, 5, or Side A.
- Floor 3 does NOT depend on Side C.
- Therefore Side C and Floor 3 have NO mutual dependency.
- They both depend on Floor 2 (for different reasons).
- They CAN be built in parallel after Phase 2.

ARGUMENT 2: MEMORY IS CHEAP TO BUILD EARLY
- Side C base infrastructure is simple: types, ingest layer, event router, 3 stores, query layer.
- This is approximately 200-400 lines of Python.
- It can be built in 1-2 weeks of part-time work.
- The cost of building it early is NEGLIGIBLE.
- The cost of building it late is ENORMOUS (lost events, retrofitting).

ARGUMENT 3: EVENT LOSS PREVENTION
- If Side C is deferred until after Floor 5:
  - Floor 3 testing events are lost
  - Floor 4 testing events are lost
  - Floor 5 testing events are lost
  - Side A testing events are lost
  - All development debugging events are lost
- Without Side C, there is NO system memory during the MOST IMPORTANT
  development period (initial testing and calibration).

ARGUMENT 4: FEEDBACK LOOP
- Side C enables:
  - Debugging Floor 3 outputs by storing and comparing them
  - Debugging Floor 4 reports by storing decision history
  - Debugging Floor 5 by storing decision snapshots
  - Calibrating thresholds by reviewing historical decisions
- Without Side C during development, debugging is blind.
- Side C acts as the system's debugging memory during the build phase.

ARGUMENT 5: SIDE C DOES NOT BLOCK FLOOR 3
- Side C is parallel-safe with Floor 3.
- They can be built by different modules in the same week.
- Side C does not slow Floor 3 down.
- Floor 3 does not wait for Side C.

ARGUMENT 6: SIDE C GROWS WITH THE SYSTEM
- Side C base starts small (types + stores + query layer).
- As each floor/side is built, it begins emitting events to Side C.
- Side C's richness grows ORGANICALLY with the system.
- No retrofitting. No backfilling. No data loss.

CONCLUSION:
Side C base is built in Phase 3 (after Floor 2 data contracts are finalized)
because:
- It has no dependency on Floor 3+
- It is cheap to build early
- It prevents event loss during Floors 3-5 development
- It provides debugging feedback during the build
- It is parallel-safe with Floor 3
- It grows organically with the system

This is architecturally correct and economically optimal.

SECTION 32 - ARCHITECTURE PROTECTION RULES
================================================================================

DURING BUILD, NEVER:

RULE 1: NEVER MIX FLOOR OWNERSHIP
- Floor 3 code must NOT go into Floor 2 or Floor 4.
- Floor 4 code must NOT go into Floor 3 or Floor 5.
- Floor 5 code must NOT go into Floor 4 or Side A.
- Each floor/module has a clear file path. If it doesn't fit, it doesn't belong.

Enforcement: Automated linting checks file paths against module types.
  floor_3_calculations/ should never contain "confidence" or "conviction".
  floor_4_heads/ should never contain "conviction" or "execution".
  floor_5_captain/ should never contain "broker" or "order".

RULE 2: NEVER MIX SIDE OWNERSHIP
- Side A code must NOT go into Side B or Side C.
- Side B code must NOT go into Side A or Side C.
- Side C code must NOT go into Side A or Side B.
- If it's execution logic, it goes in side_a_execution/.
- If it's display logic, it goes in side_b_dashboard/.
- If it's memory logic, it goes in side_c_memory/.

RULE 3: DASHBOARD MUST NEVER BECOME A DECISION LAYER
- Dashboard reveals. Dashboard does NOT decide.
- Dashboard can request actions. Dashboard does NOT execute.
- Dashboard routes commands to correct owners.
- Dashboard MUST NOT contain: trade approval logic, entry logic, conviction logic,
  confidence logic, setup detection logic, execution logic.

Enforcement: All Dashboard API routes must be prefix with `request_` or `get_`.
  No `execute_` or `approve_` or `create_trade_` routes in Dashboard.
  If a route sounds like a decision, it belongs in the backend.

RULE 4: SIDE C MUST NEVER BECOME AN INTELLIGENCE LAYER
- Side C stores facts. Side C does NOT interpret.
- Side C MUST NOT contain: scoring methods, ranking methods, recommendation methods,
  confidence calculation, conviction calculation, trade approval logic.
- Side C stores memory. It does NOT analyze memory.

Enforcement: Search Side C code for `score`, `rank`, `recommend`, `confidence`,
  `conviction`, `approve`. Zero occurrences required.

RULE 5: CAPTAIN MUST NEVER BECOME AN EXECUTION ENGINE
- Captain decides. Captain does NOT execute.
- Captain MUST NOT contain: broker API calls, order placement logic,
  order management logic, position management logic.
- Captain produces intents. Side A executes them.

Enforcement: No `broker`, `order`, `fill`, `position`, `trade_lifecycle`
  imports in floor_5_captain/ modules (except captain_interface.py which
  defines the output contract to Side A).

RULE 6: FLOOR 3 MUST NEVER BECOME AN INTERPRETATION LAYER
- Floor 3 computes. Floor 3 does NOT interpret.
- Floor 3 MUST NOT contain: confidence creation, conviction creation,
  trade approval, expert opinion, final bias.
- Floor 3 produces states, maps, scores (quality only), and candidates.
- Floor 4 interprets those outputs.

Enforcement: Search Floor 3 for `confidence`, `conviction`, `trade_approval`,
  `final_bias`. Zero occurrences required. Only `quality_score` allowed.

RULE 7: FLOOR 4 MUST NEVER BECOME A DECISION LAYER
- Floor 4 interprets. Floor 4 does NOT decide.
- Floor 4 Heads produce confidence (local domain confidence).
- Floor 4 MUST NOT produce: conviction, trade approval, final decision.
- Floor Summary aggregates head states and reports.
- Floor Summary does NOT make the final decision.

Enforcement: Search Floor 4 for `conviction`, `trade_approval`, `final_decision`.
  Zero occurrences required. Only `confidence` (local domain confidence) allowed.

RULE 8: REPLAY MUST NEVER MODIFY SYSTEM STATE
- Replay reads historical data. Replay does NOT execute.
- Replay workspace is read-only.
- Replay MUST NOT: place orders, modify positions, change modes, override decisions.
- Replay is for analysis and debugging, NOT for re-execution.

Enforcement: Replay functions have no write access to execution, broker, or
  position modules. Replay routes are read-only (GET only).

RULE 9: LOCAL-FIRST, CLOUD-OPTIONAL
- The system MUST work on a single laptop with zero paid infrastructure.
- SQLite for dev. PostgreSQL only when scale demands it.
- Local logging. Cloud logging is future enhancement.
- No Docker required. No Kubernetes. No cloud accounts.
- Future cloud upgrades must not break local operation.

Enforcement: Core system must run with `pip install -r requirements.txt`

SECTION 34 - BUILD ORDER JUSTIFICATION (PER-PHASE EXPANDED)
================================================================================

PHASE 0 (Foundation):
  Why BEFORE: Every module in every phase depends on shared types, config,
    logging, and test infrastructure. Without these, no code can be written
    correctly or tested reliably.
  Why AFTER: Foundation is the starting point. Nothing precedes it.
  Why NOT EARLIER: It IS the earliest phase.
  Why NOT LATER: Delaying Foundation means every phase builds on untyped,
    unconfigured, untestable code. Rework cost is infinite.

PHASE 1 (Floor 1):
  Why BEFORE: Floor 1 is the system's connection to the market. Without live
    data ingress, all higher floors are theoretical.
  Why AFTER: Requires Phase 0 (types, config, logging) plus external
    dependencies (Angel One SDK, WebSocket library).
  Why NOT EARLIER: Already as early as architecturally possible.
  Why NOT LATER: EVERY floor above (2-5, Sides A-C) needs data from Floor 1.
    Delaying Floor 1 delays everything.

PHASE 2 (Floor 2):
  Why BEFORE: Floor 2 is where raw data becomes trusted truth. Without
    validation, cleaning, and structuring, all intelligence is garbage.
  Why AFTER: Requires Floor 1 data to process. Floor 2's input contract IS
    Floor 1's output contract.
  Why NOT EARLIER: No data to validate until Floor 1 produces packets.
  Why NOT LATER: Floor 3 cannot compute on raw data. Side C needs data
    contracts from Floor 2. Both are blocked by Floor 2.

PHASE 3 (Side C Base):
  Why BEFORE: Memory foundation needed before Floors 3-5 emit events. Without
    Side C, all testing/development events are lost.
  Why AFTER: Requires Floor 2 data contracts (event formats, schemas, record
    families). Side C cannot define event types without knowing data formats.
  Why NOT EARLIER: Data contracts from Floor 2 define Side C's storage schema.
  Why NOT LATER: Events from Phase 4+ (Floor 3 testing, Floor 4 testing,
    Floor 5 testing) are lost without Side C. Retrofitting is lossy.
    See Section 31 for complete justification.

PHASE 4 (Floor 3):
  Why BEFORE: Intelligence factory. All higher floors (4, 5) depend on Floor 3's
    domain outputs (states, maps, scores, candidates).
  Why AFTER: Requires Floor 2's validated candles, tick streams, options
    snapshots. Floor 3 cannot compute on unvalidated data.
  Why NOT EARLIER: Cannot compute intelligence without Floor 2's structured truth.
  Why NOT LATER: Floor 4 (Heads) and Floor 5 (Captain) are entirely dependent on
    Floor 3. Delaying Floor 3 blocks the ENTIRE decision chain.

PHASE 5 (Floor 4):
  Why BEFORE: Expert interpretation. Floor 4 converts Floor 3's calculated
    intelligence into Captain-readable expert confidence and invalidation.
  Why AFTER: Requires Floor 3 domain outputs. Heads interpret states/maps/scores/
    candidates — they don't compute them from scratch.
  Why NOT EARLIER: Building Heads against hypothetical Floor 3 outputs means
    redesign when real Floor 3 outputs arrive. 2x-3x rework cost.
  Why NOT LATER: Captain needs Head reports to make decisions. Without Floor 4,
    Captain is a judge without expert witnesses.

PHASE 6 (Floor 5):
  Why BEFORE: Supreme judge. Captain is the culmination of all analysis layers.
    It produces the final decision that Side A executes.
  Why AFTER: Requires Floor 4 Head reports and Floor Summary. Captain judges
    expert interpretation — it doesn't create it.
  Why NOT EARLIER: Building Captain before Floor 4 means designing against
    hypothetical Head reports. When real reports arrive, Captain needs redesign.
    This is the MOST EXPENSIVE module to redesign.
  Why NOT LATER: Side A needs Captain decisions to execute. Dashboard needs
    Captain state to display. Full system test needs Captain in the loop.

PHASE 7 (Side A):
  Why BEFORE: Action layer. Side A converts Captain's decisions into
    mechanically executed trades with full safety governance.
  Why AFTER: Requires Captain decision output. Side A's input contract IS
    Floor 5's output contract.
  Why NOT EARLIER: Building execution before Captain means designing against
    hypothetical intent formats. Side A contracts depend on Captain contracts.
  Why NOT LATER: Side B Dashboard needs execution state to display. Full system
    integration needs execution in the loop.

PHASE 8 (Side B):
  Why BEFORE: Operator terminal. Dashboard shows everything the system does
    and allows operator control within strict governance boundaries.
  Why AFTER: Requires ALL floors (1-5) and Sides (A, C Stage A) to emit real
    data. Dashboard consumes from all layers — nothing to show without data.
  Why NOT EARLIER: Building UI before backend data exists guarantees UI redesign
    when real data formats arrive. Cost of UI redesign is HIGH (many components).
  Why NOT LATER: Without Dashboard, the system is a black box. Operator cannot
    monitor, control, or review system behavior.

PHASE 9 (Full Integration):
  Why BEFORE: Wires all components together and verifies they work as a system.
  Why AFTER: Requires all phases 1-8 complete. Cannot integrate components that
    don't exist yet.
  Why NOT EARLIER: Integration with stubs hides real contract mismatches.
    Integration must test REAL component-to-component connections.
  Why NOT LATER: Integration issues found late cost 10x more to fix.
    Each week of independent development increases divergence.

PHASE 10 (Testing Gates):
  Why BEFORE: System-wide validation. Proves correctness, safety, and
    performance before real capital is at risk.
  Why AFTER: Requires fully integrated system from Phase 9. Cannot test what
    hasn't been integrated.
  Why NOT EARLIER: System-wide testing needs everything connected. Unit tests
    run per phase. System tests need the full chain.
  Why NOT LATER: Production readiness depends on passing all testing gates.
    Delaying testing means discovering safety issues after going live.

PHASE 11 (Production Readiness):
  Why BEFORE: Goes live with REAL capital. Final safety gate.
  Why AFTER: Requires all testing gates passed (Phase 10). REAL mode cannot be
    activated without verified safety.
  Why NOT EARLIER: REAL capital requires maximum possible safety verification.
    Paper mode must prove correctness over multiple trading days.
  Why NOT LATER: Delaying REAL mode delays real-world learning. But safety
    always comes before speed.
