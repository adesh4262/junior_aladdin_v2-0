SECTION 1 - PURPOSE
================================================================================

One-line identity:
SIDE B = Operator Terminal

The dashboard is not just a visual layer.
It is the operator-facing terminal through which the system is monitored,
understood, controlled within authority, and reviewed.

Dashboard exists to support four operator functions:
1. Monitor
2. Analyse
3. Control
4. Review

The dashboard must reveal system truth,
not replace system truth.

The dashboard must make the trading machine operable,
not make the dashboard itself the machine.


================================================================================
SECTION 2 - DASHBOARD PHILOSOPHY
================================================================================

The dashboard is a desktop-first institutional cockpit.
It must feel like a serious operator console, not a generic UI.

Core philosophy:
- operator-first
- truth-first
- workflow-first
- explainability-first
- attention-protective
- failure-visible
- governance-safe

The dashboard must help the operator answer these questions quickly:
- Is the system safe?
- What mode am I in?
- What is Captain doing?
- Is there an active trade?
- Is risk under control?
- Is something wrong anywhere?

The dashboard must not:
- calculate alpha
- decide trades
- reinterpret backend truth independently
- bypass ownership
- silently hide stale or degraded conditions

================================================================================

Locked principles:

1. Dashboard = Operator Terminal

5. Dashboard protects operator attention

================================================================================

Primary user:
- Operator / System Owner

Secondary users:
- Analyst / Reviewer
- future operational reviewer / auditor

Non-users in the control sense:

Operator responsibilities supported by dashboard:
- observe system status

- request allowed controls
- review decisions and outcomes
- inspect failures and health
- use replay/review workspaces when appropriate

SECTION 5 - AUTHORITY MODEL
================================================================================

Dashboard authority is controlled, not unlimited.

Allowed operator authority:
- request mode change
- set capital limits
- trigger reconnect / sync style operational requests
- trigger kill-switch / emergency actions within defined governance
- approve allowed override confirmations
- close current trade manually if allowed by governance
- open review / replay / diagnostics workspaces

Not allowed operator authority:
- direct trade creation from dashboard
- changing Captain logic
- changing Head logic
- changing Floor 3 calculations
- changing Floor 2 validation rules
- silently bypassing Side A governance
- editing live execution policy from UI

Dashboard role:
- request
- confirm
- show result
- never secretly own backend authority

SECTION 6 - COMMAND ROUTING MODEL
================================================================================

Locked model:
Central Command Router -> Correct Owner

Flow:
Dashboard
 ↓
Dashboard Command Router
 ↓
Correct Floor / Side Owner

Command Routing Lifecycle
-------------------------------------------------------------------------------
Request
 ↓
Validation
 ↓
Routing
 ↓
Owner Response
 ↓
Dashboard Ack

Meaning:
- dashboard first emits a request
- routing/governance validation happens before owner execution
- request is then forwarded to the correct owner
- owner returns response/outcome
- dashboard surfaces acknowledgement/result back to the operator

Why this lifecycle matters:
It preserves auditability and prevents the dashboard from becoming a hidden direct
control bypass surface.

Why this is strongest:
- direct UI-to-owner calls create governance looseness
- central routing creates auditability
- command ownership remains explicit

Examples:
- mode request -> Side A Mode Router
- reconnect request -> Floor 1 owner path
- capital change -> Captain/Side A context path
- kill request -> Side A emergency governance path
- replay open -> Floor 2 / future Side C review paths

Important boundary:
Command Router routes.
It does not decide.

SECTION 13 - HEALTH SURFACE
================================================================================

Health Surface must tell the operator whether the machine is safe and which layer
is degraded.

Health State Dictionary
-------------------------------------------------------------------------------
HEALTHY
- the relevant floor/side is operating normally within expected behavior
- no meaningful degradation currently surfaced

DEGRADED
- the floor/side is still functioning but with reduced quality, elevated risk,
  or operational weakness
- operator attention is needed, but total failure has not occurred

STALE
- the floor/side is not providing sufficiently fresh operational truth for normal
  confidence in timeliness
- the last known state may still be visible, but it must be treated as stale

UNAVAILABLE
- the floor/side output is currently unavailable to the dashboard
- this is stronger than stale and means the dashboard cannot obtain current truth

LOCKED
- the floor/side is intentionally blocked or safety-restricted
- this is not the same as failure; it may be a governance/safety condition

SILENT
- the decision layer (especially Captain) is intentionally not acting
- this is not automatically a failure state
- silence must be accompanied by explainable reason where applicable

Top-level grouped view should include:
- feeds
- truth
- intelligence
- decision
- action

Deeper drill-down should map to:
- Floor 1
- Floor 2
- Floor 3
- Floor 4
- Floor 5
- Side A

Health Surface must be able to show:
- system healthy
- degraded
- stale
- unavailable
- locked
- silent

This is not cosmetic system status.
This is operator survival information.

SECTION 15 - FAILURE VISIBILITY MODEL
================================================================================

Dashboard must never hide failure.
It must classify and surface it.

Explicit failure visibility targets:
- Floor 1 degraded
- Floor 2 stale
- Floor 3 delayed
- Floor 4 unavailable
- Floor 5 silent
- Side A locked

Failure states should not be treated as one generic red light.
Operator should understand what failed and why it matters.

Expected meanings:
- Floor 1 degraded -> connection/feed problem
- Floor 2 stale -> truth/governance problem
- Floor 3 delayed -> intelligence lag problem
- Floor 4 unavailable -> expert interpretation missing
- Floor 5 silent -> decision silence (with reason), not necessarily failure
- Side A locked -> action layer intentionally blocked or safety-restricted

Failure Severity Guidance
-------------------------------------------------------------------------------
This dashboard should use failure severity as an operator guidance contract,
not as a hidden decision engine.

Suggested guidance logic:
- INFO -> noteworthy condition, not immediate danger
- CAUTION -> operator attention needed, system still broadly usable
- SEVERE -> reduced trust or reduced action authority, drill-down strongly advised
- CRITICAL -> immediate top-level operator attention, safety/governance concern

Examples of guidance intent:
- Floor 1 degraded may often surface as CAUTION or SEVERE depending on impact
- Floor 2 stale may reduce trust in truth/state and should be visibly escalated
- Floor 4 unavailable should not remain a low-severity cosmetic warning
- Floor 5 silent with a healthy no-trade reason is not automatically CRITICAL
- Side A locked can become SEVERE or CRITICAL depending on mode/context

Failure surfacing model:
- top-level severity visible immediately
- affected floor/side visible on drill-down
- cause and status visible on deeper inspection

SECTION 33 - COMMAND AND CONTROL AUTHORITY TIGHTENING
================================================================================

This section tightens Sections 5, 6, 9, 23, and 28 without redesigning them.

A. Active Trade Direct Controls
-------------------------------------------------------------------------------
Dashboard direct operator actions during an active trade should remain strictly
limited to governed controls such as:
- kill-switch
- required override confirmations
- current active trade manual close
- emergency flatten

Dashboard must NOT allow:
- new trade entry from UI
- manual SL/TGT redesign
- size increase
- Captain conviction override
- Head decision edits
- Floor calculation edits

Why this is required:
The dashboard is an operator terminal, not a hidden execution/decision engine.

B. Mode Switch While Active Trade Exists
-------------------------------------------------------------------------------
Normal mode switch requests should be blocked while an active trade exists.
Examples blocked during active trade:
- PAPER -> REAL
- REAL -> PAPER
- ALERT-style operational escalation into execution mode

Allowed exception:
- emergency safety downgrade path

Why this is required:
Mode changes should remain trade-boundary events except for safety downgrade.
This preserves execution stability.

C. Capital Change Timing Contract
-------------------------------------------------------------------------------
If no active trade exists:
- capital change may apply immediately

If active trade exists:
- new capital value is stored
- current trade remains unchanged
- new capital takes effect from the next trade

Why this is required:
Mid-trade capital mutation would distort auditability and trade consistency.

D. LIVE Enable Cadence Contract
-------------------------------------------------------------------------------
Dashboard should treat live enablement as a governed approval path,
not a simple toggle.

Accepted operational model:
- live enable request
- governance/compliance checks
- final confirmation
- approval token/session awareness
- re-confirmation on expiry or critical state transitions

Why this is required:
Real trading activation must remain intentional and reviewable.

E. Command Acknowledgement Clarification
-------------------------------------------------------------------------------
Critical controls should follow:
Request -> Validation -> Routing -> Owner Response -> Dashboard Ack

Non-critical controls may use lightweight acknowledgement.

Why this is required:
Operator must know whether important actions were merely requested or actually
accepted/executed.
