SECTION 1 - PURPOSE
================================================================================

One-line identity:
SIDE C = Memory Layer

Meaning:
- Side A acts
- Side B shows
- Side C remembers

Side C exists to:
- persist system memory
- persist trade memory
- persist decision references
- persist execution truth references
- persist health/event history
- persist replay/review links
- expose queryable historical views

Side C does not exist to think.
It does not exist to judge.
It does not exist to interpret.
It does not exist to generate recommendations.


================================================================================
SECTION 2 - SIDE C PHILOSOPHY
================================================================================

Side C is the system’s memory, observability, and traceability layer.

Its purpose is not to create new truth.
Its purpose is not to create new intelligence.
Its purpose is not to create new decisions.

Its purpose is to preserve,
link,
and expose the truth already emitted by other layers.

This makes Side C useful for:
- operator review
- forensic debugging
- replay support
- audit support
- historical trade reasoning review
- system health reconstruction

Side C is not a passive dumping ground.
It is a structured memory layer.
But it must remain non-intelligent.


================================================================================
SECTION 3 - CORE PRINCIPLES
================================================================================

Locked principles:

1. Side C stores facts
2. Side C does not create truth
3. Side C remembers
4. Side C does not decide
5. Side C links
6. Side C does not interpret
7. Side C may summarize
8. Side C may not score, rank, recommend or analyse
9. Correction = new event
10. No silent mutation

Why these principles matter:
Without them, Side C can accidentally drift upward into:
- intelligence
- interpretation
- decision support ranking
- stealth governance

That must never happen.


================================================================================
SECTION 4 - INTERNAL ARCHITECTURE
================================================================================

Locked internal split:
Hybrid layered split

Recommended internal structure:

SIDE_C_MEMORY/
│
├── ingest_layer
├── event_router
├── event_store
├── journal_store
├── reference_store
└── query_layer

Logical flow:
Source Layer / Floor / Side Owner
 ↓
Ingest Layer
 ↓
Event Router
 ↓
Event Store / Journal Store / Reference Store
 ↓
Query Layer
 ↓
Consumers (Dashboard / Replay / Review / Audit)

Why this split is strong:
- ingest stays separate from storage
- storage families remain clear
- references stay first-class
- query concerns do not pollute storage design
- future Side B / replay / audit access remains cleaner


================================================================================
SECTION 5 - INGEST LAYER
================================================================================

Purpose:
The ingest layer receives memory-worthy events and records from approved sources.

Responsibilities:
- accept inputs from approved emitters only
- identify record family
- apply envelope normalization rules
- forward to event router

What Ingest Layer does NOT do:
- interpret event meaning
- rank importance beyond routing contracts
- change ownership truth
- generate analysis

Why it exists:
Sources should not write directly into memory storage.
There should be a controlled ingress path.


================================================================================
SECTION 6 - EVENT ROUTER
================================================================================

Locked write model:
Central Event Router

Flow:
Source
 ↓
Event Router
 ↓
Correct Side C Store

Event Router responsibilities:
- classify incoming event family
- validate against write contract family
- route to:
  - event store
  - journal store
  - reference store
- ensure schema family compatibility
- preserve append-first behavior

Why central router is strongest:
- floors/sides do not write directly into stores
- governance remains centralized
- future normalization stays consistent
- memory corruption risk reduces

Event Router is not an event bus architecture.
It is a controlled memory ingress router.


================================================================================
SECTION 7 - EVENT STORE
================================================================================

Purpose:
Store normalized operational and system events that are not long-form journals,
but are important for history, diagnostics, and reconstruction.

Examples of event-store-suitable records:
- execution events
- health events
- blocked actions
- override events
- state transitions
- important system events

Why event store exists separately:
Because not every memory record is a trade journal.
Some are operational event chains.

Event store remains append-first.
It should favor traceability over convenience mutation.


================================================================================
SECTION 8 - JOURNAL STORE
================================================================================

Purpose:
Store journal-family records that represent durable narrative/business memory of
trade, decision, and review lifecycle.

Journal store is where higher-meaning record families live,
without becoming an intelligence layer.

Why separate from event store:
A trade journal is not just another operational packet.
A decision journal is not the same as a reconnect event.

This separation helps:
- retention policy
- queries
- review UX
- side-B display


================================================================================
SECTION 9 - REFERENCE STORE
================================================================================

Purpose:
Store linkable references and traceability anchors.

Examples:
- replay_ref
- review_ref
- source_ref
- packet lineage refs
- cross-system IDs

Why separate from events/journals:
References are linking instruments.
They should not be forced into narrative or operational stores awkwardly.

Reference Store strengthens:
- replay linkage
- review linkage
- forensic traceability
- future graph-like memory navigation


================================================================================
SECTION 10 - QUERY LAYER
================================================================================

Locked read direction:
Hybrid query model

Why query layer exists:
Not all consumers should read raw stores directly.
Complex review, audit, and replay queries need controlled access.

Hybrid query model means:
- simple views may be served through lightweight direct/simple projections
- complex review/replay/audit access should go through a query layer

Responsibilities:
- query composition
- read model support
- family-aware retrieval
- dashboard summary query support
- replay reference lookup
- audit lookup

What Query Layer does NOT do:
- interpret meaning
- score trades
- recommend actions
- create intelligence


================================================================================
SECTION 11 - RECORD FAMILIES
================================================================================

Locked Side C record families:
1. Trade Journal
2. Execution Events
3. Decision Journal
4. Health Events
5. Override History
6. Replay References
7. Review References

Why full family model is justified:
System memory is not only trades.
A useful memory layer must also preserve:
- what was executed
- why it was decided
- what failed
- what got blocked
- what must be replayed or reviewed later


================================================================================
SECTION 12 - TRADE JOURNAL
================================================================================

Purpose:
Store durable trade records for later review.

Trade Journal owner:
- Side A source

Side C role:
- store
- link
- expose through query models

Trade Journal should include records such as:
- trade_id
- symbol
- entry
- exit
- pnl
- mode
- exit_reason

Why this family exists:
A trade must remain reconstructable without forcing the operator to piece it
together from low-level operational logs.

Trade Journal Owner Rule:
Trade Journal truth comes from Side A.
Side C stores it.
Side C does not own the trade truth itself.


================================================================================
SECTION 13 - DECISION JOURNAL
================================================================================

Purpose:
Store durable decision-facing memory.

Decision Journal owner:
- Floor 5 source

Examples of records:
- trade_id
- decision_id
- conviction band
- decision timestamp
- heads used
- no-trade reason summary where appropriate

Why this family exists:
Trade execution history alone cannot tell why the trade existed.
Decision journal preserves the decision-side memory.

Decision Journal Owner Rule:
Decision truth comes from Captain/Floor 5.
Side C stores it.
Side C does not interpret it.


================================================================================
SECTION 14 - OVERRIDE HISTORY
================================================================================

Purpose:
Store override and special authority intervention history.

Override History owner:
- originating authority source
  (Side A override guard / Captain strategic override / operator-approved override path)

Examples:
- override_id
- override_type
- override_reason
- timestamp
- affected trade/order refs

Why this family exists:
Overrides are rare and sensitive.
They deserve explicit durable memory.

Side C stores override history.
It does not judge whether the override was correct.


================================================================================
SECTION 15 - EXECUTION EVENTS
================================================================================

Purpose:
Store operational truth emitted by Side A execution processes.

Execution event examples:
- order_placed
- order_modified

- order_rejected
- sl_hit

- partial_exit
- override_event
- reconcile event

Owner:
- Side A source

Why this family exists:
Execution event streams are not the same as trade journals.
They are operational lifecycle memory.


================================================================================
SECTION 16 - HEALTH EVENTS
================================================================================

Purpose:
Store health-relevant system events over time.

Owner:
- Floor 1 / Floor 2 source

Examples:
- feed_disconnected
- feed_reconnected
- auth_failed
- source_degraded
- data_health_changed
- stale transitions
- lock/unlock related health state transitions

Why this family exists:
Without historical health memory,
incident diagnosis becomes weak.

Side C stores the health timeline.
It does not decide health implications.


================================================================================
SECTION 17 - REPLAY REFERENCES
================================================================================

Purpose:
Store replay-facing references for later reconstruction and playback access.

Owner:
- originating replay-support source

Examples:
- trade_id -> replay_ref
- decision_id -> replay_ref
- source lineage -> replay_ref

Why this family exists:
Replay tooling needs stable link references,
not just loose text logs.


================================================================================
SECTION 18 - REVIEW REFERENCES
================================================================================

Purpose:
Store references that connect trades/events/decisions to later review artifacts.

Owner:
- originating review-support source

Examples:
- trade_id -> review_ref
- decision_id -> review_ref
- health event -> audit_ref

Why this family exists:
A memory layer should not only remember events,
it should also support later structured review.


================================================================================
SECTION 19 - BLOCKED ACTIONS
================================================================================

Purpose:
Store blocked action memory.

Owner:
- originating block source (mostly Side A execution governance paths)

Examples:
- blocked trade
- blocked order action
- blocked mode transition
- blocked override

Important because:
A blocked action is also part of system history.
It matters for diagnostics, governance, and operator review.


================================================================================
SECTION 20 - EVENT NORMALIZATION MODEL
================================================================================

Locked answer:
Common Event Envelope + Family-Specific Payload

Common Event Envelope should carry fields such as:
- event_type
- source
- timestamp
- severity
- refs

Family-specific payloads remain independent.

Why this is strongest:
- common governance stays possible
- family-specific richness stays possible
- no fake one-size-fits-all event model is forced

This keeps Side C clean and future-proof.


================================================================================
SECTION 21 - LINKING MODEL
================================================================================

Locked answer:
Extended refs mandatory, graph-like linkability possible later.

Mandatory references may include:
- trade_id
- decision_id
- order_id
- override_id
- replay_ref
- review_ref

Why this is strongest:
- enough traceability for audit/review/replay
- no need for full graph engine now
- future linkability remains open

This is one of the strongest Side C foundations.


================================================================================
SECTION 22 - RETENTION MODEL
================================================================================

Locked answer:
Family-based retention + archive tiers

Why this is strongest:
Not all memory should live equally long in equally active storage.

Typical direction:
- Trade Journal -> long-term
- Decision Journal -> long-term
- Execution Events -> medium-term
- Health Events -> medium-term
- Noisy low-level operational events -> shorter-term

Archive tiers may exist later without changing memory philosophy.


================================================================================
SECTION 23 - ARCHIVE MODEL
================================================================================

Archive model should respect record-family importance.
It should allow:
- durable memory where needed
- cheaper or less active retention tiers where appropriate

Archive is not deletion.
Archive is long-term structured memory placement.

Why this matters:
A system memory layer should not force everything into one storage temperature.


================================================================================
SECTION 24 - GOVERNANCE MODEL
================================================================================

Locked answer:
Strong governance

Need:
- approved emitters only
- schema validation
- append policy
- write contracts

Why strong governance is required:
If Side C becomes easy to write into without discipline,
its value as memory and forensic layer collapses.

Side C should be easy to read,
not easy to corrupt.


================================================================================

Central Event Router write model

- Floors/Sides do not write directly into stores arbitrarily
- approved emitters go through Event Router

Why this is strongest:
- ownership safe
- contract safe
- storage corruption risk reduced
- future normalization manageable

- replay tools should not mutate memory

This is one of the strongest Side C boundary rules.

SECTION 40 - REPLAY AUTHORITY CHAIN CONTRACT
================================================================================

Replay is a cross-system capability, but authority must remain explicit.

Replay authority chain:

Floor 2
- owns replay-capable data readiness
- owns raw / cleaned / structured replay capability

Side C
- owns storage of replay references and review references
- links replay-able artifacts to trade / decision / health memory

Side B
- owns replay workspace visibility only
- consumes replay references in read-only form

Important rule:
Replay authority is distributed but not ambiguous:
- Floor 2 = replay-capable truth owner
- Side C = replay reference memory owner
- Side B = replay workspace consumer

Why this clarification is needed:
Without this explicit chain, replay can become conceptually “everyone’s feature”
and later implementation can blur ownership.

This does not create a new replay subsystem.
It only makes the existing chain explicit.
