----------------------------------------------------------------------
SECTION 1 — PURPOSE
----------------------------------------------------------------------

Phase 11 transitions the Junior Aladdin system from a validated, tested
paper-trading system to a live capital-deployed system.  This is the
highest-stakes phase in the entire build — any mistake here costs real
money.

This phase has four pillars:

  1. SAFETY VERIFICATION — Every capital protection mechanism is verified
     one final time under conditions that mimic production.

  2. PAPER → REAL TRANSITION — The system graduates from paper broker
     to Angel One live broker through a controlled, multi-stage process
     with clear go/no-go gates at each stage.

  3. OPERATOR PREPARATION — The operator manual, runbook, troubleshooting
     guide, and alert response procedures are written and reviewed.

  4. LAUNCH — The system goes live in REAL mode with:
     - Small initial capital (defined risk budget)
     - Monitored first session (operator present)
     - Gradual scaling (capital increases only after verified sessions)

Phase 11 is the FINAL phase.  After this, the system is live.

Primary goals (in priority order):

  1. PROTECT CAPITAL — Zero unplanned losses during transition
  2. VERIFY SAFETY — Every protection mechanism works before REAL capital
  3. PAPER → REAL — Safe graduation through staged gating
  4. OPERATOR READY — Manual, runbook, alerts all prepared
  5. LAUNCH — First live session with controlled risk budget
  6. MONITOR — Post-launch observation period with escalation path

----------------------------------------------------------------------
SECTION 2 — COMPONENT ARCHITECTURE
----------------------------------------------------------------------

Phase 11 adds NO new system components.  It activates existing components
that were built in earlier phases but only used in ALERT/PAPER mode:

  Previously active:
    - All floors (1-5) — producing intelligence, interpretation, decisions
    - Side C — storing all events
    - Side B — displaying everything
    - Side A — ALERT mode (always active) + PAPER mode (testing)
    - Side A paper_broker — simulated execution

  Now activated:
    - Side A real_broker — Angel One live broker connection
    - Side A mode_router REAL path — live execution routing
    - Side A kill_switch — CRITICAL mode (flatten real positions)
    - Side A reconciliation_engine — reconciling with live broker truth
    - Side B control surface — operator can request mode switches,
      capital changes, kill switch activation on live system

──────────────────────────────────────────────────────────────────────

Purpose:  One final pass of every capital protection mechanism.
          This is NOT the same as Phase 10 testing — this verification
          runs against the EXACT configuration that will be used for
          REAL mode.

Verification checklist (ALL must pass — NO exceptions):

  RISK GATE (Side A):
  [ ] Available capital check: rejects if insufficient capital
  [ ] Required capital check: rejects if margin requirement too high
  [ ] Margin availability: rejects if margin limit exceeded
  [ ] Quantity sanity: rejects unrealistic quantities
  [ ] Lot feasibility: rejects non-standard lot sizes
  [ ] Max loss limit: rejects if per-trade or per-day loss limit exceeded
  [ ] Mode validation: rejects if intent mode ≠ execution mode
  [ ] Real lock state: rejects if loss lock active
  [ ] Duplicate prevention: rejects duplicate intents
  [ ] One-trade enforcement: rejects second trade while position active
  [ ] Stale intent detection: rejects intents older than 60 seconds
  [ ] Data health check: rejects if data stale/critical

  KILL SWITCH (Side A):
  [ ] SOFT: blocks new entries (existing trade continues)
  [ ] CRITICAL: flattens position + freezes execution
  [ ] Deactivation: restores normal operation
  [ ] Alert fires on activation (dashboard + Side C HEALTH_EVENT)

  LOSS LOCK (Floor 5):
  [ ] 3 consecutive losses → REAL mode locked
  [ ] Lock persists across sessions
  [ ] Override required to unlock (with reason logging)
  [ ] Psychology block: non-overridable

  MODE ROUTER (Side A):
  [ ] ALERT always active regardless of mode
  [ ] PAPER → REAL transition works (idle state only)
  [ ] REAL → PAPER transition works (idle state only)
  [ ] Mode switch blocked during active trade
  [ ] Dashboard reflects current mode correctly

  CAPITAL LIMITS:
  [ ] Per-trade capital limit enforced
  [ ] Per-day capital loss limit enforced
  [ ] Per-session capital limit enforced
  [ ] Account balance tracked correctly
  [ ] No negative balance possible

  DATA HEALTH POLICY:
  [ ] DEGRADED → stricter risk checks applied
  [ ] STALE → block new entries, existing trades continue
  [ ] CRITICAL → escalate, flatten if needed

  RECONCILIATION:
  [ ] Mismatch detection works (quantity, price, state)
  [ ] Broker truth preferred over local state
  [ ] UNKNOWN state → bounded reconcile → escalate if unresolved

  OPERATOR OVERRIDE:
  [ ] Override grants: mode change, kill switch, loss lock unlock
  [ ] Override scope limited (exit/risk-side only after execution start)
  [ ] Every override logged to Side C (OVERRIDE family)
  [ ] Override requires confirmation (no single-click execute)

  EMERGENCY FLOWS:
  [ ] Floor 1 disconnect → reconnect → data health → risk gate responds
  [ ] Broker disconnect → reconciliation → UNKNOWN → escalate
  [ ] Two consecutive timeouts → SOFT kill switch auto-activates
  [ ] Memory limit approaching → alert → safety degradation

SECTION 6 — WIRING DIAGRAM
----------------------------------------------------------------------

Phase 11 activates the REAL path in Side A:

                    ┌─────────────┐
   ALERT (always) ──►  ALERT path  ──► Dashboard notification
                    └─────────────┘

                    ┌─────────────┐
   PAPER (Stage B)──►  PAPER path  ──► paper_broker
                    └─────────────┘

                    ┌─────────────┐
   REAL (Stage C) ──►  REAL path   ──► real_broker ──► Angel One ──► Market
                    └──────┬──────┘
                           │
                           ▼
                    ┌─────────────┐
                    │ Reconcile   │◄──── broker truth
                    └─────────────┘

  ALL paths go through:
    captain_interface → mode_router → intent_fingerprint → risk_gate
    → state_machine → execution_core → order_lifecycle → position_manager
    → protection_model → reconciliation → logging → blocked_action_journal

  The ONLY difference between PAPER and REAL:
    - PAPER: execution_core.submit_order() → paper_broker
    - REAL:  execution_core.submit_order() → real_broker

  ALERT mode:
    - Fires notification (dashboard + log) for EVERY intent
    - Does NOT execute (no broker path)
    - Used for observation (Stage 11.4A)

SECTION 7 — DATA FLOW
----------------------------------------------------------------------

7.1  Stage 11.4A — ALERT mode data flow

  Captain intent ──► captain_interface ──► mode_router
    └── ALERT path ──► dashboard notification ──► log
    └── No execution path (PAPER + REAL disabled)

7.2  Stage 11.4B — PAPER mode data flow

  Captain intent ──► captain_interface ──► mode_router
    ├── ALERT path ──► dashboard notification (always fires)
    └── PAPER path ──► risk_gate ──► execution_core ──► paper_broker
        └── paper_broker ──► simulated fill (slippage, delay)
            └── state machine ──► position manager ──► protection model

  Identical to Phase 10 paper simulation, but with production config.

7.3  Stage 11.4C+ — REAL mode data flow

  Captain intent ──► captain_interface ──► mode_router
    ├── ALERT path ──► dashboard notification (always fires)
    └── REAL path ──► risk_gate ──► execution_core ──► real_broker
        └── real_broker ──► Angel One API ──► broker ack/fill
            └── state machine ──► position manager ──► protection model
            └── reconciliation_engine ◄── broker truth

  Key differences from PAPER:
    - Real broker sends live orders (real capital at risk)
    - Reconciliation engine actively compares local vs broker
    - Broker truth is final authority
    - SL/TP are real broker orders (not simulated)

7.4  Error data flow (REAL mode)

  Any execution error ──► reconciliation ──► broker truth check
    └── If recoverable ──► retry (max 3)
    └── If unrecoverable ──► blocked_action_journal ──► Side C
    └── If capital breach ──► CRITICAL kill switch ──► flatten
    └── If broker disconnect ──► reconnect flow ──► reconcile
    └── If unknown state ──► bounded reconcile ──► escalate

7.5  Recovery data flow (REAL mode)

  Kill switch activated:
    SOFT: block new entries, existing trade continues
    CRITICAL: flatten position → freeze → operator resolves
    Recovery: operator reviews → resolves cause → deactivates
      → replay last healthy state → resume

  Broker disconnect:
    Reconnect (auto, max 3 attempts)
    → reconcile local vs broker after reconnect
    → if match: resume normal operation
    → if mismatch: escalate (SEVERE) → operator intervention

  Loss lock (3 losses):
    Captain sets REAL mode locked
    → operator reviews trades
    → if override: logged reason + limited unlock
    → if no override: system stays locked (forced cooldown)
