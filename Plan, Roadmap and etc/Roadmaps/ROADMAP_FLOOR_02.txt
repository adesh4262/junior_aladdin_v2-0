SECTION 1 - PURPOSE
================================================================================

One-line identity:
FLOOR 2 = Data Truth Layer — the system's trusted foundation for all data.

Why this floor exists:
- Floor 1 brings raw market data into the system
- Floor 2 preserves, validates, cleans, structures, traces, and exposes it
- Floor 3 then uses that truth to create intelligence
- Floor 2 is the MOST important floor because everything above depends on whether
  the data foundation is truthful, traceable, validated, replayable, and contract-governed
- Floor 2 must do TWO things at once: preserve original truth AND make usable clean
  structured truth available

What Floor 2 creates:
- Data Contract Registry (first-class governance — incoming, internal, outgoing contracts)
- Raw Storage Layer (original raw + normalized raw envelope — dual raw model)
- Validation Pipeline (5 validators: duplicate, timestamp, continuity, schema, corruption)
- Cleaning Layer (tick cleaning, options cleaning, anomaly repair, cleaned layer writer)
- Structuring Layer (tick streams, candle streams, options snapshots, session packets)
- Metadata Side-Channel (quality facts, traceability, transform stage, health signal)
- Review Engine (hybrid: continuous + scheduled + event-triggered)
- Replay Engine (3 levels: raw replay, cleaned replay, structured replay)
- Major/Minor Classification (storage priority + processing priority)
- Floor 3 Handoff (7 output categories: validated tick stream, candles, options snapshots,
  session packets, macro support packets, metadata side-channel, computed-ready hooks)

What Floor 2 explicitly does NOT create:
- Any intelligence, analysis, or setup detection (Floor 3+ owns this)
- Any confidence or conviction scores (Floors 4-5 own this)
- Any trade judgment or market interpretation (Floor 3+ owns this)
- Any trading opinion ("this feed is good enough for trade" — forbidden)
- Any execution logic (Side A owns this)

CRITICAL ARCHITECTURE RULES (must never be violated):
- Floor 2 generates truth, metadata, and validated structure ONLY — NOT intelligence
- Floor 2 does NOT detect setups (no FVG/OB trade candidate logic — that's Floor 3)
- Floor 2 quality facts are factual (packet_completeness=96%), NOT interpretive
  ("this feed is good for trading")
- Floor 2 does NOT drift into Floor 3 territory (no structured intelligence,
  market interpretation, candidate setup meaning, domain confidence)
- Floor 2 does NOT drift into Captain territory (no trade permission, no conviction)
- Replay is a FIRST-CLASS responsibility, not optional future utility
- Data Contract Registry is a FIRST-CLASS governance responsibility

STRICT GUARDRAILS (LOCKED — 10 RULES):
1. No intelligence generation — only truth, metadata, validated structure
2. No setup detection — no FVG trade candidate, OB trade candidate, trend continuation
3. No scores that behave like intelligence scores — quality facts only
4. No opinion — allowed: "packet_completeness=96%" — forbidden: "this feed is good for trade"
5. Limited to: truth preservation, validation, cleaning, structuring, traceability,
   replay, metadata, contracts
6. Must not drift into Floor 3 territory
7. Must not drift into Captain territory
8. Replay = first-class responsibility
9. Data Contract Registry = first-class governance
10. Supports upper floors through contracts, not hidden assumptions

SUB-SYSTEM RESPONSIBILITIES:

datacenter_types.py:
- Floor 2 specific types: ValidationResult, CleaningResult, StructureResult
- ValidationTier enum (A/B/C), DataClass enum (MAJOR/MINOR)
- ReviewSignal enum (GOOD/CAUTION/DEGRADED/CRITICAL)
- TransformStage enum (RAW/VALIDATED/CLEANED/STRUCTURED)

datacenter_contracts.py:
- All feed schemas defined explicitly
- Incoming contracts (from Floor 1)
- Internal contracts (between Floor 2 sub-systems)
- Outgoing contracts (to Floor 3)

data_contract_registry.py:
- First-class governance: docs + runtime checks on key feeds
- Defines: stream/packet name, ownership, schema, freshness expectation,
  source expectations, quality metadata fields, major/minor class, consumers
- Runtime enforcement on critical feeds

INGEST SUB-SYSTEM (ingress/):

raw_ingest_router.py:
- Accept incoming packets from Floor 1
- Classify by routing identity (SPOT_FEED, OPTIONS_FEED, etc.)
- Route to appropriate processing pipeline
- Log all ingest activity

source_envelope_builder.py:
- Normalize Floor 1 envelope into Floor 2's standard envelope format
- Preserve ALL Floor 1 fields (add, never remove)
- Add Floor 2 ingest metadata (ingested_at, ingest_batch_id)

ingress_monitor.py:
- Track incoming packet rate, volume, source health
- Detect ingress anomalies (zero packets, rate drop, misrouted)
- Feed into review engine

RAW STORAGE SUB-SYSTEM (raw/):

original_raw_store.py:
- Write Floor 1 raw packets EXACTLY as received (unmodified)
- Architecture rule: original raw = preserved source truth
- Index by: received_at, source, feed_type, connection_id

normalized_raw_store.py:
- Write Floor 1 packets wrapped in Floor 2's standard operational envelope
- Same truth, standardized for downstream system handling
- Index by: normalized_timestamp, feed_type, packet_id

raw_retention_manager.py:
- Dynamic retention by data class (Major/Minor)
- Major: tick data, options chain, OI snapshots → longer retention
- Minor: support feeds, auxiliary, secondary → shorter retention
- Configurable TTLs per class

VALIDATION PIPELINE (validation/):

validation_router.py:
- Route each packet to appropriate validators based on ValidationTier
- Tier A (very strong): tick data, options chain, OI snapshots
  → ALL 5 validators run
- Tier B (strong): candle data, session packets, major structured streams
  → 4 validators (skip corruption)
- Tier C (medium/basic): macro support, secondary references, auxiliary
  → 2 validators (schema + timestamp)
- Aggregate results, compute validation_confidence

duplicate_validator.py:
- Check packet_id uniqueness on normalized_raw_store
- Check received_at + source + feed_type combination for duplicates
- Configurable dedup window (default: 100ms for ticks, 1s for others)
- Output: is_duplicate (bool), duplicate_of_packet_id (str | None)

timestamp_validator.py:
- Check chronological order (received_at should be increasing per source+feed)
- Check received_at > source_timestamp (packet can't arrive before it was sent)
- Check received_at within acceptable clock skew (configurable, default 5s)
- Output: is_ordered (bool), timestamp_delta_ms (float), clock_skew_warning (bool)

continuity_validator.py:
- Check for missing ticks/candles in expected sequence
- Expected: 1 tick per ~100ms for spot ticks during market hours
- Detect: gap > 1s → discontinuity flagged
- Track: continuity_status = GOOD / MINOR_GAP / MAJOR_GAP / GAP_RECOVERED
- Output: continuity_status, last_good_sequence_id, gap_duration_ms

schema_validator.py:
- Check ALL mandatory fields present per data contract
- Check field types match contract definition
- Check field value ranges (e.g., price > 0, volume >= 0)
- Output: schema_compliant (bool), missing_fields (list), type_mismatches (list)

corruption_validator.py:
- OHLC sanity: high >= low, high >= open, low <= close, etc.
- Price sanity: within NSE circuit limits (configurable)
- Volume sanity: not astronomical (100x+ above normal)
- OI sanity: not negative, not sudden 1000x spike
- Output: is_corrupt (bool), corruption_type (str), details (dict)

CLEANING LAYER (cleaning/):

tick_cleaner.py:
- Remove zero-volume ticks (exchange artifacts)
- Remove ticks with obviously wrong prices (fat-finger guards)
- Flag but preserve removed ticks for audit (never truly delete)
- Output: cleaned_tick (dict | None), removal_reason (str | None)

options_cleaner.py:
- OI sanity: remove negative OI records
- Premium bounds: remove strikes with unrealistic premium (100x+ of ATM)
- Flag extreme values for review
- Output: cleaned_snapshot (dict | None), anomaly_flags (list)

packet_cleaner.py:
- General packet anomalies: empty payload, malformed JSON, encoding issues
- Remove obviously broken packets, flag for review
- Output: cleaned_packet (dict | None), anomaly_flags (list)

anomaly_repair.py:
- Interpolate minor gaps (1-2 missing ticks between valid ticks)
- Flag major gaps (3+ missing ticks) — do NOT interpolate
- Repair small timestamp misalignments (within 50ms)
- Output: repaired (bool), repair_action (str | None), original_values (dict)

cleaned_layer_writer.py:
- Write validation-passed data as cleaned records
- Write anomaly-repaired data with repair metadata
- Write removed packets to quarantine store (for audit)
- Output: cleaned records → structuring layer
- Architecture rule: cleaned layer = reusable, not just a scratchpad

STRUCTURING LAYER (structuring/):

tick_stream_builder.py:
- Group validated ticks into continuous stream
- Timestamp-ordered sequence with sequence_id
- Stream metadata: start_time, end_time, tick_count, gaps
- Output: TickStream (structured, queryable)

candle_stream_builder.py:
- Build 1m OHLCV candles from tick stream
- OHLCV: open, high, low, close, volume for each 1m window
- Track candle completeness (partial vs full minute)
- Validate candle: high >= low, open/close within [high, low]
- Output: CandleStream(1m candles), CandleMetadata per candle

options_snapshot_builder.py:
- Build OI snapshots at configurable intervals (default: 5m)
- Group by expiry, strike, option type (CE/PE)
- Fields: strike, OI, premium, IV, delta, change_in_OI
- Output: OptionsSnapshot per interval

session_packet_builder.py:
- Build explicit session packets: opening, mid-session, closing
- Session boundaries based on market hours (9:15-3:30)
- Fields: session_id, session_type, session_phase, session_status
- Broader references: Asia, London, New York if applicable
- Output: SessionPacket per session phase change

major_minor_classifier.py:
- Classify each feed as Major or Minor
- Major: tick data, candle streams, options chain, OI snapshots, core market feeds
- Minor: support feeds, auxiliary feeds, secondary references, slower contextual
- Effects applied to: storage priority, processing priority, validation strictness,
  replay importance, retention behavior, output routing priority
- Output: classification record per feed

structured_writer.py:
- Write all structured outputs to structured store
- Coordinate writes in dependency order (ticks → candles → options → session)
- Ensure atomicity per time window

METADATA SIDE-CHANNEL (metadata/):

packet_metadata_builder.py:
- Build packet-level metadata: received_at, envelope fields, packet_size, source
- Indexed by packet_id for fast lookup
- Available for dashboard debugging

source_trace_builder.py:
- Track source lineage through pipeline stages
- Minimum: source, fetched_at, validated_at, review_status, transform_stage
- Enables: "where did this packet come from? what stage did it pass through?"
- Output: traceability record per packet

transform_stage_tracker.py:
- Track raw → validated → cleaned → structured for each packet
- Timestamp each stage completion
- Flag packets stuck at a stage (not progressing)
- Output: stage history per packet

quality_fact_builder.py:
- Build factual quality metrics (NOT judgmental)
- Fields: raw_trust_level (0-1), validation_confidence (0-1),
  packet_completeness (0-1), continuity_status (GOOD/MINOR_GAP/MAJOR_GAP),
  source_health_state (HEALTHY/DEGRADED/STALE/AUTH_FAILED/DISCONNECTED)
- These are QUALITY FACTS, not intelligence
- Output: QualityFacts per packet/stream

retention_metadata_builder.py:
- Track retention decisions: class (Major/Minor), retention class, deletion date,
  archive flag
- Support audit trail for data lifecycle
- Output: RetentionRecord per data class

REVIEW ENGINE (review/):

continuous_review_engine.py:
- Ongoing health monitoring of data quality
- Feed health check, packet loss detection, continuity check, source degradation
- Runs on every batch of ingested data
- Output: health_events, quality_flags, review_signals

scheduled_audit_engine.py:
- Scheduled governance checks: raw vs structured verification, retention verification,
  audit reports, quality summaries
- Configurable schedule (daily, weekly)
- Output: audit_reports, verification_results

event_triggered_review_engine.py:
- Triggered on: packet corruption spike, source degradation, validation failure burst,
  abnormal continuity break cluster
- Deep analysis on triggered events
- Output: incident_reports, investigation_summaries

raw_vs_structured_verifier.py:
- Compare original raw data with structured output for selected samples
- Verify no data loss during transformation
- Output: verification_score (0-1), discrepancies (list)

retention_verifier.py:
- Verify retention policy is being followed
- Check: data not deleted early, archive process running, old data expiring correctly
- Output: compliance_status, violations (list)

audit_report_writer.py:
- Generate human-readable audit reports
- Include: quality summaries, health trends, verification results, incident log
- Output: structured audit reports

review_signal_builder.py:
- Convert review outcomes into 4-level transportable signal:
  GOOD / CAUTION / DEGRADED / CRITICAL
- Route through metadata side-channel to upper floors
- Captain receives ONLY light signal through upper-floor summary paths
- NOT a direct Captain decision feed

REPLAY ENGINE (replay/):

raw_replay_engine.py:
- Replay raw packets from original_raw_store
- Support time range queries (start_time, end_time)
- Support source + feed_type filters
- Output: exact same packet sequence as original ingest

cleaned_replay_engine.py:
- Replay cleaned packets from cleaned store
- Include repair/removal metadata where applicable
- Support validation result filters (e.g., replay only VALIDATED packets)
- Output: cleaned packet stream

structured_replay_engine.py:
- Replay structured streams (candles, tick streams, options snapshots)
- Support structured-level time range queries
- Output: same structured format as real-time pipeline

replay_contracts.py:
- Define replay query interface: time range, filters, output format
- Version the replay contracts for future compatibility
- Document replay use cases: debugging, testing, verification

replay_controller.py:
- Manage replay sessions
- Coordinate raw → cleaned → structured replay order
- Ensure replay produces identical output (bit-for-bit for raw)
- Track replay session metadata (session_id, user, purpose)

OUTPUT SUB-SYSTEM (output/):

floor3_handoff_builder.py:
- Assemble Floor 3 handoff contract: ALL 7 output categories
- validated_tick_stream, validated_candle_streams, options_snapshots,
  session_packets, macro_support_packets, metadata side-channel, computed-ready hooks
- Verify every mandatory field before handoff
- Raise ContractViolationError on missing fields

metadata_sidechannel_builder.py:
- Package quality facts, traceability, transform_stage, review signal for transport
- Route through side-channel (not in main data stream)
- Keep side-channel lightweight (factual metadata only, no raw data)

session_stream_router.py:
- Route per-session structured outputs to correct consumers
- Opening session, mid-session, closing session routing
- Support session context propagation

review_status_router.py:
- Route review signals to appropriate consumers
- Light signal to Captain summary path
- Detailed reports to Side B (dashboard) and Side C (memory)

datacenter_output_gateway.py:
- Central output point for all Floor 2 outbound data
- Govern output contracts, enforce schema compliance
- Log all outbound data for audit

GOVERNANCE SUB-SYSTEM (governance/):

registry_loader.py:
- Load data contract registry from config/registry definitions
- Validate registry format
- Support hot-reload for contract updates

runtime_contract_checks.py:
- Runtime enforcement of contract compliance on key feeds
- Check all outbound data against registered contracts
- Log violations, raise ContractViolationError on critical mismatches

source_policy_registry.py:
- Define policies per source: allowed feed types, validation tiers, retention classes
- Enforce source-specific rules at ingress
- Support adding new sources without code changes

retention_policy_registry.py:
- Define retention policies per data class
- Configuration-driven TTLs
- Support archive path configuration

STEP 2.4 — VALIDATION PIPELINE (validation/)
----------------------------------------------------------------------
Files: floor_2_datacenter/validation/validation_router.py
       floor_2_datacenter/validation/duplicate_validator.py
       floor_2_datacenter/validation/timestamp_validator.py
       floor_2_datacenter/validation/continuity_validator.py
       floor_2_datacenter/validation/schema_validator.py
       floor_2_datacenter/validation/corruption_validator.py

Purpose: Validate data quality — catch duplicates, bad timestamps,
         continuity gaps, schema violations, corrupt data.

Build order within this step:
2.4.1 — duplicate_validator.py (most critical — prevent reprocessing)
2.4.2 — timestamp_validator.py (temporal ordering)
2.4.3 — continuity_validator.py (gap detection)
2.4.4 — schema_validator.py (field compliance)
2.4.5 — corruption_validator.py (OHLC/price sanity)
2.4.6 — validation_router.py (orchestrates all validators)

Input contract:
- From raw storage sub-system: normalized raw envelope

Output contract (per validator):
- Each validator produces: ValidationResult (validator_name, passed, details, confidence)
- validation_router produces: aggregate validation decision (PASS/FAIL/FLAG)

Dependencies: Step 2.3 (raw storage for source data)
Validation:
- duplicate_validator: 10 identical packets → 9 detected as duplicates
- timestamp_validator: out-of-order packet → flagged with timestamp_delta_ms
- continuity_validator: insert 5s gap → MAJOR_GAP detected
- schema_validator: packet missing mandatory field → schema_compliant=False
- corruption_validator: high < low OHLC → is_corrupt=True
- validation_router: Tier A → all 5 validators run; Tier C → 2 validators run
- Dynamic validation by data type works correctly

----------------------------------------------------------------------
STEP 2.5 — CLEANING LAYER (cleaning/)
----------------------------------------------------------------------
Files: floor_2_datacenter/cleaning/tick_cleaner.py
       floor_2_datacenter/cleaning/options_cleaner.py
       floor_2_datacenter/cleaning/packet_cleaner.py
       floor_2_datacenter/cleaning/anomaly_repair.py
       floor_2_datacenter/cleaning/cleaned_layer_writer.py

Purpose: Clean validated data — remove artifacts, repair minor anomalies,
         preserve audit trail of all removals/repairs.

Build order:
2.5.1 — tick_cleaner.py (most common — tick artifacts are frequent)
2.5.2 — packet_cleaner.py (general packet anomalies)
2.5.3 — options_cleaner.py (options-specific cleaning)
2.5.4 — anomaly_repair.py (interpolation of minor gaps)
2.5.5 — cleaned_layer_writer.py (writes cleaned output)

Input contract:
- Validated packets from validation pipeline
- Validation results indicating which packets passed/failed

Output contract:
- Cleaned packets (validation-passed + anomaly-repaired)
- Quarantine records (removed packets with removal_reason)
- Cleaning metadata (what was removed/repaired and why)

Dependencies: Step 2.4 (validation results determine what gets cleaned)
Validation:
- tick_cleaner: zero-volume tick → removed, flagged for audit
- options_cleaner: negative OI → removed
- packet_cleaner: empty payload → removed
- anomaly_repair: single missing tick between two valid ticks → interpolated
- anomaly_repair: 5 consecutive missing ticks → NOT repaired (gap flagged)
- cleaned_layer_writer: cleaned packets written, removed packets quarantined
- Original raw is NEVER modified — cleaning creates fresh cleaned records

STEP 2.6 — STRUCTURING LAYER (structuring/)

       floor_2_datacenter/structuring/structured_writer.py

Purpose: Transform cleaned data into structured, reusable formats.

Build order:
2.6.1 — major_minor_classifier.py (classification needed by all other builders)
2.6.2 — tick_stream_builder.py (ticks → structured tick stream)
2.6.3 — candle_stream_builder.py (ticks → 1m OHLCV candles)
2.6.4 — options_snapshot_builder.py (OI snapshots)
2.6.5 — session_packet_builder.py (session boundaries)
2.6.6 — macro_support_builder.py (VIX, FII/DII, global cues, calendar)
2.6.7 — structured_writer.py (persists all structured outputs)

Input contract:
- Cleaned packets from cleaning layer

SECTION 9 - CROSS-COMPONENT IMPACTS

REPLAY IMPACT:
- Floor 2 OWNS replay capability — it is a FIRST-CLASS responsibility
- 3 replay levels: raw, cleaned, structured — each independently replayable
- Replay must be deterministic (identical output from identical input)
- Floor 3+ testing depends on Floor 2 replay being available and correct
- Replay is read-only — it NEVER modifies system state
- Replay sessions are tracked via replay_controller (session_id, user, purpose)
- Replay output must match real-time pipeline output for same input
- This enables: debugging, testing, verification, forensic review
- Floor 2's replay architecture enables ALL upper floors to be tested against
  known historical data without connecting to live market

DASHBOARD IMPACT:
- Floor 2 exposes factual operational data for dashboard, NOT judgments:
  - Data health light signal (GOOD/CAUTION/DEGRADED/CRITICAL)
  - Validation/review summary facts (pass rates, anomaly counts)
  - Source health summary (lifecycle states across all sources)
  - Replay availability markers (replay-ready data ranges)
  - Structured truth availability markers (which feeds are producing)
  - Traceability data (packet lineage for debugging)
- Dashboard must NOT treat these as trading signals
- Dashboard may show Floor 2 health in Health Surface panel
- Detailed review reports go to Side B and Side C, NOT directly to Captain

MEMORY IMPACT (Side C):
- Floor 2 is an approved emitter for Side C:
  - Data health events (quality degradation events, health state changes)

  - Replay references (replay session records for later lookup)
  - Structured truth availability markers
  - Review light signal references
- Floor 2 remains the truth/governance owner of those facts
- Side C remains storage/memory owner only — it stores what Floor 2 emits
- This preserves exact separation: Floor 2 = truth owner, Side C = memory owner

EXECUTION IMPACT (Side A):
- Floor 2 does NOT interface directly with Side A

  - DataHealth signal from Floor 2 → Side A policy (DEGRADED: cautious, STALE: block new)

- Floor 2 does NOT make any execution decisions

SECTION 10 - COMMON MISTAKES + ANTI-PATTERNS
================================================================================

WHAT CAN BREAK ARCHITECTURE IN THIS FLOOR:

Mistake 1: Creating intelligence-like scores in metadata
- Problem: Adding fields like "feed_trade_worthiness" or "market_quality" to
  quality_fact_builder.py
- Architecture violation: Floor 2 produces FACTS, not judgment. "This feed is
  good for trading" is Floor 4 territory.
- Prevention: Quality facts must be descriptive (packet_completeness=96%),
  never prescriptive (this data supports a trade).
- Detection: Any quality_fact field with "recommend", "suggest", "trade",
  "entry", "favorable" → P0 bug.

Mistake 2: Building Floor 3 logic inside Floor 2 structuring layer
- Problem: Adding setup detection, zone analysis, or domain confidence in
  candle_stream_builder or tick_stream_builder
- Architecture violation: Floor 2 structures data. Floor 3 creates intelligence.
- Prevention: If you're writing "if breakout_detected" or "if FVG found" in
  Floor 2 → stop. That's Floor 3 code.
- Detection: Any "setup", "signal", "bias", "zone_type" field in Floor 2 output.

Mistake 3: Computing confidence/conviction in Floor 2
- Problem: Adding confidence_score or conviction_level to quality_facts
- Architecture violation: Confidence = Floor 4. Conviction = Floor 5.
- Prevention: Floor 2 quality = factual (0-1 scale describing process confidence,
  NOT market confidence).
- Detection: Any field named "confidence" or "conviction" in floor_2_datacenter/.

Mistake 4: Making Floor 2 depend on Floor 3+ types
- Problem: Importing Floor 3 types/classes in Floor 2 code
- Architecture violation: Bottom-up dependency. Floor 2 should only depend on
  shared/ types.
- Prevention: ALL Floor 2 imports come from shared/. No floor_3_* imports.
- Detection: `grep "from junior_aladdin.floor_3" floor_2_datacenter/` returns nothing.

Mistake 5: Adding setup/trade detection in review engine
- Problem: Review engine flags "setup detected" or "trade opportunity" as
  review event
- Architecture violation: Review engine = governance/truth. Setups = Floor 3+.
- Prevention: Review events describe data health anomalies, NOT market setups.
- Detection: Review event types include market interpretation fields.

Mistake 6: Not implementing replay engine in Phase 2
- Problem: Deferring replay to "later" because it's not urgent
- Architecture violation: Replay is FIRST-CLASS and must be built during Phase 2.
  Retrofitting replay later is exponentially harder.
- Prevention: Replay engine built during Phase 2. Replay must be operational
  before Phase 2 exit criteria are considered met.
- Detection: Phase 2 exit criteria checklist doesn't include replay.

Mistake 7: Forgetting dual-raw model
- Problem: Storing only normalized raw without original raw
- Architecture violation: Original source truth lost. Debugging impossible.
- Prevention: Both original_raw_store and normalized_raw_store must exist.
  Original raw = EXACT copy. Normalized raw = additive only.
- Detection: No original_raw_store table/collection.

Mistake 8: Applying same validation depth to all data types
- Problem: Running all 5 validators on macro support packets (wasteful)
- Architecture violation: Dynamic validation by tier exists to optimize
  performance. Tier C data needs only 2 validators.
- Prevention: validation_router correctly classifies Tier A/B/C and routes
  accordingly.
- Detection: All data types running through same validator count.

Mistake 9: Making storage schema hard to extend
- Problem: Rigid SQLite schema that requires migration for new feed types
- Architecture violation: Floor 2 should support new data types without
  schema changes.
- Prevention: Use JSON fields for payload, normalized envelope has extensible
  metadata fields.
- Detection: Adding a new feed type requires schema migration.

WHAT CREATES REWORK LATER:
- Not implementing replay engine now → exponentially harder to retrofit
  Fix: Implement replay in Phase 2, Steps 2.10.1-2.10.5
- Not tracking transform_stage → debugging becomes detective work
  Fix: Track each packet's stage in transform_stage_tracker.py
- Forgetting dual-raw model → losing original source truth for audit
  Fix: Both original_raw_store + normalized_raw_store in raw/ sub-system
- Applying same validation depth to all data types → wasted CPU on Tier C
  Fix: validation_router correctly routes by ValidationTier
- Making storage schema rigid → painful migration for new feeds
  Fix: Flexible schema with JSON payload fields + metadata
- Not preserving original raw alongside normalized raw → audit failure
  Fix: Dual Raw model enforced from day one
- Letting Floor 2 accumulate intelligence code → architecture blurs
  Fix: Strict code review + automated linting for Floor 2 boundary rules
- Not defining contracts before coding → integration mismatches later
  Fix: Step 2.1 (types + contracts) built before any pipeline code
- Building validators in wrong order → missing interdependencies
  Fix: Build order: duplicate → timestamp → continuity → schema → corruption
