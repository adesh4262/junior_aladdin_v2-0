SECTION 1: PURPOSE
================================================================================

Floor 3 is the calculation layer of the Junior Aladdin system. It consumes
validated, cleaned, and structured data from Floor 2 Data Center and applies
three independent calculation domains:

1. SMC (Smart Money Concepts) — Market structure, FVG, Order Blocks, CHOCH
2. ICT (Inner Circle Trader) — PD Arrays, Kill Zones, Liquidity concepts
3. Technical — RSI, MA, ATR, Volume Profile, standard indicators

Floor 3 produces calculated signals (NOT interpretations, NOT setups, NOT
decisions). Each output is:
- Deterministic: same input → same output, always
- Repeatable: can be re-run in replay mode with identical results
- Auditable: every output carries a calculation_log with full provenance

Floor 3 outputs flow exclusively to Floor 4 (Department Heads). Floor 3 does
NOT communicate with Captain, Side A, or Side B directly.

CRITICAL ARCHITECTURE RULES:
Rule 1: Floor 3 NEVER interprets — it only calculates
Rule 2: Floor 3 NEVER outputs confidence, conviction, scores, or rankings
Rule 3: Floor 3 NEVER outputs setups or trade signals
Rule 4: Floor 3 is domain-agnostic — it does not know what the signals mean
Rule 5: Every calculation is immutable — signal_id never changes after creation
Rule 6: Floor 3 has zero dependency on Side C, Side A, Floor 4, Floor 5
Rule 7: Floor 3 must be fully functional in replay mode (no live data needed)
Rule 8: No engine imports from another engine — SMC does not call ICT

STEP 3.7 — f3_orchestrator.py (Depends on Steps 3.4-3.6)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

              f3_contracts, f3_config, f3_ingress, f3_output_builder
Imports from: All above modules

Responsibilities:
- receive_data(packets: list) → routes to appropriate domain engine(s)
- route_to_domain(data: CalculationInput, domain: CalculationDomain) → EngineRunReport
- collect_results(reports: list) → list of CalculatedSignal
- handle_calculation_cycle() → Floor3Summary

Routing logic:
- Market phase determines which domains to activate:
  - PRE_MARKET: SMC + ICT (structure + kill zone preparation)
  - REGULAR: ALL domains active
  - POST_MARKET: SMC + Technical (structure update + analysis)
  - CLOSED: No calculations (return empty summary)
- Data type determines engine:
  - tick_stream → Technical (RSI, MA, ATR need tick/1m data)
  - candle_stream → ALL domains
  - options_snapshot → Technical (volume profile only)
- Each engine run is independent — error in one does not block others
- Timeout protection: if engine exceeds calculation_timeout_ms, mark as ERROR

Validation:
- All engines must return EngineRunReport before orchestrator proceeds
- Empty engine report (no signals) is valid — not an error
- Floor3Summary always produced, even if all engines return empty

STEP 3.9 — f3_validator.py (Depends on Steps 3.4-3.8)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Dependencies: f3_types.py, f3_contracts.py
Imports from: f3_types, f3_contracts, f3_output_builder

Validation layer runs AFTER calculation, BEFORE output dispatch to Floor 4.

Validation checks:
1. Output contract compliance:
   - Every CalculatedSignal has signal_id (UUID, not None, not empty)
   - Every CalculatedSignal has CalculationLog
   - Every CalculationLog has input_hash
   - Floor3Summary is not None

2. Domain isolation check:
   - No SMC signal has ICT metadata
   - No ICT signal has Technical metadata
   - Domain field matches actual engine domain

3. Quality validation:
   - NOMINAL: full data, no warnings, clean calculation
   - DEGRADED: partial data, some warnings, calculation completed
   - INSUFFICIENT_DATA: below min_data_points_per_calculation

4. Determinism check (replay mode only):
   - signal_id matches expected (re-run produces same signals)
   - calculation_log matches expected
   - input_hash matches stored input hash

5. Contract boundary check:
   - No Floor 4 types leaked into output
   - No Captain types leaked into output
   - No Side A types leaked into output

SECTION 7: DATA FLOW
================================================================================

LIVE MODE FLOW:

1. Floor 2 produces validated structured data stream
2. f3_ingress polls Floor 2 output (push or poll — TBD at integration phase)
3. f3_ingress validates each packet against InputContract
4. f3_ingress converts packets to CalculationInput list
5. f3_orchestrator receives CalculationInput list
6. Orchestrator checks current market phase (from Floor 2 Session Packets)
7. Orchestrator activates domain engines based on market phase + data type
8. Each engine runs independently with its sub-calculators
9. Engine returns EngineRunReport with CalculatedSignal list
10. f3_output_builder collects all engine outputs
11. Output builder generates signal_id for each signal
12. Output builder builds CalculationLog with input_hash
13. Output builder builds Floor3Summary
14. f3_validator checks all outputs for contract compliance
15. Validated outputs dispatched to Floor 4
16. HEALTH_EVENT emitted to Side C

REPLAY MODE FLOW:

1. Replay orchestrator sets f3_replay_adapter.replay_mode_active = True
2. f3_ingress detects replay mode — reads from f3_replay_adapter
3. f3_replay_adapter reconstructs Floor 2 data from Side C REPLAY_REF events
4. Same flow as live mode (Steps 4-16), but with replay mode flag
5. f3_validator checks determinism — outputs must match recorded outputs exactly
6. Any mismatch = replay failure (logged, system alerted)

KEY DATA RULES:
- Each engine calculation cycle is atomic — all 3 engines run or none
- Engines do NOT share data — SMC results are not available to ICT
- Signals flow one direction only: Floor 3 → Floor 4 (no reverse flow)
- Empty signal list is valid (market closed → no calculations needed)
- input_hash enables Floor 4 to verify data provenance without re-running calculations

SECTION 9: CROSS-COMPONENT IMPACTS

IMPACT ON REPLAY:

Floor 3 replays entirely through Floor 2 replayed data. When replay mode is
active:
- f3_ingress reads from f3_replay_adapter (reconstructed PacketEnvelope from Side C)
- All engines run identically (pure functions, deterministic)
- f3_validator checks determinism: signal_id, input_hash, and output MUST match
- Floor 3 does NOT need its own replay store — it reuses Floor 2 replay stream

Key constraint: Floor 3 replay depends on Side C REPLAY_REF store being populated
with Floor 2 data. Without Floor 2 replay data, Floor 3 cannot verify replay.

IMPACT ON DASHBOARD (Side B):

Floor 3 does NOT send data to Side B directly. Side B sees Floor 3 signals
through Floor 4's summarized output. Dashboard can display:
- Active signals per domain (count, type, quality)
- Engine status (IDLE, CALCULATING, ERROR)
- Floor 3 health (via HEALTH_EVENT → Side C → Side B read model)

Dashboard does NOT display raw Floor 3 calculation logs or intermediate results.

IMPACT ON MEMORY (Side C):

Floor 3 contributes:
- HEALTH_EVENT: engine status changes, validation failures, replay results
- No other event types — Floor 3 does not create TRADE_JOURNAL, DECISION_JOURNAL,
  EXECUTION_EVENT, OVERRIDE, BLOCKED_ACTION, or REVIEW_REF events

IMPACT ON EXECUTION (Side A):

Floor 3 does NOT communicate with Side A. All Floor 3 outputs pass through
Floor 4 (interpretation) and Floor 5 (decision) before reaching Side A.

Floor 3 is intentionally isolated from execution to maintain the separation
between calculation and decision layers.

SECTION 10: COMMON MISTAKES + ANTI-PATTERNS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ARCHITECTURE-BREAKING MISTAKES:

MISTAKE 1: Engine doing interpretation instead of calculation
  - WRONG: smc_engine returns "bullish_fvg_setup" or "buy_signal"
  - RIGHT: smc_engine returns "BULLISH_FVG at price X, timestamp Y"
  - PREVENTION: Every engine output reviewed for interpretive language
  - DETECTION: CI regex check: reject output fields containing "setup", "signal" (in trading sense), "trade"

MISTAKE 2: Engine outputting confidence, conviction, or score
  - WRONG: fvg_calculator returns "confidence: 0.85" or "strength: STRONG"
  - RIGHT: fvg_calculator returns "fvg_type: BULLISH_FVG, gap: 2.5 pips, mitigated: False"
  - PREVENTION: Quality field only in CalculatedSignal (NOMINAL/DEGRADED/INSUFFICIENT_DATA)
  - DETECTION: CI check: no "confidence", "conviction", "score", "rank", "strength" in output

MISTAKE 3: Engine producing setups or trade suggestions
  - WRONG: ict_engine returns "kill_zone_entry: BUY at price X"
  - RIGHT: ict_engine returns "active_kill_zone: LONDON_OPEN, asian_range: {high, low}"
  - PREVENTION: Architect code review — every output field must be factual, not suggestive

MISTAKE 4: Engine importing from another domain
  - WRONG: smc/ob_calculator.py imports from ict/pd_array_calculator.py
  - RIGHT: Each domain is completely independent
  - PREVENTION: CI check: no imports between smc/, ict/, technical/ directories

MISTAKE 5: Engine depending on Floor 4+ or Side A/B
  - WRONG: smc_engine.py imports from floor_4 or side_a
  - RIGHT: Floor 3 has zero dependency on anything above it
  - PREVENTION: CI check: no imports from floor_4, floor_5, side_a, side_b

MISTAKE 6: signal_id being mutable or re-generated
  - WRONG: Regenerating signal_id on replay (breaks determinism)
  - RIGHT: signal_id is UUID v4 generated once, never changed
  - PREVENTION: f3_validator checks signal_id stability in replay mode

MISTAKE 7: CalculationLog missing input_hash
  - WRONG: Skipping input_hash because "replay not needed yet"
  - RIGHT: input_hash ALWAYS computed, always stored, from Day 1
  - PREVENTION: f3_validator.reject_if_missing_input_hash()

MISTAKE 8: Engine using shared state between calculation cycles
  - WRONG: smc_engine caching previous results across cycles
  - RIGHT: Each calculation cycle is independent, pure functions throughout
  - PREVENTION: Code review: no class-level state, no module-level mutable globals

WHAT CREATES REWORK LATER:

1. Skipping contract validation at ingress — "Floor 2 data is always clean"
   → Fix: validate every packet at ingress, even in development
   → Why: hides bugs that surface in integration as "Floor 3 wrong" when data was bad

2. Hardcoding calculation parameters in engine code
   → Fix: ALL parameters in f3_config.py, loaded at runtime
   → Why: tuning requires code changes instead of config changes

3. Writing non-deterministic calculations (using current time, random, etc.)
   → Fix: inject timestamp, use seed-free deterministic math
   → Why: replay fails, debugging impossible, audit broken

4. Skipping CalculationLog for "simple" calculations (RSI, MA)
   → Fix: EVERY signal gets a CalculationLog, no exceptions
   → Why: Floor 4 needs audit trail for every signal, including "obvious" ones

5. Letting one engine crash the entire Floor 3
   → Fix: try/except around EVERY engine.run() call, independent engine execution
   → Why: SMC error should not block ICT signals

6. Not testing with empty data during development
   → Fix: test Floor 3 with zero candles, one candle, missing fields
   → Why: market open/close edges crash untested code

7. Writing engines that assume specific timeframe or data availability
   → Fix: check data availability before calculation, emit warning if insufficient
   → Why: system adapts to available data, not crashes on missing data
