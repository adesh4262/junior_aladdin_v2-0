SECTION 1: PURPOSE
================================================================================

Floor 4 is the expert interpretation layer of the Junior Aladdin system. It
consumes calculated intelligence from Floor 3 (states, maps, scores, candidates)
across 6 domains and converts them into structured Captain-readable expert reports.

Floor 4 exists because:
- Captain should NOT interpret raw Floor 3 calculations directly
- Raw calculations lack domain-specific context, memory, and judgment
- Floor 4 reduces complexity before it reaches Captain
- Floor 4 adds explainability, invalidation, and freshness to every opinion

Floor 4 produces TWO output layers:
- Layer A (Individual Head Reports): detailed, structured, domain-specific
- Layer B (Floor Summary): compact, aggregate, quick-read intelligence layer

Floor 4 outputs flow EXCLUSIVELY to Floor 5 (Captain). Floor 4 does NOT
communicate with Side A, Side B, or Side C directly (though reports may be
referenced by Side B through Captain's output).

GOLDEN CHAIN (MUST NEVER BE VIOLATED):
QUALITY = FLOOR 3 (calculations, scores, maps)
CONFIDENCE = FLOOR 4 (Heads interpret, create confidence)
CONVICTION = FLOOR 5 (Captain judges, creates conviction)

CRITICAL ARCHITECTURE RULES:
Rule 1: Floor 4 creates CONFIDENCE, NOT conviction (conviction = Floor 5)
Rule 2: Heads INTERPRET, they do NOT recalculate (Floor 3 owns calculation)
Rule 3: Heads suggest OPPORTUNITY, they NEVER approve trade (Captain approves)
Rule 4: EVERY Head MUST produce invalidation (non-negotiable)
Rule 5: Macro Head — NO primary_setup, NO backup_setup (gate/context only)
Rule 6: Psychology Head — NO primary_setup, NO backup_setup (brake head only)
Rule 7: SMC/ICT Heads — context_quality_score MANDATORY
Rule 8: MAX 2 setups per actionable head (primary + backup)
Rule 9: Heads are INDEPENDENT — no head rewrites another head's opinion
Rule 10: Floor Summary by itself does NOT make decisions — Captain judges

Define common memory interface/patterns for all heads.

Key design decisions:
- Memory depth is HEAD-SPECIFIC (not uniform)
- SMC: longer structural memory (multi-session)
- Psychology: mainly intraday behavior memory
- Options: session + recent shift memory
- Macro: current day + prior carryover context
- Memory interface is shared, depth policies are head-specific

Common memory interface:
- remember(key, value, ttl) → store domain-relevant item
- recall(key) → retrieve stored item
- forget(key) → remove specific item
- clear_expired() → remove all expired items
- get_context() → return all active context for report generation
- Stale items must be removable explicitly

Validation:
- Memory items must have TTL
- Expired items must be cleared before report generation
- Memory should NOT bloat — keep only domain-relevant state

Define common refresh rules that align with Captain's two-speed architecture:

A. 1-MINUTE BASE REFRESH
- Keep Captain updated at intraday speed
- Refresh tactical interpretation, setup relevance, confidence, invalidation
- Aligns with Captain's heavy cycle (1m candle close)

B. 5-MINUTE / 15-MINUTE DEEP REFRESH
- Deeper structural refresh
- Stronger zone validity update
- Bigger timeframe confirmation refresh
- Not every head rethinks structure every 1m (noise reduction)

C. TICK-LEVEL LIGHT WATCH
- Trigger watch only
- Invalidation touch awareness
- Micro confirmation watch for already-known setups
- NOT full thinking on every tick (performance + noise)

RefreshPolicy dataclass:
- base_refresh_interval_seconds: 60 (1m)
- deep_refresh_interval_seconds: 300 (5m) or 900 (15m) — head-specific
- tick_watch_enabled: bool
- min_ticks_between_refresh: int (for tick watch throttling)
- stale_after_seconds: int (when state becomes STALE without update)

Output: RefreshPolicy for each head type

Manage head state lifecycle:
- State: READY / UNCERTAIN / STALE
- Freshness score + tag computation

State transition rules:
- READY: last_deep_update within freshness window, no internal conflict
- UNCERTAIN: conflicting signals, borderline freshness, partial data
- STALE: last_deep_update beyond stale threshold, degraded data

Freshness computation:
- freshness_score: 0.0 (stale) to 1.0 (fresh)
- freshness_tag: FRESH (score > 0.7), WARM (0.3-0.7), STALE (< 0.3)
- last_deep_update: timestamp of last meaningful refresh

State + Freshness interaction:
- READY + FRESH/WARM → normal trust
- UNCERTAIN + any → reduced interpretive weight
- STALE + STALE → caution, reduced trust, possible block per head type

For Captain's weighted confluence:
- State/freshness conditions influence how confidently a head can speak
- BUT confidence ownership remains with Floor 4 (not determined by freshness alone)

Standard invalidation object creation.
Invalidation is MANDATORY in ALL heads (non-negotiable).

Invalidation creation:
- create_invalidation(reason, conditions, severity) → InvalidationObject
- check_invalidation(invalidation, market_data) → triggered (bool), triggered_at
- merge_invalidations(list) → combined invalidation summary

Why mandatory: Captain's thesis integrity tracking and active trade supervision
depend on invalidation. Without it, a head says "I like this" but never says
"I am wrong if this breaks."

Build the Floor Summary — compact aggregate intelligence for Captain.

How Floor Summary is created:
1. Collect all head states → ready_heads_count, uncertain_heads_count,
   stale_heads_count
2. Collect all head primary/backup setup summaries → active_setup_count,
   primary_setups_by_head, backup_setups_by_head
3. Collect bias alignment and confidence shape → floor_bias_snapshot,
   floor_confidence_snapshot
4. Collect stale/conflict warnings → conflict_present, stale_warning_present
5. Build head_health_snapshot (per-head: state + freshness_tag)
6. Build core_head_health_snapshot (SMC, ICT, Technical, Macro, Psychology)
7. Identify strongest signals → strongest_domain_signal,
   strongest_context_signal, strongest_risk_warning
8. Evaluate no_setup intelligence → no_setup_context list
9. Compress into Captain-readable FloorSummary

NO_SETUP intelligence propagation:
- READY_NO_SETUP: head is healthy, no valid setup exists (healthy silence)
- UNCERTAIN_NO_SETUP: no valid setup, interpretation unstable/conflicted
- STALE_NO_SETUP: no valid setup, report health degraded/stale

When Captain uses Floor Summary first:
- routine heavy cycle start
- quick first-pass understanding
- fast check whether deep drill-down is necessary

When Captain drills down into individual reports:
- conflict present
- strong conviction candidate exists
- stale core head exists (especially SMC/ICT)
- context quality issue present
- intervention review required
- opposite case appears significant

Output: FloorSummary dataclass (all mandatory fields populated)

SECTION 7: DATA FLOW
================================================================================

HEAVY CYCLE (1m candle close — aligns with Captain's heavy cycle):

1. Floor 3 produces domain outputs for current 1m candle
2. Each head receives its specific Floor 3 inputs (states, maps, scores, candidates)
3. Head checks refresh_policy: is it time for base or deep refresh?
4. Head updates domain memory (head_memory_base)
5. Head performs interpretation:
   a. Read Floor 3 domain states
   b. Evaluate against head's domain knowledge
   c. Check memory for relevant context
   d. Assess setup quality (context_quality_score for SMC/ICT)
6. Head selects primary/backup setups (or determines no-setup)
7. Head creates invalidation (MANDATORY for all heads)
8. Head updates state + freshness (head_state_manager):
   - Compute freshness_score, freshness_tag
   - Determine state: READY/UNCERTAIN/STALE
9. Head produces HeadReport with all mandatory fields
10. head_report_schema.validate(report) — contract check
11. floor_summary_builder collects all 6 head reports
12. Summary builder computes: bias, confidence, conflict, stale, strongest signals
13. Summary builder builds head_health_snapshot + core_head_health_snapshot
14. Summary builder propagates NO_SETUP intelligence
15. FloorSummary + 6 HeadReports dispatched to Captain

LIGHT CYCLE (tick level — align with Captain's light cycle):

1. Head checks whether tick triggers any armed trigger
2. Head checks whether tick touches any invalidation condition
3. Head updates freshness_score (slight decay on each tick without refresh)
4. NO full interpretation, NO setup selection, NO memory deep update
5. Only trigger watch + invalidation awareness + freshness decay

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CAPTAIN CONSUMPTION MODEL:

Step 1 — Read Floor Summary FIRST (summary-first workflow)
- Quick overall state
- Conflict detection
- Stale warnings
- Strongest domain signal snapshot

Step 2 — Drill down into individual reports ONLY when needed:
- Conflict present
- Strong conviction candidate found
- Core head stale (especially SMC/ICT)
- Context quality low or unusual
- Intervention review needed
- Opposite case seems strong

SECTION 9: CROSS-COMPONENT IMPACTS
================================================================================

IMPACT ON REPLAY:

Floor 4 is NOT a replay owner. Replay truth comes from lower layers.
But Floor 4 must remain replay-compatible:
- Reports must be reconstructable from Floor 3 states/maps/scores/candidates
- Floor Summary logic must be deterministic from same head inputs
- Freshness/time-dependent fields must account for replay mode
- Same Floor 3 input → same Floor 4 output (determinism check)

Key constraint: Floor 4 replay depends on Floor 3 producing valid replay data.
Without Floor 3 replay data, Floor 4 outputs cannot be verified.

IMPACT ON DASHBOARD (Side B):

Floor 4 is one of the strongest feed layers for Side B:
- Floor Summary: bias, confidence, head health, conflict status
- Head Reports: state, freshness, primary/backup setups, invalidation
- Explainability: no-trade reasoning, confidence attribution, setup details
- Freshness visualization: which heads are fresh/warm/stale
- Conflict visualization: where heads disagree

Dashboard does NOT display raw Floor 3 calculation data through Floor 4 —
it uses Floor 4's interpreted outputs directly.

IMPACT ON MEMORY (Side C):

Floor 4 does NOT store memory. But Floor 4 outputs may be referenced by
Side C in review or decision contexts:
- Head report references in decision review
- No-trade reason summaries referenced
- Floor Summary references linked to Captain decision history
- HEALTH_EVENT: head state changes, validation failures

Floor 4 contributes:
- HEALTH_EVENT: head state transitions, stale warnings, errors

IMPACT ON EXECUTION (Side A):

Floor 4 does NOT communicate with Side A. All Floor 4 outputs pass through
Floor 5 (Captain) before reaching Side A.
- Heads suggest opportunity → Captain approves → Side A executes
- Heads provide invalidation → Captain monitors thesis → Side A manages trade

SECTION 10: COMMON MISTAKES + ANTI-PATTERNS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ARCHITECTURE-BREAKING MISTAKES:

MISTAKE 1: Head producing conviction instead of confidence
  - WRONG: smc_head returns conviction: 85 (belongs to Floor 5)
  - RIGHT: smc_head returns confidence: 0.75, context_quality_score: HIGH
  - PREVENTION: All head outputs reviewed for conviction-language
  - DETECTION: CI check: no "conviction" field in floor_4_heads output
  
MISTAKE 2: Macro Head becoming a directional setup machine
  - WRONG: macro_head returns primary_setup: {direction: "BUY"}
  - RIGHT: macro_head returns caution_level: LOW, event_risk_flag: False
  - PREVENTION: locked rule: Macro Head NEVER produces setups
  - DETECTION: CI check: macro_head output has no primary_setup field

MISTAKE 3: Psychology Head becoming a directional or entry engine
  - WRONG: psychology_head returns setup suggestions
  - RIGHT: psychology_head returns trade_allowed: True/False, block_reason
  - PREVENTION: locked rule: Psychology Head NEVER produces setups
  - DETECTION: CI check: psychology_head output has no setup fields

MISTAKE 4: SMC/ICT not producing context_quality_score
  - WRONG: smc_head returns report without context_quality_score
  - RIGHT: smc_head ALWAYS includes context_quality_score in report
  - PREVENTION: head_report_schema rejects if context_quality_score missing
  - DETECTION: contract validation test catches missing field

MISTAKE 5: Any head missing invalidation
  - WRONG: Any head report without invalidation field
  - RIGHT: EVERY head report includes invalidation (never None)
  - PREVENTION: head_report_schema rejects if invalidation is None
  - DETECTION: CI check: all head reports have invalidation

MISTAKE 6: Head duplicating Floor 3 calculations
  - WRONG: smc_head recalculates FVG from raw candles
  - RIGHT: smc_head consumes Floor 3's FVG map and interprets it
  - PREVENTION: Code review: no Floor 3 type calculations in head code
  - DETECTION: CI check: no imports from floor_3_calculations domain modules

MISTAKE 7: Cross-head coupling (one head depending on another)
  - WRONG: smc_head imports from technical_head for "context"
  - RIGHT: Each head is completely independent (pure isolation)
  - PREVENTION: Architecture: heads import from infra only, not other heads
  - DETECTION: CI check: no imports between head modules

MISTAKE 8: Floor Summary containing too much detail
  - WRONG: Floor Summary includes full head reports, raw zones, calculation logs
  - RIGHT: Floor Summary is compact snapshot (bias, confidence, health, conflict)
  - PREVENTION: FloorSummary schema limits fields to essential snapshot only

MISTAKE 9: Head using candidate_grade as confidence or conviction
  - WRONG: head sets confidence = candidate_grade value
  - RIGHT: head interprets candidate_grade as quality input, creates own confidence
  - PREVENTION: Code review: confidence derived from interpretation, not copy

WHAT CREATES REWORK LATER:

1. Not standardizing head report schema early
   → Fix: head_report_schema built FIRST, before any head module
   → Why: inconsistent reports = Captain parsing chaos

2. Making heads too complex before Captain tests them
   → Fix: build minimal viable head first, then enrich
   → Why: Captain's needs define what heads should provide, not theoretical richness

3. Skipping head_state_manager (no freshness awareness)
   → Fix: freshness tracked from day one in every head
   → Why: stale opinions used as fresh = dangerous decisions

4. Not defining NO_SETUP intelligence propagation
   → Fix: setup_absence_context included in report from day one
   → Why: Captain can't distinguish healthy silence from stale silence

5. Testing heads only with aligned data
   → Fix: test conflict scenarios (heads disagree), not just harmony
   → Why: Captain's conflict detection only works if tested

6. Building all heads before testing any against Captain
   → Fix: test one head through Captain flow before building all 6
   → Why: integration surprises multiply with each head added

7. Macro Head getting over-engineered with setups
   → Fix: locked rule with CI enforcement
   → Why: Macro as setup engine = Floor 4 identity drift

8. Not preserving replay compatibility (time-dependent logic)
   → Fix: inject timestamps, use deterministic freshness in replay mode
   → Why: replay breaks if heads depend on wall-clock time
