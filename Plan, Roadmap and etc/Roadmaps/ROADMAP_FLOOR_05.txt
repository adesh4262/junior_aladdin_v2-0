SECTION 1: PURPOSE
================================================================================

Floor 5 is the supreme decision brain of the Junior Aladdin system. Captain
consumes interpreted expert intelligence from Floor 4 (Floor Summary + 6 Head
Reports), applies weighted confluence, checks opposite case, builds conviction,
creates armed conditional plans, and outputs TRADE/WAIT/BLOCKED decisions.

Captain is NOT:
- A raw data collector, data cleaner, or indicator calculator
- An SMC/ICT/Options math engine
- A generic rule-bot that fires on simple signals
- An execution engine (Side A owns execution)

Captain IS:
- Final judge and trade permission authority
- Final confluence evaluator and conviction creator
- Final trade constructor and capital-discipline enforcer
- Strategic battlefield supervisor during active trades
- Final intervention authority in exceptional situations (RARE)

GOLDEN CHAIN (MUST NEVER BE VIOLATED):
QUALITY = FLOOR 3 (calculations, scores, maps)
CONFIDENCE = FLOOR 4 (Heads interpret, create confidence)
CONVICTION = FLOOR 5 (Captain judges, creates conviction)

Captain uses a TWO-SPEED BRAIN:
- HEAVY CYCLE (1m candle close): full decision analysis (23 steps)
- LIGHT CYCLE (tick level): watch armed plans, monitor active trade thesis

CRITICAL ARCHITECTURE RULES:
Rule 1: Captain owns CONVICTION, NOT confidence (confidence = Floor 4)
Rule 2: Captain does NOT recalculate quality or confidence — it judges them
Rule 3: Captain does NOT consume Floor 3 packets as routine input (via Floor 4 only)
Rule 4: Two-speed brain: heavy cycle on candle close, light cycle on tick
Rule 5: One active trade at a time (locked)
Rule 6: Psychology block is NON-OVERRIDABLE (locked)
Rule 7: 3 losing trades → REAL mode LOCK (locked)
Rule 8: Summary-first workflow: read Floor Summary first, drill-down if needed
Rule 9: Armed plans created after heavy cycle, watched by light cycle
Rule 10: Intervention is RARE, not routine
Rule 11: Captain decides, Side A executes
Rule 12: Silence (WAIT/BLOCKED) is a valid, actively reasoned output

FIRST module in heavy cycle. If permission fails, Captain goes to WAIT/BLOCKED.

8 checks (BLOCKED if ANY fails):
1. Market open check: not holiday, not after hours, not closed session
2. Psychology block check: trade_allowed from Psychology Head (NON-OVERRIDABLE)
3. Active trade check: no second trade if one exists (one-trade rule)
4. Data health check: data_health_signal not CRITICAL
5. Real mode lock check: if REAL mode and loss_lock_manager.is_locked() → block
6. Mode validation: valid mode selected
7. Session policy check: opening/closing window restrictions
8. Capital availability: capital available > 0

Locked rules:
- Psychology block CANNOT be overridden by Captain (non-negotiable)
- Active trade block CANNOT be overridden (one-trade rule)

Output: PermissionResult(allowed, block_reason, blocked_by list)

Combine directional head reports with weighted alignment logic.

Head priority (locked):
1. SMC (highest weight for structural truth)
2. ICT (next highest for institutional context)
3. Technical (support/confirmation)
4. Options (derivatives confirmation)
5. Macro (light context gate — lowest directional weight)

Trust weighting depends on 5 factors:
1. Head type priority (SMC > ICT > Technical > Options > Macro)
2. Head state: READY (normal), UNCERTAIN (reduced), STALE (minimal)
3. Freshness: freshness_score + freshness_tag
4. Context quality (SMC/ICT only): context_quality_score (HIGH/MEDIUM/LOW)
5. Invalidation clarity: clear vs ambiguous

Minimum meaningful alignment: ~60% OR 3 of 5 heads aligned
But NOT simple democracy — weighted (SMC+ICT opposing = stronger veto).

Output: ConfluenceResult(confluence_quality, conflict_present,
       aligned_heads, opposing_heads, dominant_direction)

Pre-mortem failure analysis. Checks strongest opposite argument before
committing to a trade.

Checks:
- Nearby resistance/support that invalidates thesis
- Weak volume or unconfirmed move
- Fake sweep risk
- Options wall opposition (strong wall against direction)
- Contradictory macro context
- Same-zone trap history (from setup_memory_store)
- Opposite structure case strength

If opposite case is too strong:
- Conviction reduces (>10% reduction)
- Trade can downgrade to WAIT or REJECT

Output: OppositeCase(exists, strength, reasons, mitigating_factors)

Builds three internal scores:
- permission_score (0-100): how permissive is the current environment
- conviction_score (0-100): how confident is Captain in this trade
- no_trade_score (0-100): how strong is the case for NOT trading

Inputs:
- PermissionResult (from permission_gate)
- MarketStory (from market_story_engine)
- ConfluenceResult (from confluence_engine)
- OppositeCase (from opposite_case_engine)
- Psychology report (trade_allowed, caution_level)
- Session policy (aggression_modifier)
- Setup memory (trap_count, same_zone_failures)

Conviction bands (locked):
- 0-39  = REJECT (no trade)
- 40-59 = WEAK / WATCH (monitor but no action)
- 60-74 = TRADABLE BUT PATIENT (trade possible, wait for confirmation)
- 75-89 = STRONG (high confidence, good to proceed)
- 90+   = ELITE (exceptional confluence, rare)

Output: ConvictionScore(permission_score, conviction_score,
       no_trade_score, conviction_band)

Determines Captain mood from current context:

- PATIENT: moderate confidence, waiting for better entry
- AGGRESSIVE: high confidence, strong confluence, good session
- DEFENSIVE: active trade exists, market choppy, recent loss

SECTION 7: DATA FLOW
================================================================================

HEAVY CYCLE FLOW:

1. Market produces new 1m candle
2. Floor 3 calculates intelligence from new candle
3. Floor 4 interprets into Head Reports + Floor Summary
4. Captain engine receives Floor 4 outputs
5. permission_gate checks 8 conditions
6. If BLOCKED → silence_reason_logger → output WAIT/BLOCKED → END
7. market_story_engine builds current market story
8. narrative_timeline_engine records significant events
9. Floor Summary read FIRST (summary-first workflow)
10. Drill-down decision: conflict? stale core head? high conviction?
11. If needed: read individual head reports for deeper context
12. Report trust weighting applied per head
13. Stale core head (SMC/ICT) logic applied
14. NO_SETUP intelligence evaluated
15. confluence_engine computes weighted alignment
16. opposite_case_engine checks strongest failure scenario
17. conviction_engine computes 3 scores + conviction band
18. personality_engine determines Captain mood
19. trade_class_engine assigns trade class
20. setup_memory_store updated with current state
21. armed_plan_engine builds/updates conditional plans
22. Expired plans purged, confidence decay applied
23. If active trade exists → thesis integrity review
24. If TRADE approved → trade_constructor builds execution plan
25. decision_snapshot freezes state for audit
26. silence_reason logged if no trade
27. Output dispatched to Side A (execution), Side B (dashboard), Side C (memory)

LIGHT CYCLE FLOW:

1. New tick arrives
2. armed_plan_engine.watch_plans() — check triggers on active armed plans
3. active_trade_supervisor.review_thesis() — quick thesis check
4. intervention_engine — review if conditions warrant rare intervention
5. If armed plan triggered → trigger execution output
6. If intervention justified → output intervention decision
7. NO heavy analysis, NO full cycle recomputation

IMPACT ON REPLAY:

Floor 5 is NOT a replay owner, but decisions must be replay-compatible:
- Same Floor 4 data → identical Captain decisions
- Same data → identical conviction scores and armed plans

- Time-sensitive logic (session_phase, confidence_decay) must use
  injected timestamps in replay mode

replay data. Without Floor 4 replay, Captain decisions cannot be verified.

- CaptainState: mood, conviction_band, decision, market story
- Trade/no-trade reasoning with structured silence reasons
- Active trade supervision notes
- Intervention notes (when triggered)
- Real mode lock / override requirement status
- Decision timeline (from decision snapshots)

IMPACT ON MEMORY (Side C):

Captain contributes:
- DECISION_JOURNAL: every decision snapshot
- HEALTH_EVENT: state changes, errors, mode transitions
- DECISION_JOURNAL entries for: conviction bands, no-trade reasons,
  armed plan creations, intervention events

Captain does NOT store raw data — only decision-level memory.

IMPACT ON EXECUTION (Side A):

Captain provides strategic direction; Side A handles tactical execution:
- Captain: decides TRADE/WAIT/BLOCKED, constructs trade plan
- Side A: runs risk gate, manages order lifecycle, handles broker
- Captain may intervene RARELY (not routine management)
- Side A manages position, Captain monitors thesis

SECTION 10: COMMON MISTAKES + ANTI-PATTERNS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ARCHITECTURE-BREAKING MISTAKES:

MISTAKE 1: Captain recalculating quality or confidence
  - WRONG: captain_engine recomputes smc_quality_score or head confidence
  - RIGHT: Captain consumes quality from Floor 3 via Floor 4, confidence from Floor 4
  - PREVENTION: CI check: no Floor 3/4 calculation imports in floor_5_captain/
  - DETECTION: conviction used where quality/confidence belong → P0 violation

MISTAKE 2: Captain consuming Floor 3 packets as routine input
  - WRONG: captain_engine reads raw Floor 3 domain packets directly
  - RIGHT: Captain consumes ALL Floor 3 data through Floor 4 only
  - PREVENTION: Architecture rule: Floor 4 is the ONLY bridge from Floor 3 to Capt.
  - DETECTION: CI check: no direct floor_3_calculations imports in floor_5_captain/

MISTAKE 3: Light cycle becoming heavy
  - WRONG: light cycle calls confluence_engine, conviction_engine on every tick
  - RIGHT: light cycle only watches armed plans + active trade + intervention
  - PREVENTION: CI check: light_cycle module does NOT import heavy modules
  - DETECTION: Performance test shows tick-time computation > 10ms

MISTAKE 4: Intervention becoming routine
  - WRONG: intervention_engine triggers on ordinary pullbacks
  - RIGHT: intervention is RARE, only for structural thesis breaks
  - PREVENTION: Intervention count tracked in tests (must be < 5% of cycles)
  - DETECTION: Integration test shows intervention count vs total cycles

MISTAKE 5: Captain doing execution work
  - WRONG: captain_engine places orders directly via broker API
  - RIGHT: Captain outputs DecisionOutput, Side A handles execution
  - PREVENTION: CI check: no broker/Side A execution imports in floor_5_captain/

MISTAKE 6: Not respecting non-overridable psychology block
  - WRONG: Captain overrides psychology_head.trade_allowed=False
  - RIGHT: Psychology block is ABSOLUTE — cannot be overridden
  - PREVENTION: permission_gate blocks before any further analysis
  - DETECTION: Test verifies psych block = BLOCKED always

MISTAKE 7: Allowing re-entry without fresh trigger + cooldown
  - WRONG: Captain allows same-direction trade immediately after exit
  - RIGHT: Fresh trigger + cooldown required before re-entry
  - PREVENTION: setup_memory_store tracks recent exits + cooldown

MISTAKE 8: Over-engineering conviction engine
  - WRONG: 50+ conditions, complex formulas, decision paralysis
  - RIGHT: Few things deeply, not everything lightly
  - PREVENTION: Conviction engine limited to known factors

MISTAKE 9: Not handling NO_SETUP intelligence properly
  - WRONG: Captain treats all "no setup" the same
  - RIGHT: Distinguishes healthy silence from uncertain/stale silence
  - PREVENTION: NO_SETUP + state evaluation before silence logging

WHAT CREATES REWORK LATER:

1. Building armed_plan_engine before basic Captain works
   → Fix: get heavy cycle producing TRADE/WAIT/BLOCKED first, then add armed plans
   → Why: armed plans depend on working confluence + conviction

2. Making conviction engine too complex to calibrate later
   → Fix: keep conviction logic simple, parameterized in config
   → Why: tuning weights is normal; rewriting formula is not

3. Not wiring decision snapshots from day one
   → Fix: decision_snapshot_writer built DURING Phase 6, not deferred
   → Why: losing historical decisions = losing calibration data

4. Not testing with real historical market data
   → Fix: replay 1+ days of data through Floor 5, verify decisions make sense
   → Why: synthetic data hides edge cases

5. Forgetting SMC/ICT priority in weighted confluence
   → Fix: head priority list visible and enforced in tests
   → Why: ignoring structure heads undermines Captain's best intelligence

6. Light cycle importing heavy modules accidentally
   → Fix: CI check enforce light_cycle module whitelist
   → Why: one wrong import tanks performance on every tick

7. Not separating silence reasons meaningfully
   → Fix: all 12 silence reasons distinct and tested
   → Why: "no trade" without context is useless for operator
