SECTION 1 — PURPOSE
----------------------------------------------------------------------

Phase 9 validates every contract between every floor and side in the
live system, then locks all wiring so the Captain can fire armed plans
against real broker connections.  This is NOT a build phase — no new
logic, no new modules, no new features.  It is the proving ground
where every data path, every state transition, every error cascade,
and every recovery path is tested end-to-end.

Primary goals (in priority order):

  1. VERIFY every ingest-to-output contract (Floor 1 → Floor 2 → Floor 3
     → Floor 4 → Floor 5 → Side A → broker)
  2. VERIFY every cross-floor data dependency resolves at runtime
     (no stale references, no orphan packets, no unhandled None)
  3. VERIFY every error path produces correct Side C events
     (no silent failures, no unlogged rejections)
  4. VERIFY replay restores exact state across all floors and sides
     (no divergence between live and replay)
  5. VERIFY the dashboard (Side B) renders every data source correctly
     (no missing panels, no stale read models)
  6. VERIFY the kill switch + recovery path protects capital across
     all system states (no bypass)
  7. LOCK the integration by freezing contract versions
     (no more schema changes without explicit deprecation)

SECTION 2 — COMPONENT ARCHITECTURE
----------------------------------------------------------------------

The full system is a 2-line bus architecture:

  Floor Layer (data → intelligence → decision)
    Floor 1  — Market Data Ingestion (raw ticks → cleaned feeds)
    Floor 2  — Data Validation + Structuring (structured data + health)
    Floor 3  — Calculation Engines (SMC / ICT / Technical signals)
    Floor 4  — Department Heads (interpretation + enrichment)
    Floor 5  — Captain (decision + conviction + armed plans)

  Side Layer (execution + storage + visibility)
    Side A   — Execution Engine (risk gate + state machine + broker)
    Side C   — Memory / Event Store (append-only event journal)
    Side B   — Dashboard / Operator Terminal (visualisation + control)

Data flows LEFT-TO-RIGHT through the floor layer, then DOWN to Side A
for execution, with Side C recording every event and Side B consuming
read models for operator visibility.

There is NO right-to-left data flow.  No floor consumes from a higher
floor.  No side writes into the floor layer.

             ┌─────────┐
    Market ──► Floor 1  ──► Floor 2  ──► Floor 3  ──► Floor 4  ──► Floor 5 ──► Side A ──► Broker
             └────┬────┘   └────┬────┘   └────┬────┘   └────┬────┘   └────┬───┘   └────┬───┘
                  │              │              │              │              │             │
                  ▼              ▼              ▼              ▼              ▼             ▼
             ┌──────────────────────────────────────────────────────────────────────────────┐
             │                           Side C — Event Store                               │
             │  (TRADE_JOURNAL | DECISION_JOURNAL | EXECUTION_EVENT | HEALTH_EVENT | …)     │
             └──────────────────────────────────────────────────────────────────────────────┘
                  │              │              │              │              │
                  ▼              ▼              ▼              ▼              ▼
             ┌──────────────────────────────────────────────────────────────────────────────┐
             │                      Side B — Dashboard (read models only)                    │
             └──────────────────────────────────────────────────────────────────────────────┘

----------------------------------------------------------------------

4.1  System-level output contract (what the system produces)

  The complete system produces:
    - Armed plans (from Captain) → executed by Side A
    - Execution records (from Side A) → stored in Side C
    - Decision journals (from Captain) → stored in Side C
    - Health events (from all floors + Side A) → stored in Side C
    - Trade journals (from Side A) → stored in Side C
    - Operator display (from Side B) → rendered in browser
    - Alerts (from Side B) → pushed to operator

  The complete system consumes:
    - Market data (NSE equities, indices, FX, options) via Floor 1
    - Macro data (via Floor 2 macro support)
    - Operator commands (from Side C → Side A via Side B)
    - Operator overrides (from Side B → Side A)
    - Replay data (from Side C → all floors + sides)

4.3  Contract versioning

  After Phase 9, ALL contracts are FROZEN at version 1.0.
  Any schema change requires:
    1. Deprecation notice (version bump to 1.1)
    2. 2-week dual-write period
    3. Full Phase 9 re-verification

4.4  Replay output contract

  Replay must produce EXACTLY the same output as live execution for
  the same input data.  Deterministic modules must be bit-for-bit
  identical.  Non-deterministic modules (e.g., broker latency) must
  produce equivalent state.

SECTION 7 — DATA FLOW
----------------------------------------------------------------------

7.1  Live data flow (normal operation)

  Tick ──► Floor 1 (raw) ──► Floor 1 (cleaned) ──► Floor 2 (validated) ──► Floor 2 (structured)
       ──► Floor 3 (calculated) ──► Floor 4 (interpreted) ──► Floor 5 (decided)
       ──► Side A (executed) ──► Broker

  Every step logs events to Side C.
  Side B reads summaries from Side C read models and direct floor data sources.

7.2  Replay data flow

  Side C REPLAY_REF ──► Floor 1 (replay mode) ──► Floor 2 (replay mode) ──► Floor 3 (replay mode)
       ──► Floor 4 (replay mode) ──► Floor 5 (replay mode) ──► Side A (replay mode)

  Replay mode skips broker connection, uses replay adapter instead.
  Replay produces comparison report showing differences (if any).
  If replay diverges from live: flag as CRITICAL, investigate immediately.

7.3  Error data flow

  Any floor or side error → Side C HEALTH_EVENT (error severity)
  Any risk gate rejection → Side A BLOCKED_ACTION (logged with severity)
  Any broker error → Side A BLOCKED_ACTION → Side C EXECUTION_EVENT (failed state)
  Any system-level error → KillSwitch → Side C HEALTH_EVENT (critical)

7.4  Recovery data flow

  Recover from KillSwitch → clear error state → replay last healthy state
  Recover from LOCKED → operator override required → Side A state reset
  Recover from UNKNOWN → reconciliation → Side A state determined

SECTION 10 — COMMON MISTAKES
----------------------------------------------------------------------

10.1 Architecture-breaking mistakes (will fail Phase 9)

  Mistake 1: Skipping contract validation between stages
    Prevention: Each stage requires ALL contract tests to pass before
                the next stage begins.  NO exceptions.
    Detection:   Stage gate failures

  Mistake 2: Running stages out of order
    Prevention: Stage dependency map is enforced by test harness.
    Detection:   Stage fails because dependency not met

  Mistake 3: Testing REAL broker before PAPER broker passes
    Prevention: REAL broker tests only in Stage 9.5, after all PAPER
                tests pass.  REAL capital is never at risk.
    Detection:   Risk gate blocks REAL mode until PAPER verified

  Mistake 4: Side B consuming raw Side C stores
    Prevention: Contract test ct_dash_007 verifies read models only.
                CI pipeline includes a grep for "raw_store" in Side B code.
    Detection:   Contract test fails / CI pipeline rejects

  Mistake 5: Skipping replay verification
    Prevention: Each stage includes replay verification.  Full system
                replay (Stage 9.8 step 15) is required for Phase 9 pass.
    Detection:   Stage gate fails / final exit criteria not met

  Mistake 6: Modifying contracts during Phase 9
    Prevention: ALL contracts are frozen during Phase 9.  Any schema
                change requires restarting from Stage 9.1.
    Detection:   Contract version mismatch error

  Mistake 7: Running end-to-end test with incomplete wiring
    Prevention: Stage 9.8 requires Stages 9.5, 9.6, and 9.7 to pass first.
    Detection:   Stage gate fails

  Mistake 8: Testing with synthetic data only (not real recorded data)
    Prevention: Stage 9.8 runs against 3 sets of recorded NSE market data.
    Detection:   Non-functional tests fail (real market patterns differ
                 from synthetic)

  Mistake 9: Phase 9 considered optional
    Prevention: Phase 9 is MANDATORY before any REAL capital is deployed.
                No armed plan can execute against a REAL broker without
                Phase 9 sign-off.
    Detection:   Exit criteria not met — system refuses REAL mode

10.2 What creates rework later (and how to prevent it)

  Rework 1: Missing contract tests
    Fix:      Add the missing test, rerun the stage.
    Prevent:  Each contract must have at least one test before the
              stage gate is built.

  Rework 2: Unhandled None in any data path
    Fix:      Add None check, log to Side C HEALTH_EVENT, rerun.
    Prevent:  Type-check every input/output boundary.

  Rework 3: Stale read models in Side B
    Fix:      Update read model, rerun dashboard wiring.
    Prevent:  Side B refresh scheduler includes health check for
              read model freshness.

  Rework 4: Replay state leaking into live state
    Fix:      Isolate replay state, rerun replay tests.
    Prevent:  Replay uses separate in-memory state (not shared with live).

  Rework 5: EmitterRegistry allows non-approved emitter
    Fix:      Update registry, rerun contract test.
    Prevent:  ct_mem_007 explicitly tests rejection of fake emitter.

  Rework 6: Side B command handler uses execute_ instead of request_
    Fix:      Rename handler, rerun contract test.
    Prevent:  ct_dash_009 verifies request_ prefix.

  Rework 7: Integration test order-dependence
    Fix:      Make tests independent, rerun.
    Prevent:  Each integration test creates its own data (no shared state).
