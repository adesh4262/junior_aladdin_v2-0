SECTION 1 - BUILD PHILOSOPHY
================================================================================

JUNIOR ALADDIN is built BOTTOM → UP.

This is locked and non-negotiable.

Why bottom-up:
- FLOOR 1 brings market data into the system
- FLOOR 2 validates, cleans, and structures that data
- FLOOR 3 transforms structured truth into intelligence
- FLOOR 4 interprets that intelligence as expert opinion
- FLOOR 5 judges that opinion into a decision
- SIDE A executes the decision
- SIDE B displays everything
- SIDE C remembers everything

Each layer depends on the layer below it.
Building top-down means building blind — no data to test against.

THE GOLDEN CHAIN:
QUALITY   = FLOOR 3 (calculations, scores, maps)
CONFIDENCE = FLOOR 4 (Heads interpret, create confidence)
CONVICTION = FLOOR 5 (Captain judges, creates conviction)

This chain must NEVER be violated during implementation.

SECTION 5 - DEPENDENCY VALIDATION (WHY THIS ORDER)
================================================================================

PHASE 0 — FOUNDATION FIRST:
- Nothing depends on it, everything depends on it.
- Without project skeleton, types, config, logging — no module can be written.
- Zero dependencies. Starting point of ALL code.

PHASE 1 — FLOOR 1 FIRST AMONG FLOORS:
- Floor 1 is the system's window to the market.
- Without market data ingress, no data exists to process.
- Floor 1 has zero dependencies on other floors — only on external API (Angel One).
- Every floor above depends on Floor 1 data existing.

PHASE 2 — FLOOR 2 SECOND:
- Floor 1 produces raw packets. Floor 2 transforms raw → validated → cleaned → structured.
- Floor 3 cannot compute intelligence on raw/unvalidated data.
- Floor 2 is the truth foundation. Without it, everything above is garbage-in/garbage-out.

PHASE 3 — SIDE C BASE THIRD:
- Side C base needs Floor 2 data contracts (event formats, schemas).
- Side C does NOT need Floor 3+ to be complete.
- Starting Side C early ensures memory exists FROM PHASE 4 onward.
- Every subsequent phase can emit events immediately.
- Side C base is CHEAP to build early (~200 lines: types + stores + query).

PHASE 4 — FLOOR 3 FOURTH:
- Floor 3 requires Floor 2's validated candles, tick streams, options snapshots.
- Without clean data, all intelligence is garbage.
- Floor 3 is the first layer that creates meaning from data.
- Floor 3 is the bottleneck for ALL higher floors (4, 5, Side A).

PHASE 5 — FLOOR 4 FIFTH:
- Heads cannot interpret what Floor 3 hasn't calculated yet.
- Floor 4 is entirely dependent on Floor 3's domain outputs.
- Without Floor 3 states/maps/scores/candidates, Heads have nothing to form opinions on.

PHASE 6 — FLOOR 5 SIXTH:
- Captain requires Head reports and Floor Summary from Floor 4.
- Without expert interpretation, Captain has nothing to judge.
- Captain is the CULMINATION of everything below — built last among floors.

PHASE 7 — SIDE A SEVENTH:
- Side A executes Captain's approved intents.
- Without Captain producing decisions, Side A has nothing to execute.
- But paper broker + state machine can build in parallel after contracts are defined.

PHASE 8 — SIDE B EIGHTH:
- Dashboard consumes data from ALL floors and sides.
- Building it earlier means building against nonexistent/stub backends.
- Dashboard is the LAST thing built because it shows everything that came before.

PHASES 9-11 — FINAL:
- Full integration requires all components to exist.
- Testing requires a complete integrated system.
- Production readiness requires tested, verified code.

SECTION 17 - SIDE C TIMING VALIDATION
================================================================================

SIDE C BASE COMES BEFORE FLOOR 3.

This is a carefully considered architectural decision. Here is the justification:

ARGUMENT 1: INDEPENDENT DEPENDENCY
- Side C base depends on: Floor 2 data contracts (event formats, schemas)
- Floor 3 depends on: Floor 2 validated data outputs
- Both depend on Floor 2 — NOT on each other
- Side C does NOT block Floor 3; Floor 3 does NOT block Side C

ARGUMENT 2: CHEAP TO BUILD EARLY
- Side C base is approximately 200 lines of code
- Types + stores + query layer ≈ 200 lines
- The cost of building early ≈ negligible
- The cost of building late = lost events from Phases 4-7

ARGUMENT 3: EVENT LOSS PREVENTION
- If Side C is deferred to after Floor 5 or Side A:
  - Phase 4 Floor 3 events are lost
  - Phase 5 Floor 4 events are lost
  - Phase 6 Floor 5 decision snapshots are lost
  - Phase 7 Side A execution events are lost
- Retrofitting memory later means backfilling from logs — lossy and unreliable

ARGUMENT 4: FEEDBACK LOOP
- Side C enables review of Floor 3/4/5 outputs during development
- Without Side C, debugging relies entirely on console logs
- With Side C base in place, developers can query what happened

ARGUMENT 5: PARALLEL-SAFE BY DESIGN
- Side C base build does NOT block Floor 3 start
- Side C base build does NOT require Floor 3 designers to wait
- Both can proceed in parallel after Phase 2 data contracts are finalized

ARGUMENT 6: ORGANIC GROWTH
- Side C Stage A (base infra) = Phase 3
- Side C Stage B (full richness) = grows alongside Phases 4-8
- This is organic: base first, richness later, no retrofitting

CONCLUSION:
Side C base in Phase 3 is architecture-wise correct.
It preserves memory from day one. It costs almost nothing. It blocks nothing.
It is the strongest architectural choice.

SECTION 24 - CROSS-PHASE RISKS
================================================================================

RISK 1: CONTRACT DRIFT
- Problem: Floor A changes its output contract. Floor B (consumer) doesn't update.
- Prevention: Contract registry in Floor 2. Contract tests after every integration.
- Detection: Integration test fails. Manual code review.

RISK 2: OWNERSHIP CREEP
- Problem: Floor 1 starts doing validation (Floor 2's job). Floor 3 starts creating confidence (Floor 4's job).
- Prevention: Architecture protection rules. Code review. Automated lint checks.
- Detection: Variables/functions named "confidence" in floor_3_calculations/.

RISK 3: SCOPE CREEP
- Problem: "While we're building X, let's also build Y" — Y is not in the plan.
- Prevention: Build exactly what the roadmap says. Anything else = separate phase.
- Detection: Phase completion checklist vs delivered code.

RISK 4: PREMATURE OPTIMIZATION
- Problem: Optimizing performance before correctness is verified.
- Prevention: Correctness before performance. Mock data → verify → optimize if needed.
- Detection: Time spent on optimization before basic functionality works.

RISK 5: OVER-ENGINEERING
- Problem: Building for "what if" scenarios that may never happen.
- Prevention: Build for what the architecture SPECIFIES. Not for what you IMAGINE.
- Detection: Code complexity far exceeding what the contract requires.

RISK 6: TESTING DEFERRED
- Problem: "We'll add tests later" → later never comes.
- Prevention: Tests written DURING module build, not after. Test file created before implementation.
- Detection: `pytest` shows untested modules.

RISK 7: DOCUMENTATION DEFERRED
- Problem: "We'll document later" → later never comes.
- Prevention: Docstrings written during implementation. README updated per phase.
- Detection: Module has no docstring, no usage examples.

RISK 8: INTEGRATION DELAYED TOO LATE
- Problem: All modules built in isolation, then integration reveals contract mismatches everywhere.
- Prevention: Early and frequent integration testing. Connect to adjacent modules as soon as both exist.
- Detection: First integration attempt reveals multiple contract mismatches.
