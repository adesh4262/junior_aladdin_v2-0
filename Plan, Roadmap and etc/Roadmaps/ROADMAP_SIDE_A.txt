SECTION 1: PURPOSE
================================================================================

Side A is the tactical execution layer of the Junior Aladdin system. Captain
decides — Side A executes. Side A receives Captain's approved execution intent,
runs pre-execution safety checks through the Risk Gate, manages the full order
lifecycle, communicates with broker/simulator, maintains position truth, and
enforces capital protection rules.

Side A philosophy (locked): PROTECT FIRST, THEN OPTIMIZE

Meaning in priority order:
1. Protection first
2. Risk control second
3. Execution quality third
4. Optimization last

Side A is NOT:
- A decision layer (Captain decides, Side A executes)
- An intelligence layer (no market interpretation)
- A strategy layer (no trade invention)
- A discretionary trade judge

Side A IS:
- The tactical executor of Captain's strategic decisions
- The final operational safety net (reduces size for safety, never increases)
- The protection enforcer (Protect First, Then Optimize)
- The state machine owner (12 major execution states)
- The reconciliation authority (local vs broker truth)
- The auditable truth recorder for every action

CRITICAL ARCHITECTURE RULES:

Rule 1: Side A does NOT create trades — it only receives approved intent.
Rule 2: Side A does NOT interpret market meaning — no SMC/ICT/technical judgment.
Rule 3: Side A does NOT create confidence or conviction (belongs to Floors 4-5).
Rule 4: Side A may reject, delay, shrink, retry, or flatten for safety.
         It may NEVER "invent" a better trade.
Rule 5: Side A may REDUCE size for safety. NEVER increase beyond Captain intent.
Rule 6: After execution start, Captain must NOT override the entry side.
         Override is limited to exit/risk-side intervention (enforced by Captain).
Rule 7: ONE active live trade at a time (operationally enforced by Side A).
Rule 8: UNKNOWN/PENDING_RECONCILE states are explicit and bounded.
Rule 9: Broker truth = final live authority. Local snapshot = recovery aid.
Rule 10: Every action must be auditable — blocked actions, overrides, emergencies.

THE CRITICAL SAFETY GATE. Asks only: "Can this approved trade be safely and
validly executed right now?"

12 pre-order checks (BLOCKED if ANY fails):

1. Available capital check: available_capital >= required_capital
2. Required capital check: margin requirement meets limit
3. Margin availability: margin check passes
4. Quantity sanity: lot_size × premium ≤ max_risk_per_trade
5. Lot feasibility: quantity in standard lot multiples
6. Max loss limit: pre-defined per-trade and per-day limits
7. Mode validation: intent mode matches current execution mode
8. Real lock state: if REAL locked by loss_lock_manager → BLOCK
9. Duplicate execution prevention: intent_fingerprint.is_duplicate() → BLOCK
10. One-trade enforcement: no second trade if active position exists
11. Stale intent detection: intent older than max_age_seconds → REJECT
12. Data health check: if STALE → block new; if CRITICAL → escalate

Methods:
- evaluate(intent: ExecutionIntent, context: ExecutionContext) → RiskCheckResult
  Runs all 12 checks. Returns PASSED or FAILED with detailed reasons.
- get_failed_checks() → list of failed check names + reasons
- is_blocked_for_safety() → bool (summary check for dashboard)

SIZE POLICY (locked):
- Captain size = default truth. Side A NEVER increases size.
- Reduce-only adjustment allowed for capital/margin/safety reasons.
- Any reduce-only adjustment must be logged.

Output: RiskCheckResult → if FAILED → blocked_action_journal + WAIT/BLOCKED to Captain
         if PASSED → forward to execution_core

The behavioral backbone of Side A. Defines all valid state transitions.

Major states:
- IDLE: no intent, waiting
- INTENT_RECEIVED: Captain intent received, awaiting risk gate
- RISK_PASSED: risk gate passed, proceeding to order placement
- ORDER_PENDING: order submitted, awaiting acknowledgement
- PARTIAL_FILL: order partially filled
- FILLED: order fully filled
- PROTECTED: SL/TGT protection placed
- MANAGING: active trade management (trail, breakeven adjustments)
- EXITING: exit in progress
- CLOSED: trade fully closed
- FAILED: order rejected or non-recoverable
- LOCKED: critical lock triggered
- UNKNOWN_PENDING_RECONCILE: state unclear, reconciliation needed

State transition table (locked):
| Current State         | Event/Condition                     | Next State                 |
|----------------------|-------------------------------------|----------------------------|
| IDLE                 | Captain intent received             | INTENT_RECEIVED            |
| INTENT_RECEIVED      | Risk gate passed                    | RISK_PASSED                |
| INTENT_RECEIVED      | Risk gate failed                    | FAILED                     |
| RISK_PASSED          | order submitted                     | ORDER_PENDING              |
| ORDER_PENDING        | partial fill                        | PARTIAL_FILL               |
| ORDER_PENDING        | full fill                           | FILLED                     |
| ORDER_PENDING        | reject                              | FAILED                     |
| ORDER_PENDING        | timeout/unclear                     | UNKNOWN_PENDING_RECONCILE  |
| PARTIAL_FILL         | protection staged                   | PROTECTED                  |
| FILLED               | protection placed                   | PROTECTED                  |
| PROTECTED            | active management begins            | MANAGING                   |
| MANAGING             | exit condition triggered            | EXITING                    |
| EXITING              | close complete                      | CLOSED                     |
| ANY ACTIVE STATE     | critical lock / emergency flatten   | LOCKED or CLOSED           |
| UNKNOWN              | reconcile success                   | MANAGING / PROTECTED       |
| UNKNOWN              | unrecoverable / fail-safe           | FAILED / CLOSED            |

Methods:
- transition(event: str) → new state or raises on invalid transition
- get_current_state() → ExecutionMajorState
- is_active() → bool (any state between INTENT_RECEIVED and CLOSED/FAILED)
- is_safe_to_proceed() → bool
- can_receive_new_intent() → bool (only IDLE or CLOSED/FAILED)
- reset() → IDLE

SECTION 7: DATA FLOW
================================================================================

COMPLETE EXECUTION FLOW (PAPER mode shown — REAL mode identical flow):

Note: In REAL mode, replace paper_broker with real_broker at Step 7.
All other steps — validation, routing, risk gate, state machine, lifecycle
management, protection, reconciliation, logging — are IDENTICAL.
This is the PAPER/REAL parity contract: same interface, same flow,
same safety guarantees.

1. Captain produces DecisionOutput (TRADE approved)
2. captain_interface receives DecisionOutput → validates all fields
3. mode_router receives ExecutionIntent
   → ALERT notification fires (dashboard + log)
   → Routes to PAPER path (since mode=PAPER)
4. intent_fingerprint checks for duplicates
   → If duplicate: risk gate blocks, journal records, flow ends
5. risk_gate evaluates 12 checks against intent + current context
   → If any check fails: blocked_action_journal records → return WAIT/BLOCKED
   → If all pass: RiskCheckResult.passed = True
6. execution_state_machine transitions IDLE → INTENT_RECEIVED → RISK_PASSED
7. execution_core receives RISK_PASSED → constructs order from intent
8. execution_core.submit_order() to paper_broker
   → paper_broker applies: slippage, delay, possible rejection
   → Returns simulated ack with order_id
9. execution_state_machine → ORDER_PENDING
10. paper_broker simulates fill (with delay, possible partial)
    → Returns fill data: filled_qty, price, timestamp
11. execution_core.handle_fill() →
    - state_machine → FILLED (or PARTIAL_FILL)
    - order_lifecycle_manager.update_state()
    - position_manager.update_fill(filled_qty, price)
12. protection_model.stage_protection(position)
    → Creates SL + TGT orders via order_lifecycle_manager
    → state_machine → PROTECTED
13. state_machine → MANAGING (active trade management)
14. position_manager handles: trail, breakeven, partial exits
15. On exit trigger (SL hit / TGT hit / time exit / intervention):
    → state_machine → EXITING
    → execution_core closes position
    → state_machine → CLOSED
16. reconciliation_engine checks periodically + on events:
    → Detect → Classify → Compare → Reconcile → Safe Action
17. kill_switch can activate at ANY time:
    → SOFT: blocks new entries, existing trade continues
    → CRITICAL: flatten + freeze immediately
18. execution_logging_layer records EVERY step:
    → Layer 1: all steps above
    → Layer 2: critical events (risk blocks, rejects, emergencies, overrides)
19. blocked_action_journal records every safety block

SECTION 9: CROSS-COMPONENT IMPACTS
================================================================================

IMPACT ON REPLAY:

Side A execution logs are replay-compatible but serve a different purpose
than Floor 2 replay:

- Floor 2 replay: reproduce market data to get identical decisions
- Side A replay: reproduce execution trail for audit/forensic review

Key constraints:
- Execution logs must be sufficient to reconstruct what happened
- Brokers cannot be replayed (paper broker can simulate)
- Side A states and transitions are logged for analysis
- Timing is approximate in replay (not microsecond-accurate)

IMPACT ON DASHBOARD (Side B):

Side A is one of the most important visibility sources for Side B:

- Mode state: ALERT/PAPER/REAL
- Execution state: IDLE → INTENT_RECEIVED → ... → CLOSED
- Order lifecycle: current order states
- Active position: filled_qty, avg_price, SL, TGT, P&L
- Blocked actions: recent blocks with reasons and severity
- Override events: when Captain or operator overrode safety
- Escalation level: NORMAL/CAUTION/SEVERE/EMERGENCY
- Realtime lock / caution state
- Execution logs summaries
- Unknown/reconcile state visibility (CRITICAL for operator)

Dashboard displays execution state; it does NOT control execution directly.

IMPACT ON MEMORY (Side C):

Side A contributes to Side C memory:

- TRADE_JOURNAL: trade_id-linked lifecycle events, fills, P&L, slippage
- EXECUTION_EVENT: order lifecycle, ack/reject, state transitions
- BLOCKED_ACTION: all blocked actions with reasons and severity
- OVERRIDE: override events (Captain override, operator override)
- HEALTH_EVENT: state changes, errors, kill switch events
- REPLAY_REF: reconcile events linking local to broker truth

Each memory event carries:
- trade_id (mandatory for all trade-linked events)
- order chain references (for order lifecycle traceability)
- Override references (when override triggered)
- Reconcile references (when reconcile occurred)

IMPACT ON EXECUTION (Side A self):

Captain provides strategic direction; Side A handles tactical execution:
- Captain decides TRADE/WAIT/BLOCKED, provides trade plan
- Side A runs risk gate, manages order lifecycle, handles broker
- Captain may intervene rarely (exit/risk-side only)
- Side A manages position, Captain monitors thesis
- Side A enforces one-trade discipline operationally

SECTION 10: COMMON MISTAKES + ANTI-PATTERNS
================================================================================

ARCHITECTURE-BREAKING MISTAKES:

MISTAKE 1: Side A creating trades instead of receiving them
  - WRONG: Side A module generates its own trade ideas
  - RIGHT: Side A only receives and executes Captain's approved intent
  - PREVENTION: CI check: no trade decision logic in side_a_execution/
  - DETECTION: Module produces trade generation output → P0 violation

MISTAKE 2: Side A becoming an intelligence layer
  - WRONG: risk_gate evaluates market conditions, not just safety
  - RIGHT: risk_gate asks "can we execute safely?" not "should we execute?"
  - PREVENTION: No Floor 3/4 calculation imports in side_a_execution/
  - DETECTION: risk_gate logic references SMC/ICT/technical → P0 violation

MISTAKE 3: Perfect paper fills
  - WRONG: paper_broker returns perfect fills at exact price, no delay
  - RIGHT: paper_broker applies slippage, delay, possible rejection
  - PREVENTION: Paper broker must have configurable realism factors
  - DETECTION: Integration test shows 0 slippage applied → test failure

MISTAKE 4: Ignoring broker truth during reconciliation
  - WRONG: reconciliation always prefers local state over broker truth
  - RIGHT: Broker truth = final live authority. Local = recovery aid.
  - PREVENTION: reconciliation_engine always defers to broker truth
  - DETECTION: Test shows local state overriding broker state → P0 violation

MISTAKE 5: Allowing two active trades simultaneously
  - WRONG: position_manager allows second position while one exists
  - RIGHT: One-trade enforcement at risk gate + position manager
  - PREVENTION: risk_gate check 10 enforces one-trade rule
  - DETECTION: Integration test shows dual positions → test failure

MISTAKE 6: Captain override creating uncontrolled actions
  - WRONG: Override allows any action (including new entries after execution)
  - RIGHT: Override limited to exit/risk-side after execution start
  - PREVENTION: captain_interface validates override scope
  - DETECTION: Override creates new trade entry → P0 violation

MISTAKE 7: Not logging blocked actions
  - WRONG: Risk gate blocks silently, no record created
  - RIGHT: Every block recorded in blocked_action_journal with severity
  - PREVENTION: blocked_action_journal.record() called on every risk gate block
  - DETECTION: Blocked action count is zero when risk gate blocked → test fail

MISTAKE 8: Using MARKET orders in REAL mode without extreme justification
  - WRONG: real_broker uses MARKET orders by default
  - RIGHT: LIMIT primary, SL-LIMIT protection (conservative by design)
  - PREVENTION: real_broker rejects MARKET orders unless explicitly allowed
  - DETECTION: Test shows MARKET order placed in REAL mode → test failure

MISTAKE 9: Handling UNKNOWN state as success
  - WRONG: UNKNOWN/PENDING_RECONCILE state treated as filled/success
  - RIGHT: UNKNOWN requires bounded reconciliation → escalate if unresolved
  - PREVENTION: state_machine enforces UNKNOWN → reconcile → resolved or escalate
  - DETECTION: Test shows UNKNOWN → CLOSED without reconcile → test failure

WHAT CREATES REWORK LATER:

1. Building real_broker before paper_broker is solid
   → Fix: paper_broker first with realistic simulation
   → Why: paper proves state machine, risk gate, lifecycle before live capital

2. Not testing risk gate with all 12 check combinations
   → Fix: test_risk_gate_comprehensive.py with every check variation
   → Why: missed check combination can let unsafe trade through

3. Skipping blocked action journal
   → Fix: blocked_action_journal built in same sprint as risk_gate
   → Why: missing blocked actions = losing safety audit trail

4. Not integrating data_health_policy early
   → Fix: data_health_policy wired to risk_gate from day one
   → Why: degraded/stale handling is safety-critical, not optional

5. Making logging too verbose or too sparse
   → Fix: Layer 1 for all events, Layer 2 only for critical events
   → Why: too verbose = noise, too sparse = missing audit data

6. Not maintaining paper/real interface parity
   → Fix: paper_broker and real_broker share same interface
   → Why: breaking parity means testing one doesn't help the other

7. Overriding state machine transitions
   → WRONG: execution_core changes state without going through state machine
   → RIGHT: ALL state changes go through execution_state_machine
   → Why: orphan state changes create unrecoverable positions
