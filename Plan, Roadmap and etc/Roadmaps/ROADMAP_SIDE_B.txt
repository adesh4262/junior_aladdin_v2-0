SECTION 1: PURPOSE
================================================================================

Side B is the operator terminal of the Junior Aladdin system. Dashboard reveals
system truth, allows controlled actions, supports review, and provides
explainability. It is the LAST component built because it consumes data from
ALL floors and sides — building it earlier means building against nonexistent
or stub backends.

Side B is NOT:
- A decision layer (it can request, NOT decide)
- A reinterpretation layer (it can reveal, NOT reinterpret)
- An execution layer (it never places orders directly)
- An intelligence layer (no scoring/ranking/analysis)

Side B IS:
- The operator's window into every floor and side
- The explainability layer (trade reason, no-trade reason, captain summary)
- The controlled action interface (mode, capital, kill-switch controls)
- The review and replay workspace (read-only analysis)
- The alert and health surface (system state at a glance)
- The failure visibility layer (per-floor/side state drill-down)

CRITICAL ARCHITECTURE RULES:

Rule 1: Dashboard = Operator Terminal, NOT decision layer.
         Side B can request, NOT decide.
Rule 2: Side B can reveal, NOT reinterpret.
         Display Floor 4 confidence as reported — do NOT recalculate.
Rule 3: Side B routes commands, NOT owns them.
         Commands go to correct floor/side owner, NOT executed by dashboard.
Rule 4: Side C consumption through READ MODELS only.
         Dashboard NEVER queries raw Side C stores directly.
Rule 5: Replay workspace = read-only.
         Replay cannot trigger execution, cannot place orders.
Rule 6: No-trade reason = first-class visibility.
         "No trade" has equal visibility weight to "trade."
Rule 7: Desktop-first, NOT mobile-first.
         This is a trading terminal, not a phone app.
Rule 8: HOT/WARM/COLD refresh model.
         HOT: critical data (positions, prices) — sub-second refresh.
         WARM: important data (head reports, captain state) — 1-5 second refresh.
         COLD: reference data (logs, history) — on-demand refresh.
Rule 9: State-Based Adaptive UI.
         Controls shrink/expand based on system state.
         During active trade: execution controls prominent.
         During no-trade: explainability panels prominent.
Rule 10: Dashboard must work on localhost.
         Single operator, single laptop, zero cloud infrastructure.

data_aggregator.py:
Polls all data sources on configurable intervals, aggregates into single
DashboardState object.

Methods:
- poll_all() → DashboardState (collects from ALL sources)
- poll_hot() → partial state (critical data only: positions, prices, alerts)
- poll_warm() → partial state (important data: head reports, captain)
- get_aggregated_state() → last cached DashboardState
- get_state_snapshot(components) → requested subset of state

Session_cache.py:
In-memory cache with three refresh tiers.

Tier architecture:
- HOT: positions, prices, alerts, execution state (500ms)
- WARM: head reports, captain state, floor summary (3s)
- COLD: reference data, logs, history (30s or on-demand)

Methods:
- get(key, tier) → cached value
- set(key, value, tier) → store with tier-specific expiry
- invalidate(key) → force refresh on next read
- get_cache_stats() → hit/miss rates per tier
- clear_session() → reset cache

Common handler pattern:
- validate(params) → is_valid, error_message

- acknowledge(result) → return ack to dashboard
- log(action) → record in execution log

OUTPUTS (Side B routes commands to owners):

| Output Command    | Destination   | Format              | Key Fields                      |
|------------------|---------------|---------------------|---------------------------------|
| mode_change      | Side A        | ControlCommand      | new_mode, operator_id           |
| capital_update   | Side A        | ControlCommand      | new_capital_limit, reason       |
| kill_switch      | Side A        | ControlCommand      | SOFT/CRITICAL/OFF, reason       |
| override_grant   | Floor 5       | ControlCommand      | override_confirmation, trade_id |
| reconnect_request| Side A        | ControlCommand      | target_broker                   |
| account_reset    | Side A        | ControlCommand      | confirm, new_balance            |

CONTRACT VIOLATIONS THAT MUST REJECT:

1. Command with `execute_` prefix in route name → HALT (should be `request_`)
2. Side C query targeting raw store path → HALT (must go through read model)
3. Command to non-existent floor/side → REJECT
4. Replay workspace command that affects live state → HALT (read-only violation)
5. Operator command without operator context → CAUTION + REJECT
6. Side B displaying recalculated confidence/conviction → P0 violation
7. Dashboard UI placing orders directly → HALT (execution belongs to Side A)

SECTION 7: DATA FLOW
================================================================================

COMPLETE DASHBOARD DATA FLOW:

1. DATA AGGREGATION (backend):
   a. Background pollers fire on their HOT/WARM/COLD intervals
   b. Each data_source adapter calls its floor/side API or reads shared state
   c. Data is normalized into standard response schemas (data_contracts.py)
   d. session_cache stores each result with tier-appropriate TTL
   e. WebSocket pushes updates to connected dashboard clients

2. API RESPONSE (backend → frontend):
   a. Frontend sends HTTP GET to /api/* endpoint
   b. Route handler reads from session_cache (or triggers poll if cache expired)
   c. Returns JSON matching data_contracts.py schema
   d. WebSocket pushes real-time updates for HOT-tier data

3. STATE MANAGEMENT (frontend):
   a. api_client.js receives response → state_manager.js updates local state
   b. WebSocket pushes update state incrementally
   c. refresh_scheduler.js maintains refresh cycle
   d. Components subscribed to state changes re-render

4. COCKPIT RENDERING:
   a. app.js initializes cockpit view
   b. cockpit.js reads from state_manager and renders panels
   c. Panel components update independently (no full page re-render)
   d. State-Based Adaptive UI: panels emphasize based on system state

5. OPERATOR COMMAND:
   a. Operator clicks control button
   b. control_panel.js → api_client.js sends POST to /api/control/*
   c. control_routes.py validates and forwards to command_handler
   d. command_handler routes to correct floor/side owner
   e. Owner processes command and returns acknowledgement
   f. Dashboard updates to reflect new state

6. WORKSPACE NAVIGATION:
   a. Operator clicks workspace tab
   b. app.js switches to selected workspace
   c. Workspace component fetches relevant data
   d. Chart surface re-renders for replay or diagnostics

SECTION 9: CROSS-COMPONENT IMPACTS
================================================================================

IMPACT ON REPLAY:

Side B is the primary consumer of replay data.

- Replay workspace is READ-ONLY — no execution from replay
- Replay data comes from Floor 2 replay engine
- Replay displays identical state to live mode but with "REPLAY" banner
- Chart surface updates during replay at configurable speed
- Decision replay: shows what Captain WOULD have decided
- Replay sessions are selectable by date range and data sources

IMPACT ON DASHBOARD (Side B self):

Dashboard is Side B. Cross-component impacts are the INPUTS to Side B:

- Floor 1 provides: source health, connection status
- Floor 2 provides: data health, validation stats, replay capability
- Floor 3 provides: domain states, chart data, CMSP
- Floor 4 provides: head reports, floor summary, explainability context
- Floor 5 provides: captain state, decisions, armed plans, stories
- Side A provides: execution state, positions, blocked actions
- Side C provides: historical data through read models

IMPACT ON MEMORY (Side C):

Side B does NOT write to Side C directly.
Side B consumes Side C data through read models only.

- Read models used: trade_history, decision_history, health_events,
  blocked_action_history, override_history
- NEVER raw store queries from Side B
- Dashboard is a consumer of memory, not a producer

IMPACT ON EXECUTION (Side A):

Side B controls execution parameters but does NOT execute:

- Mode changes route to Side A (mode_router)
- Kill switch commands route to Side A (kill_switch)
- Override confirmations route to Floor 5 (override_guard)
- Capital updates route to Side A (risk_gate)
- Reconnect requests route to Side A (execution_core/reconciliation)

Side B displays execution state but never overrides Side A's execution logic.

SECTION 10: COMMON MISTAKES + ANTI-PATTERNS
================================================================================

ARCHITECTURE-BREAKING MISTAKES:

MISTAKE 1: Dashboard making decisions instead of revealing them
  - WRONG: Side B shows "Buy now" button that creates trades
  - RIGHT: Side B shows "Request trade" that sends to Captain for approval
  - PREVENTION: CI check: no `execute_` route prefix, only `request_`
  - DETECTION: Route creates trade without Captain approval → P0 violation

MISTAKE 2: Side B recalculating Floor 4 confidence or Floor 5 conviction
  - WRONG: Dashboard re-aggregates head biases into custom confidence score
  - RIGHT: Side B displays Floor 4 confidence exactly as reported
  - PREVENTION: CI check: no Floor 3/4 calculation imports in side_b_api/
  - DETECTION: Dashboard shows confidence that differs from Floor 4 output

MISTAKE 3: Dashboard querying raw Side C stores directly
  - WRONG: side_b_api queries raw SQLite tables or raw store APIs
  - RIGHT: side_b_api queries Side C query layer (read models only)
  - PREVENTION: side_c_source.py only exposes read model methods
  - DETECTION: API route accesses raw store path → P0 violation

MISTAKE 4: Dashboard placing orders directly through broker API
  - WRONG: side_b_dashboard has broker API keys and places LIMIT orders
  - RIGHT: Dashboard changes mode to PAPER/REAL, Side A handles execution
  - PREVENTION: No broker API keys in Side B config. No broker SDK in Side B.
  - DETECTION: Side B code imports broker SDK → P0 violation

MISTAKE 5: Building UI before backend data exists
  - WRONG: Building Captain panel before Floor 5 produces CaptainState
  - RIGHT: Build data sources first, UI components after data contract verified
  - PREVENTION: Entry criteria requires data-producing floors/sides to be complete
  - DETECTION: UI component displays empty/error for non-existent data

MISTAKE 6: Replay workspace allowing execution
  - WRONG: Replay has "Place order" button that sends to live broker
  - RIGHT: Replay workspace is READ-ONLY, all action buttons disabled
  - PREVENTION: Replay routes refuse any command that modifies live state
  - DETECTION: Replay session modifies position → P0 violation

MISTAKE 7: Mobile-first instead of desktop-first
  - WRONG: Dashboard designed for 400px phone screens
  - RIGHT: Dashboard designed for 1920px+ desktop monitors
  - PREVENTION: UI components use desktop-appropriate layout
  - DETECTION: Key trading info hidden behind hamburger menus

WHAT CREATES REWORK LATER:

1. Building UI before backend data contracts are verified with each floor/side
   → Fix: data_aggregator and data_sources built and tested FIRST
   → Why: UI that displays wrong-format data requires full rewrite

2. Not using HOT/WARM/COLD refresh model from day one
   → Fix: session_cache.py + refresh_scheduler.js built in Phase A
   → Why: retrofitting refresh tiers means rewriting data flow

3. Over-engineering frontend with heavy framework
   → Fix: plain HTML/CSS/JS or lightweight framework (no build toolchain)
   → Why: this is a single-operator terminal, not a consumer web app

4. Forgetting State-Based Adaptive UI
   → Fix: cockpit.js respects system state for panel prominence
   → Why: showing all controls all the time creates operator confusion

5. Not making no-trade reason first-class
   → Fix: explainability_panel.js has equal weight for trade and no-trade
   → Why: operator needs to understand WHY not trading, not just why trading

6. Hardcoding chart timeframe or overlays
   → Fix: chart_surface.js configurable by operator preferences
   → Why: each operator has different chart preferences

7. Missing failure visibility for degraded/stale/unknown states
   → Fix: floor_drilldown.js + health_panel.js show ALL states prominently
   → Why: hidden failures lead to operator surprises
