SECTION 1 - PURPOSE
================================================================================

One-line identity:
SIDE C = Memory Layer — the system's durable structured memory.

Why Side C exists:
- Side A acts, Side B shows, Side C remembers
- System needs durable memory beyond console logs and session state
- Trade journals, decision journals, execution events, health events,
  override history, replay references, review references, blocked actions —
  all must be preserved, linked, and queryable
- Side C enables: operator review, forensic debugging, replay support,
  audit support, historical trade reasoning review, system health reconstruction

What Side C does:
- Persists system memory (operational events, trades, decisions)
- Persists decision references (conviction bands, no-trade reasons)
- Persists execution truth references (order lifecycle, fills)
- Persists health/event history (connection state, data quality)
- Persists replay/review links (cross-system traceability)
- Exposes queryable historical views through read models

What Side C explicitly does NOT do (CORE PRINCIPLES — LOCKED):

PRINCIPLE 1: Side C stores facts, does NOT create truth
PRINCIPLE 2: Side C remembers, does NOT decide
PRINCIPLE 3: Side C links, does NOT interpret
PRINCIPLE 4: Side C may summarize, may NOT score/rank/recommend/analyze
PRINCIPLE 5: Correction = new event (no silent mutation)
PRINCIPLE 6: Append-first only — no mutation, no deletion
PRINCIPLE 7: Approved emitters only — no arbitrary writes
PRINCIPLE 8: Side C owns storage, NOT truth ownership
PRINCIPLE 9: Memory access principle — consumers request views, do not own stores
PRINCIPLE 10: Side C does NOT drift into intelligence, interpretation, or decision territory

ARCHITECTURE RULES (MUST NEVER BE VIOLATED):
- Side C stores have NO `score`/`rank`/`recommend` methods (P0 violation)
- Side C stores have NO trade decision logic (belongs to Captain)
- Side C stores have NO market interpretation (belongs to Floor 3+)
- No raw Side C store consumption from Side B (must go through query layer / read models)
- Dashboard consumes Side C through READ MODELS / PROJECTIONS, not raw stores
- Replay authority: Floor 2 = replay-capable truth owner, Side C = replay reference memory owner only

MODULE RESPONSIBILITIES:

c_types.py:
- MemoryEvent dataclass: event_type, source, emitter, timestamp, severity,
  family, payload, refs (dict of reference links)
- MemoryEnvelope dataclass: family, event_type, source, emitter, timestamp,
  severity, refs, envelope_id
- EventFamily enum: TRADE_JOURNAL, DECISION_JOURNAL, EXECUTION_EVENT,
  HEALTH_EVENT, OVERRIDE, REPLAY_REF, REVIEW_REF, BLOCKED_ACTION
- Severity enum: INFO, CAUTION, SEVERE, CRITICAL
- MemoryQuery dataclass: families (list), start_time, end_time,
  refs_filter (dict | None), limit, offset
- RetentionPolicy dataclass: family, max_age_days, archive_after_days

ingest_layer.py:
- Accept inputs ONLY from approved emitters (Floor 1, Floor 2, Floor 5, Side A)
- Validate emitter identity against emitter_registry.py
- Validate event schema against write_contracts.py
- Apply MemoryEnvelope normalization
- Forward to Event Router
- Reject unauthorized emitters with logged error (no crash)
- Reject malformed events with logged error + Severity.CAUTION

event_router.py:
- Classify incoming event into correct family (TRADE_JOURNAL, etc.)
- Validate event against write contract for that family
- Route to: event_store, journal_store, or reference_store based on family
- Enforce append-first behavior (never modify, never delete)
- Log every route decision

event_store.py:
- Write operational/system events: execution events, health events,
  blocked actions, override events, state transitions
- Append-first, indexed by (family, timestamp, source)
- Query by: family + timerange, source, severity
- No mutation methods — only append + query

journal_store.py:
- Write durable journal records: trade journal, decision journal
- Higher-level narrative memory (not operational packets)
- Append-first, indexed by (family, timestamp, trade_id, decision_id)
- Query by: family, trade_id, decision_id, timerange
- Journal records may be linked to multiple events via refs

reference_store.py:
- Write linkable references: replay_ref, review_ref, source_ref,
  packet lineage refs, cross-system IDs
- Key-value semantics: ref_key → ref_value + metadata
- Indexed by ref_key, ref_type, timestamp
- References are linking instruments, NOT narrative records

query_layer.py:
- Simple read projections for Side B dashboard consumption
- Family-aware retrieval by timestamp range
- Reference lookup by ID
- Trade history summaries, decision review summaries
- Override history summaries, recent blocked action summaries
- Health history summaries
- NO complex analytics — no scoring, no ranking, no recommendation
- NOT for direct raw store access from consumers

retention_manager.py:
- Family-based retention policy with configurable TTLs
- Trade Journal → long-term (default: 365 days)
- Decision Journal → long-term (default: 365 days)
- Override History → long-term (default: 365 days)
- Execution Events → medium (default: 90 days)
- Health Events → medium (default: 90 days)
- Blocked Actions → medium (default: 90 days)
- Replay References → aligned with replay usefulness (default: 30 days)
- Review References → aligned with review usefulness (default: 30 days)
- Archive support: move to colder storage (configurable, not deletion)
- Archive is NOT deletion — preserves reference usability

read_model_builder.py:
- Build lightweight read models/projections for Side B consumption
- Trade history projection
- Decision review projection
- Health timeline projection
- Override history projection
- These are READ MODELS — Side B consumes these, NOT raw stores

contracts/write_contracts.py:
- Write contracts per event family
- Define mandatory fields per family
- Define field types and validation rules
- Used by event_router for runtime validation

contracts/read_contracts.py:
- Read contracts per query type
- Define query parameters and response format
- Used by query_layer for request validation

contracts/emitter_registry.py:
- Define approved emitters with:
  - emitter_id: str (e.g., "floor_1", "floor_5", "side_a")
  - allowed_families: list[EventFamily]
  - source_owner: str (e.g., "Floor 1", "Floor 5", "Side A")
- Used by ingest_layer for emitter validation
- Each emitter can write to specific families only
- Example:
  - Floor 1: HEALTH_EVENT only
  - Floor 2: HEALTH_EVENT, REPLAY_REF, REVIEW_REF
  - Floor 5: DECISION_JOURNAL
  - Side A: TRADE_JOURNAL, EXECUTION_EVENT, BLOCKED_ACTION, OVERRIDE

STEP 3.3 — INGEST LAYER (ingest_layer.py)

STEP 3.4 — EVENT ROUTER (event_router.py)
----------------------------------------------------------------------
File: side_c_memory/event_router.py

Purpose: Classify, validate, and route events to correct store.

Input contract:

Output contract:

  - Routes to correct store
  - Raises ContractViolationError if family not recognized

Routing table:
- TRADE_JOURNAL → journal_store

- EXECUTION_EVENT → event_store
- HEALTH_EVENT → event_store
- OVERRIDE → event_store
- BLOCKED_ACTION → event_store
- REPLAY_REF → reference_store
- REVIEW_REF → reference_store

Dependencies: Step 3.1 (c_types.py for EventFamily), Step 3.3 (ingest_layer for envelopes)
Validation:
- HEALTH_EVENT → route_event returns "event_store"
- TRADE_JOURNAL → route_event returns "journal_store"
- REPLAY_REF → route_event returns "reference_store"
- Unknown family → ContractViolationError
- Append-first enforced: route does NOT modify event data

  - Ordered by timestamp ascending
- get_event(event_id: str) → MemoryEnvelope | None
- count_events(families, start_time, end_time) → int
- NO delete, NO update, NO mutation methods

Dependencies: Step 3.1 (c_types.py), Step 3.4 (event_router for routing)
Validation:
- Append HEALTH_EVENT → stored, returned on query
- Append EXECUTION_EVENT → stored, returned on query
- Append TRADE_JOURNAL → ValueError (wrong store — belongs to journal_store)
- Query by timerange → correct events returned
- Query by family → only matching family returned
- Append 100 events → query with limit=10 → 10 returned
- Append then query same event_id → same event returned unchanged (append-only)

STEP 3.10 — RETENTION MANAGER (retention_manager.py)
----------------------------------------------------------------------
File: side_c_memory/retention_manager.py

Purpose: Enforce family-based retention policies.

Input contract:
- RetentionPolicy definitions per family
- Access to all stores for data cleanup

Output contract:
- apply_retention_policy() → dict (families_affected, events_archived,
  events_expired, errors)
  - Checks each family's max_age_days against current time
  - Events older than max_age_days → moved to archive (if archive_after_days set)
  - Events older than archive + max_age → expired (deleted if no archive)
  - Returns summary of actions taken
- get_retention_status() → dict (per-family: total_count, oldest_event_age_days,
  next_expiry_days)
- set_retention_policy(family: EventFamily, policy: RetentionPolicy) → None
  - Override default retention for a family
  - Raises ValueError if policy.max_age_days < 1

Default retention policies (LOCKED):
- TRADE_JOURNAL: max_age_days=365, archive_after_days=90
- DECISION_JOURNAL: max_age_days=365, archive_after_days=90
- OVERRIDE: max_age_days=365, archive_after_days=90
- EXECUTION_EVENT: max_age_days=90, archive_after_days=30
- HEALTH_EVENT: max_age_days=90, archive_after_days=30
- BLOCKED_ACTION: max_age_days=90, archive_after_days=30
- REPLAY_REF: max_age_days=30, archive_after_days=None (no archive, direct expiry)
- REVIEW_REF: max_age_days=30, archive_after_days=None

Dependencies: Steps 3.5-3.7 (all stores for cleanup operations)
Validation:
- Set retention to 1 day → apply → events older than 1 day archived
- Set retention to -1 → ValueError raised
- get_retention_status → returns correct count and age per family
- Archive preserves events (not deletion)
- Expiry actually removes events from active store

SECTION 4 - INPUT/OUTPUT CONTRACTS
================================================================================

SIDE C AS A WHOLE:

INPUT CONTRACT (from approved emitters):
- Emitter must be registered in emitter_registry
- Event must match write contract for its family
- Event must include: event_type, source, emitter, timestamp, severity, family, payload, refs
- Emitter identity is authenticated at ingest layer

OUTPUT CONTRACT (to consumers through query layer / read models):
- Trade history summaries: trade_id, entry, exit, pnl, mode, timestamps
- Decision review summaries: decision_id, conviction band, reason, timestamp
- Health timeline: events grouped by severity, timerange
- Override history: override events with reasons
- Blocked actions: blocked actions with reasons and severities
- All outputs are READ-ONLY views — consumers cannot modify Side C state

ERROR CONTRACT:
- Unauthorized emitter: rejected at ingest layer, logged (Severity.CAUTION)
- Unauthorized family: rejected at ingest layer, logged (Severity.CAUTION)
- Malformed event: rejected at ingest layer, logged (Severity.CAUTION)
- Write contract violation: rejected at event_router, logged
- Query with invalid params: query_layer returns empty list or raises ValueError
- Retention policy error: logged, does not crash

PER-MODULE CONTRACTS:

| Module                | Input                                    | Output                                   | Error                       |
|-----------------------|------------------------------------------|------------------------------------------|-----------------------------|
| c_types.py            | None (type definitions)                   | Enums, dataclasses                       | N/A                         |
| emitter_registry.py   | Emitter definitions                       | Registry, approval checks                | KeyError on unknown emitter |
| write_contracts.py    | Family definitions                        | Field validation                         | ValueError on invalid def   |
| ingest_layer.py       | Event data + emitter identity             | Normalized MemoryEnvelope                | ValueError on rejection     |
| event_router.py       | MemoryEnvelope                            | Store routing decision                   | ContractViolationError      |
| event_store.py        | MemoryEnvelope (operational families)     | Stored event + event_id                  | ValueError on wrong family  |

SECTION 10 - COMMON MISTAKES + ANTI-PATTERNS
================================================================================

WHAT CAN BREAK ARCHITECTURE IN THIS SIDE:

Mistake 1: Adding scoring/ranking/recommendation methods to stores
- Problem: Adding `get_best_trades()`, `recommend_entry()`, `score_setups()` methods
- Architecture violation: Side C = Memory Layer. Scoring = Floor 3+. Ranking = Floor 4+.
- Prevention: If you're writing a method name with "score", "rank", "recommend",
  "best", "top", "analyze" in Side C → stop. That's not memory, that's intelligence.
- Detection: Code review catches these method names in side_c_memory/.
  CI lint check: `grep -r "def score\|def rank\|def recommend\|def analyze" side_c_memory/`
  should return nothing.

Mistake 2: Allowing non-approved emitters to write
- Problem: Accepting events from any source without emitter validation
- Architecture violation: Memory corruption risk. Side C becomes untrustworthy.
- Prevention: ingest_layer MUST validate every emitter against emitter_registry.
  Unknown emitter → reject with logged error.
- Detection: Ingest logs show writes from non-approved source → P0 investigation.

Mistake 3: Mutating stored events instead of appending corrections
- Problem: Updating a trade journal entry instead of appending a correction
- Architecture violation: Violates Principle 5 — "correction = new event, no silent mutation"
- Prevention: Stores have NO update/delete methods. Only append + query.
- Detection: Any `update` or `delete` method in store classes → P0 bug.

Mistake 4: Letting Side B consume raw stores directly
- Problem: Side B dashboard queries event_store directly instead of going through
  read_model_builder → query_layer
- Architecture violation: Side B becomes coupled to raw storage. Contract boundary broken.
- Prevention: ALL Side C consumption from Side B goes through read_model_builder.
  Raw store access restricted to query_layer only.
- Detection: Side B code imports side_c_memory.event_store directly → architecture violation.

Mistake 5: Making Side C a dependency for Floor 3/4/5 operation
- Problem: Floors 3/4/5 fail/crash if Side C is unavailable
- Architecture violation: Side C is PARALLEL-SAFE by design. It does NOT block floors.
- Prevention: Floors must handle Side C being unavailable gracefully
  (log warning, continue without memory).
- Detection: Floor code raises error on Side C write failure instead of logging and continuing.

Mistake 6: Adding trade decision logic to query layer
- Problem: Query layer filtering trades by "quality" or suggesting "best entries"
- Architecture violation: Query layer is read access. Decision logic = Floor 5.
- Prevention: Query layer returns factual data only. No interpretation, no filtering by opinion.
- Detection: Query layer methods accept parameters like "min_quality" or "only_best".

Mistake 7: Not defining approved emitters list early
- Problem: Starting store implementation before emitter_registry is defined
- Architecture violation: Governance gaps. Unknown emitters can write data.
- Prevention: Step 3.2 (emitter_registry) built BEFORE any store code (Step 3.5+).
- Detection: stores/ exist but contracts/emitter_registry.py does not.

WHAT CREATES REWORK LATER:
- Not defining emitter_registry early → governance gaps, unknown data sources
  Fix: Build Step 3.2 before any store code
- Forgetting MemoryEnvelope on ingress → non-standardized records, hard to query
  Fix: Ingest layer always normalizes envelope before routing
- Making stores too complex before floors emit real events → over-engineering
  Fix: Build stores simple (append + query), add features when floors need them
- Not indexing by (family, timestamp) → slow queries when Side B needs them
  Fix: Index all stores by family + timestamp from day one
- Making retention too aggressive → losing useful debugging data
  Fix: Use default retention policies (365d for journals, 90d for events, 30d for refs)
- Not separating event/journal/reference stores → everything becomes a messy single store
  Fix: Three distinct stores from day one with clear routing rules
- Letting query layer become too complex → Side C drifts toward intelligence
  Fix: Query layer does simple retrieval only. Complex analysis belongs in higher floors.
