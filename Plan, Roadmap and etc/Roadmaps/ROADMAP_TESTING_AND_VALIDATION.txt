----------------------------------------------------------------------
SECTION 1 — PURPOSE
----------------------------------------------------------------------

Phase 10 is the system-wide validation gate between full integration
(Phase 9) and production readiness (Phase 11).  Every module, every
contract, every path, every edge case is tested under realistic
conditions before real capital is at risk.

This is a TEST INFRASTRUCTURE build phase.  No new system logic, no
new features, no new business rules.  The test harness (scenario
runner, profilers, comparator, simulation runner) is infrastructure
code — it orchestrates validation without adding system behaviour.

Phase 10 is the gauntlet the system must run to prove it is safe,
correct, and performant enough for production.

Primary goals (in priority order):

  1. ACHIEVE ≥80% code coverage across ALL modules (unit tests)
  2. VERIFY every floor/side pair integration path (integration tests)
  3. VERIFY deterministic replay: 1+ day of market data produces
     identical outputs on replay
  4. VERIFY edge case resilience: empty data, corrupt data, disconnects,
     timeouts, max load, duplicate feeds
  5. VERIFY performance: tick processing latency within acceptable
     bounds, heavy cycle under 30 seconds
  6. VERIFY stress tolerance: high-volume tick bursts, sustained load
  7. VERIFY paper mode: 100+ simulated trades with ZERO critical failures
  8. GENERATE a test completion report with pass/fail metrics per gate

----------------------------------------------------------------------
SECTION 2 — COMPONENT ARCHITECTURE
----------------------------------------------------------------------

Phase 10 adds NO new system components.  It introduces a testing
infrastructure layer that wraps the existing system:

  test_harness/
  ├── conftest.py                    # Shared pytest fixtures, mock generators
  ├── mock_data/
  │   ├── recorded_ticks/            # 3 sets of 3-hour NSE tick recordings
  │   ├── recorded_candles/          # Derived OHLCV from recorded ticks
  │   └── corrupt_samples/           # Known-bad data for edge cases
  ├── scenarios/
  │   ├── normal_market.py           # Normal market conditions
  │   ├── high_volatility.py         # Rapid price movement
  │   ├── low_liquidity.py           # Thin order book
  │   ├── disconnect_reconnect.py    # Network interruption
  │   ├── duplicate_feeds.py         # Duplicate tick injection
  │   ├── corrupt_feeds.py           # Corrupt/malformed packets
  │   ├── empty_session.py           # No data session
  │   ├── burst_load.py              # High-volume tick bursts
  │   └── sustained_load.py          # Extended high-volume ticks
  ├── paper_simulation/
  │   ├── simulation_runner.py       # Orchestrates 100+ paper trades
  │   ├── trade_scenarios.py         # Win/loss/partial/flat scenarios
  │   └── results_analyzer.py        # Aggregates simulation results
  ├── performance/
  │   ├── latency_profiler.py        # Per-module latency measurement
  │   ├── memory_profiler.py         # Memory usage tracking
  │   └── cycle_timer.py             # Heavy/light cycle timing
  ├── coverage/
  │   ├── coverage_config.ini        # pytest-cov configuration
  │   └── coverage_reporter.py       # Generates per-module coverage report
  └── reports/
      ├── gate_results.json          # Pass/fail per gate
      ├── coverage_report.json       # Coverage by module
      ├── performance_report.json    # Latency + memory metrics
      ├── simulation_report.json     # Paper trade results
      └── completion_certificate.json# Final Phase 10 sign-off

CRITICAL RULE:
The test harness is SEPARATE from the system code.  It does NOT
import from floor_*/side_* modules directly in test infrastructure.
It launches the system as a black box and verifies outputs through
Side B data sources and Side C read models (for integration tests).
Unit tests import modules directly with mocked dependencies.

SECTION 3 — STEP-BY-STEP BUILD ORDER
----------------------------------------------------------------------

Phase 10 has 7 GATES.  Each gate is a validation stage — no gate
may begin until the previous gate passes all checks.  Each gate
produces a machine-readable results file.

──────────────────────────────────────────────────────────────────────
GATE 10.1 — UNIT TEST COVERAGE GATE
──────────────────────────────────────────────────────────────────────

PURPOSE:  Measure and enforce ≥80% code coverage across all modules.

BUILD STEP 10.1.1 — coverage_config.ini
  Purpose:  Configure pytest-cov to measure coverage across all
            floor_*, side_*, and shared/ modules.
  Content:  paths, exclude patterns, fail-under threshold (80%),
            omit test files and __init__.py
  Validation: pytest --cov runs without error

BUILD STEP 10.1.2 — coverage_reporter.py
  Purpose:  Parse pytest-cov JSON output and generate per-module
            coverage report with pass/fail indicators.
  Input:    pytest-cov JSON output
  Output:   coverage_report.json (per-module coverage % + overall)
  Validation: known coverage data → correct report

VERIFICATION CHECKLIST (GATE 10.1):
  [ ] coverage_config.ini exists with correct paths
  [ ] coverage_reporter.py parses pytest-cov output correctly
  [ ] `pytest --cov --cov-report=json -x` runs without error
  [ ] ALL modules have ≥80% coverage
  [ ] coverage_report.json generated and saved
  [ ] Any module below 80% flagged with specific file list

FAILURE ACTION:
  If coverage < 80%: identify under-covered modules, add missing
  unit tests, rerun gate.  Phase 10 does NOT proceed until ≥80%.

──────────────────────────────────────────────────────────────────────
GATE 10.2 — UNIT TEST INTEGRITY GATE
──────────────────────────────────────────────────────────────────────

PURPOSE:  Ensure every unit test is meaningful (tests assertions,
          not just runs-without-error).

This gate does NOT add new code.  It runs an audit pass:

VERIFICATION CHECKLIST (GATE 10.2):
  [ ] Every test file has ≥1 assertion (no assert-free tests)
  [ ] Every test function has ≥1 assertion or pytest.raises
  [ ] No tests skipped by default (unless explicitly documented why)
  [ ] No tests using @pytest.mark.skip without comment
  [ ] No tests catching all exceptions (bare `except:`)
  [ ] Every mock fixture validates at least the expected call
  [ ] Edge case tests exist for: None input, empty input, corrupt input
  [ ] Error case tests exist for: exception propagation, fallback paths
  [ ] No flaky tests identified (run each test 3×, all must pass)
  [ ] All tests complete in < 30 seconds total

FAILURE ACTION:
  Fix or document each integrity violation.  Rerun gate.

──────────────────────────────────────────────────────────────────────
GATE 10.4 — REPLAY VALIDATION GATE
──────────────────────────────────────────────────────────────────────

PURPOSE:  Verify deterministic replay across all floors and sides
          using ≥1 day of recorded market data.

BUILD STEP 10.4.1 — replay_comparator.py
  Purpose:  Compare live vs replay output for a given data range.
            Produces diff report showing any divergence points.
  Input:    Live output records (from Side C), replay output records
  Output:   Replay comparison report with:
            - Total records compared
            - Records matching
            - Records diverging (with field-level diff)
            - Non-deterministic modules flagged (broker, network)
            - Overall verdict: PASS or FAIL (FAIL if any deterministic
              module diverges)
  Dependencies: Side C must have replay query support
  Validation: Known identical data → PASS. Known different data → FAIL.

VERIFICATION CHECKLIST (GATE 10.4):
  [ ] replay_comparator.py works correctly

  Then run the 10 replay tests defined in Phase 9 (rp_001-010):
  [ ] rp_001: Floor 1 replay (raw data → cleaned data)
  [ ] rp_002: Floor 2 replay (cleaned data → structured data)
  [ ] rp_003: Floor 3 replay (structured data → calculated signals)
  [ ] rp_004: Floor 4 replay (signals → head reports)
  [ ] rp_005: Floor 5 replay (head reports → armed plans)
  [ ] rp_006: Side A replay (execution events → position state)
  [ ] rp_007: Side C replay (all events → read models)
  [ ] rp_008: Full system replay (all floors + sides)
  [ ] rp_009: Replay with partial data (test edge cases)
  [ ] rp_010: Replay with error injection (test error recovery)

  [ ] ALL 10 replay tests pass with ≥1 day of market data
  [ ] Replay comparison reports saved for each test
  [ ] replay_report.json generated

CRITICAL:
  If ANY deterministic module diverges between live and replay,
  the replay gate FAILS.  Non-deterministic divergence (broker
  latency, network jitter) is flagged but does NOT fail the gate.

FAILURE ACTION:
  Investigate divergence root cause.  Fix the module.  Re-record
  market data run.  Rerun gate.

GATE 10.7 — PAPER MODE SIMULATION GATE
──────────────────────────────────────────────────────────────────────

PURPOSE:  Verify the system can execute 100+ simulated paper trades
          without any critical failure.  This is the FINAL gate before
          production readiness.

BUILD STEP 10.7.1 — simulation_runner.py
  Purpose:  Orchestrate 100+ paper trades across multiple sessions.
            Feeds recorded market data, lets Captain decide, routes
            through Side A paper broker, tracks results.
  Input:    Recorded market data (multiple sessions)
  Output:   Simulation results (trade count, win/loss, critical
            failures, exceptions, risk gate rejections)
  Dependencies: All floors + sides must be fully operational
  Validation: Runs through 10 trades → results expected

BUILD STEP 10.7.2 — trade_scenarios.py
  Purpose:  Define specific trade scenarios to exercise the system:
            - Winning trades (price moves in favor)
            - Losing trades (price moves against, SL hit)
            - Partial fills (only part of order fills)
            - No-fill scenarios (order not filled)
            - Protection triggers (SL/TP hit correctly)
            - Psychology block active (no trades expected)
            - Loss lock triggers (3 losses → REAL lock)
            - Operator override scenarios
  Content:  For each scenario: data range, expected outcome,
            verification criteria

BUILD STEP 10.7.3 — results_analyzer.py
  Purpose:  Aggregate simulation results and produce final report.
  Input:    Simulation runner output
  Output:   simulation_report.json with:
            - Total trades attempted
            - Total trades executed
            - Win/loss count + ratio
            - Average trade duration
            - Average P&L per trade
            - Risk gate rejection count (by check type)
            - Kill switch activations
            - Critical failures (ZERO required)
            - Exceptions (ZERO required)
            - Loss lock activations
            - Override count
            - Overall verdict: PASS or FAIL
  Dependencies: simulation_runner.py output
  Validation: Known simulation data → correct analysis

VERIFICATION CHECKLIST (GATE 10.7):
  [ ] simulation_runner.py works
  [ ] trade_scenarios.py contains all required scenarios
  [ ] results_analyzer.py works

  [ ] Losing trades hit SL correctly

  [ ] No-fill scenarios expire correctly

  [ ] Loss lock activates after 3 consecutive losses

  [ ] simulation_report.json generated

SECTION 7 — DATA FLOW
----------------------------------------------------------------------

7.1  Unit test data flow

  [test file] ──► import module ──► create mock input ──►
      call function ──► assert output matches expected

  Data is synthetic.  No recorded market data needed.
  Each unit test creates its own input data inline.

7.2  Integration test data flow

  [conftest.py fixture] ──► start all floors + sides ──►
      inject recorded tick ──► wait for processing ──►
      read Side C events + Side B data sources ──►
      assert correct events/output ──► stop system ──►
      clean up temporary data

  Data is recorded market data (from Phase 9 preparation).
  Each integration test injects fresh data into a fresh system instance.

7.3  Replay data flow

  [replay test] ──► run system with recorded data (live run) ──►
      capture all outputs ──► run replay with same data ──►
      capture all outputs ──► compare live vs replay ──►
      report divergence (if any)

  Data is recorded market data (≥1 day).
  Replay reads from Side C REPLAY_REF; does NOT re-ingest from Floor 1.

7.4  Edge case data flow

  [scenario] ──► start system ──► inject abnormal data ──►
      monitor system health (no crash) ──► check Side C events ──►
      check recovery (if applicable) ──► stop system

  Data is corrupt/abnormal samples or specific injection patterns.

7.5  Performance data flow

  [profiler] ──► start system ──► inject high-volume data ──►
      measure per-module latency ──► measure memory ──►
      record results ──► stop system

  Data is recorded burst/sustained load data.

7.6  Paper simulation data flow

  [simulation] ──► start system ──► feed multi-session data ──►
      let Captain decide ──► Side A paper broker executes ──►
      track every trade ──► analyze results ──►
      report pass/fail ──► clean up

  Data is recorded market data from multiple sessions.
  System runs autonomously — no human intervention during simulation.

SECTION 10 — COMMON MISTAKES
----------------------------------------------------------------------

10.1 Architecture-breaking mistakes (will fail Phase 10)

  Mistake 1: Measuring coverage but not fixing gaps
    Prevention: Gate 10.1 fails if coverage < 80%.  No workaround.
    Detection:   coverage_report.json shows per-module gaps

  Mistake 2: Integration tests that pass without meaningful assertions
    Prevention: Gate 10.2 requires ≥1 assertion per test
    Detection:   pytest-audit finds assert-free tests

  Mistake 3: Using synthetic data for all replay tests
    Prevention: Gate 10.4 requires ≥1 day of recorded market data
    Detection:   replay_report.json shows data_source = synthetic

  Mistake 4: Ignoring replay divergence
    Prevention: Gate 10.4 fails if deterministic module diverges
    Detection:   replay comparator finds field-level differences

  Mistake 5: Running edge case tests with data that doesn't trigger
             the edge case
    Prevention: Each scenario defines expected system behavior;
                test fails if expected behavior not observed
    Detection:   scenario_results.json shows expected=observed mismatch

  Mistake 6: Running performance tests on a non-idle machine
    Prevention: Performance gate specifies baseline conditions:
                - No other heavy processes running
                - CPU idle > 90% before test
                - Memory available > 8GB
    Detection:   performance_report.json flags if baseline conditions
                 not met

  Mistake 7: Paper simulation with too few trades
    Prevention: Gate 10.7 requires ≥100 trades minimum
    Detection:   simulation_report.json shows trade count < 100

  Mistake 8: Ignoring critical failures in paper simulation
    Prevention: Gate 10.7 fails on ANY critical failure
    Detection:   simulation_report.json shows critical_failures > 0

  Mistake 9: Running Phase 10 while still making code changes
    Prevention: Phase 10 is validation-only.  Any code change during
                Phase 10 requires restarting from Gate 10.1.
    Detection:   git diff shows uncommitted changes

10.2 What creates rework later (and how to prevent it)

  Rework 1: Low coverage discovered late
    Fix:      Add unit tests for uncovered modules, rerun gate
    Prevent:  Run coverage locally after every Phase 0-8 module build

  Rework 2: Integration tests that depend on test order
    Fix:      Make each test independent, rerun
    Prevent:  conftest.py creates fresh system instance per test

  Rework 3: Replay divergence caused by non-deterministic code
    Fix:      Make module deterministic (seed random, fix time), rerun
    Prevent:  Use deterministic randomness (seeded) in all modules

  Rework 4: Edge case not covered by existing scenarios
    Fix:      Add scenario, rerun gate
    Prevent:  Review edge case list before building scenarios

  Rework 5: Performance regression from recent change
    Fix:      Profile, identify bottleneck, optimize, rerun
    Prevent:  Run performance tests after every significant change

  Rework 6: Paper simulation failures due to risk gate configuration
    Fix:      Adjust risk gate parameters, rerun simulation
    Prevent:  Test risk gate configuration with small simulation first

  Rework 7: Test infrastructure itself has bugs
    Fix:      Fix test harness, rerun gate
    Prevent:  Unit test the test harness (test conftest fixtures,
              test scenario runner, test comparator logic)
